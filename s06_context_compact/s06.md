# s06: Context Compact

前面几章的 agent 会把所有用户消息、模型回复、工具调用和工具结果都保存在 `context` 里。短任务没有问题，但长任务会很快遇到两个麻烦：

- 工具输出太长，占满上下文。
- 历史消息太多，模型难以抓住当前重点。

本章加入上下文压缩机制，让 agent 可以在长任务中继续工作。

## 运行方式

```bash
cargo run -p s06_context_compact
```

## 本章新增能力

- 新增 `compact` 工具，允许模型主动请求压缩历史。
- 自动估算上下文大小，超过阈值后触发压缩。
- 保存完整 transcript 到 `.transcripts/`。
- 将超长 bash 输出落盘到 `.task_outputs/tool-results/`，上下文里只保留预览。
- 对旧的 tool result 做轻量压缩，只保留最近几个完整结果。
- 记录最近读过的文件，压缩摘要中提醒 agent 必要时重新打开。

## 核心问题

Agent harness 的上下文本质上是一条行动轨迹：

```text
user goal
assistant tool_use
tool_result
assistant tool_use
tool_result
...
assistant final answer
```

这个轨迹越完整，模型越能理解过程；但它也越容易变长。上下文管理的目标不是“简单删除旧消息”，而是保留能让任务继续推进的信息：

- 当前目标。
- 重要发现和决策。
- 已经读过或改过的文件。
- 剩余工作。
- 用户约束和偏好。

## 自动压缩

`LoopState::agent_loop()` 每轮请求模型前都会做两件事：

```rust
micro_compact(&mut self.context);

if estimate_context_size(&self.context) > CONTEXT_LIMIT {
    println!("[auto compact]");
    self.compact_history(None).await?;
}
```

`CONTEXT_LIMIT` 当前设置为 50000。这里用序列化后的字符数做近似估算，不是精确 token 计数，但足够演示机制。

## micro_compact

`micro_compact()` 处理的是“局部瘦身”：保留最近 3 个 tool result，较早且较长的 tool result 替换成占位文本：

```text
[Earlier tool result compacted. Re-run the tool if you need full detail.]
```

这样做的原因是：旧工具结果通常已经被模型阅读并总结过，继续完整保留收益不高；而最近几个工具结果往往仍然和当前步骤强相关。

## 完整压缩

当上下文超过阈值，或者模型调用 `compact` 工具时，会进入 `compact_history()`：

```text
写出完整 transcript
  -> 调用模型总结历史
  -> 加入 focus 和最近文件
  -> 用一条 compacted summary 替换整个 context
```

压缩后上下文变成：

```text
This conversation was compacted so the agent can continue working.

<summary>
```

这等于给模型一个新的起点，但保留了继续任务所需的信息。

## transcript

压缩前会调用 `write_transcript()`，把完整消息历史写到：

```text
.transcripts/transcript_<timestamp>.jsonl
```

这很重要。压缩会丢失上下文中的细节，但 transcript 让调试和追溯仍然可行。

## 超长输出落盘

`persist_large_output()` 针对 bash 输出做特殊处理：

- 如果输出不超过 30000 字符，直接返回。
- 如果超过阈值，完整内容写入 `.task_outputs/tool-results/<tool_use_id>.txt`。
- 上下文里只返回文件路径和前 2000 字符预览。

这是一种常见的 agent harness 技巧：让大结果留在文件系统里，让上下文只保存索引和摘要。

## compact 工具

`compact` 工具本身的 `invoke()` 只返回：

```text
Compacting conversation...
```

真正的压缩动作不在工具内部完成，而是在 `execute_tool_call()` 识别到 `name == "compact"` 后触发：

```rust
if name == "compact" {
    manual_compact = true;
    compact_focus = input.get("focus").and_then(|v| v.as_str());
}
```

这是因为压缩需要访问整个 `LoopState.context` 和模型客户端，不是普通工具能独立完成的动作。

## Rust 实现要点

- 用 `serde_json::to_string()` 估算上下文大小。
- 用 JSONL 保存 transcript，便于逐行读取。
- 用 `SystemTime` 生成 transcript 文件名。
- 用 `BufWriter` 写出完整消息历史。
- 用 `LoopState` 保存 `CompactState`，记录压缩状态和最近文件。

## 本章的局限

- 字符数不是精确 token 数。
- 压缩摘要质量依赖模型。
- 压缩后旧 tool result 不在上下文中，必要时需要重新读取 transcript 或重新执行工具。
- 当前只对 bash 大输出做落盘处理，其他工具也可能产生大输出。
