# s13

## 目标

`s13` 的核心目标，是把 agent 从“工具调用时只能同步等待”推进到“慢命令可以先放到后台，主循环继续做别的事”。

这一章对应 Python 原版里的核心思想是：

> long-running work should use runtime execution slots, not block the whole agent loop

这里最重要的区分和 `s12` 一样需要讲清楚：

- `s12` 的 task record 是 durable work item
- `s13` 的 background task 是 runtime execution slot

它们都可以落到磁盘，但语义不同。

- task board 解决的是“工作状态如何持久化”
- background task 解决的是“慢命令执行时主循环如何不被卡住”

所以这一章的重点不是再做一套 task graph，而是：

- 给慢命令一个后台执行入口
- 让完成结果回到 agent loop
- 让运行时状态和输出文件落到 `.runtime-tasks/`

---

## 当前 Rust 版做了什么

当前 `s13` 可以概括成一句话：

**把后台命令执行接进现有工具框架，并把完成通知重新注入主循环。**

Rust 版最终落成了三层结构：

1. `SharedBackgroundManager` 负责后台任务状态、落盘和通知通道
2. `background_run / check_background` 作为正常工具接回 loop
3. `agent_loop()` 在每次发起模型请求前注入后台结果

这样主循环仍然保持教学上的主线：

1. 先吸收后台结果
2. 再调用模型
3. 如果模型要用工具，就执行工具

关键调用仍然留在主循环里，后台执行细节则被收进 manager。

---

## 当前 Rust 版的结构

### 1. `BackgroundTaskStatus / BackgroundTaskRecord`

Rust 版没有沿用字符串状态，而是把后台任务状态收敛成 `enum`：

- `Running`
- `Completed`
- `Timeout`
- `Error`

对应的记录结构 `BackgroundTaskRecord` 则明确表达了后台任务真正需要保留的数据：

- `id`
- `status`
- `result: Option<String>`
- `command`
- `started_at`
- `finished_at: Option<_>`
- `result_preview`
- `output_file`

这样做的重点不是“更 Rust”，而是把运行时概念边界收清楚：

- 未完成的任务没有结果，所以用 `Option`
- 状态不是随便写的字符串，而是有限集合
- 对外展示和落盘记录都围绕同一份结构
- 时间点作为真正的时间类型持久化，而不是裸整数

时间这一层，Rust 版也额外做了一个明确取舍：

- 落盘状态保存 UTC 时间点
- 面向读取结果时再补本地时区展示

这样可以同时满足两件事：

- 持久状态稳定、可迁移
- 人在查看任务详情时不需要自己换算时区

### 2. `SharedBackgroundManager`

当前 Rust 版没有把后台逻辑散落在工具和主循环里，而是集中到 `SharedBackgroundManager`。

它负责四件事：

- 创建 `.runtime-tasks/`
- 为每个后台任务写 `<task_id>.json` 和 `<task_id>.log`
- 在内存里维护当前进程可见的后台状态
- 通过 notification channel 把后台状态更新送回主循环

这里和 Python 原版最接近的地方是：

- `background_run` 会立刻返回 task id
- 实际命令在后台继续跑
- 后续状态更新会在下一次模型调用前注回对话，不直接打断当前控制流

但 Rust 版没有做一个长期驻留、显式暴露给外面的 worker pool。

原因是这一章执行的是外部 shell 命令，真正耗时主要在子进程本身，而不是 Rust 线程里的 CPU 计算。对这个教学场景来说：

- 用 manager 封装 `tokio::spawn`
- 用 channel 传回完成通知

已经能把耦合度压得很低，同时保留清晰边界。

### 3. `background_run / check_background`

Rust 版把后台能力接成了两个普通工具：

- `background_run`
- `check_background`

这和 Python 原版保持一致：

- 启动后台任务不是特殊模式，而是一次正常 tool use
- 查询后台任务也不是旁路逻辑，而是普通工具调用

所以模型不需要学习第二套控制流。

### 4. 主循环只负责主线

`agent_loop()` 没有直接管理后台任务的执行细节。

它只做两件和这一章直接相关的事：

1. 调模型前 drain notification channel
2. 把后台状态更新包装成 `<background-results>` 注回上下文

这样做的教学价值在于：

- 后台执行确实进入了 agent loop
- 但 loop 仍然保留“请求模型 -> 执行工具 -> 继续”的清晰骨架
- “如何组织后台结果” 这件事留在 manager 内部，而不是散落在主循环里

## 和 Python 原版的对应关系

如果按教学意图对照，当前 Rust 版和 Python 原版的主线是一致的：

1. 都给慢命令提供后台执行入口
2. 都会把后台完成结果重新交还给模型
3. 都会把运行时记录写到磁盘目录里

也就是说，这一章讲的不是 durable task board，而是：

- 如何把“执行中的工作”从同步阻塞，变成后台槽位
- 如何让主循环在下次请求模型前吸收这些结果

---

## 和 Python 原版的相同点

### 1. 都保留了 `background_run` 和 `check_background`

两边都让模型可以：

- 启动后台命令
- 轮询或列出后台状态

### 2. 都会把完成结果摘要重新注入上下文

两边都不是要求模型自己记得去读某个全局状态，而是在下一轮模型调用前把完成通知交回去。

Rust 版这里额外保留了一个清楚的边界：

- 真正超时才进入 `Timeout`
- “跑得太久” 只是额外观察到的 stalled 信息，不冒充失败状态

### 3. 都把输出写到单独文件

Rust 版同样会写：

- `.runtime-tasks/<task_id>.json`
- `.runtime-tasks/<task_id>.log`

这样后台任务的完整输出不只存在于一段临时上下文里。

---

## 和 Python 原版的不同点

### 1. Rust 版把后台状态收敛成显式类型

Python 原版里后台任务状态主要是字典和字符串。

Rust 版把它们收敛成：

- `BackgroundTaskStatus`
- `BackgroundTaskRecord`
- `BackgroundNotification`

这样状态层和通知层的边界更清楚。

时间表示也一起做了收敛：

- 持久层保存 UTC 时间点
- 展示层再补本地时间字符串

### 2. Rust 版把通知机制显式做成 channel

Python 原版是列表加锁，完成后 append，再由主循环 drain。

Rust 版这里显式使用 notification channel，让“后台执行”和“主循环吸收结果”之间只通过通知消息通信。工具层不需要知道主循环怎么消费这些消息，主循环也不需要知道命令怎么跑完。

### 3. Rust 版没有额外做一套常驻线程池接口

这是有意的取舍。

这一章的工作负载是“启动外部命令然后等待它结束”，不是把一堆 CPU 任务丢给 Rust worker 线程计算。对这种场景，显式线程池不是最小必要抽象。

所以 Rust 版最终选择的是：

- manager 负责创建后台任务
- 每个任务异步等待子进程完成
- 完成后通过 channel 发通知

如果后面要加入：

- 并发上限
- 优先级
- 排队策略

再把 manager 内部升级成 dispatcher + worker pool 会更自然；但在当前教学目标下，没有把这层复杂度提前暴露出来。
