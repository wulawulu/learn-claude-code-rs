# s13: Background Tasks

Some commands take longer than a single agent turn should block for: test suites, build jobs, servers, data processing, and watchers.

This chapter adds background task support. The agent can start a command, continue working, and check the result later.

## Run

```bash
cargo run -p s13_background_tasks
```

## New Capabilities

- Add `BackgroundManager`.
- Add `background_run`.
- Add `background_check`.
- Run commands in background processes.
- Persist background task metadata.
- Capture stdout, stderr, exit status, and running state.

## Why Background Tasks

If every shell command blocks the agent loop, long-running work freezes the conversation.

Background tasks let the agent:

- Start a long command.
- Continue investigating or editing.
- Check the command later.
- Use results when they are ready.

This is the first step toward asynchronous agent workflows.

## background_run

Example input:

```json
{
  "command": "cargo test --workspace"
}
```

The tool starts the command in a background process and returns a task id.

The agent can then continue other work.

## background_check

Example input:

```json
{
  "id": 1
}
```

The tool returns:

- Whether the process is still running.
- Exit status if finished.
- Captured stdout and stderr.

## BackgroundManager

The manager owns task records and process state. It keeps the tool layer small:

```text
tool input
  -> BackgroundManager
  -> rendered status
```

## Rust Notes

- Use `tokio::process::Command` for async process handling.
- Store background metadata separately from conversation context.
- Return ids so later tool calls can query state.
- Keep output bounded before returning it to the model.

## Limits

- No streaming output.
- No cancellation tool yet.
- No process supervision after the parent process exits.
- No restart policy.
- Output handling is simple.
