# s18: Worktree Task Isolation

s17 让 teammate 可以长期运行并自动认领任务，但多个 agent 如果都在同一个工作区里修改文件，很容易互相覆盖。真正的 coding agent harness 需要把“要做什么”和“在哪里做”分开。

本章引入 git worktree lane：任务仍然由 task system 管理，执行环境则由 worktree system 管理。subagent 可以被绑定到某个 worktree，只在自己的 lane 里读写和运行命令。

## 运行方式

```bash
cargo run -p s18_worktree_task_isolation
```

运行后会使用：

```text
.tasks/
.worktrees/
.worktrees/index.json
.worktrees/events.jsonl
```

`worktree_*` 工具依赖当前目录位于 git 仓库中。如果不是 git 仓库，基础 agent 仍能运行，但 worktree 工具会返回错误。

## 本章新增能力

- 新增 `WorktreeManager`。
- 用 `.worktrees/index.json` 记录 lane 状态。
- 用 `git worktree add` 创建隔离工作区。
- task 可以绑定到 worktree。
- subagent 可以在指定 task 或 worktree 内运行。
- 新增 `worktree_run`，在 lane 内执行命令。
- 新增 `worktree_closeout`，保留或移除 lane。
- 用 `.worktrees/events.jsonl` 记录 worktree 事件。

## 为什么需要 Worktree Isolation

多 agent 并行执行时，单一工作区有几个风险：

- 两个任务同时编辑同一个文件。
- 子代理为了探索运行破坏性命令。
- 半成品修改污染主线工作区。
- 很难判断某个任务到底改了哪些内容。

git worktree 提供了一个实用边界：

```text
同一个仓库
  -> 多个独立工作目录
  -> 每个目录绑定不同分支
  -> 每个任务在自己的 lane 中完成
```

本章的目标不是实现完整沙箱，而是把 coding agent 的工作目录从“全局 workspace”变成“可分配的任务 lane”。

## TaskRecord 的 Worktree 字段

本章扩展了 task 记录：

```rust
pub struct TaskRecord {
    pub id: u64,
    pub subject: String,
    pub status: TaskStatus,
    pub owner: String,
    pub worktree: String,
    pub worktree_state: WorktreeState,
    pub last_worktree: String,
    pub closeout: Option<CloseoutRecord>,
    ...
}
```

这些字段表达任务和执行环境的关系：

- `worktree`：当前绑定的 lane。
- `last_worktree`：最近使用过的 lane。
- `worktree_state`：lane 是 active、kept 还是 removed。
- `closeout`：lane 结束时的处理记录。

这样 task 不只是“做什么”，还知道“在哪里做过”。

## WorktreeRecord

`.worktrees/index.json` 保存所有 lane：

```rust
pub struct WorktreeRecord {
    pub name: String,
    pub path: String,
    pub branch: String,
    pub task_id: Option<u64>,
    pub status: WorktreeState,
    pub last_entered_at: Option<f64>,
    pub last_command_at: Option<f64>,
    pub last_command_preview: Option<String>,
    pub closeout: Option<CloseoutRecord>,
}
```

它是 worktree manager 的 durable index。agent 可以通过工具查询当前有哪些 lane、每个 lane 对应哪个 task、最后运行过什么命令。

## WorktreeManager

`WorktreeManager` 负责所有 worktree 生命周期操作：

- `create()`：创建 git worktree，并可绑定 task。
- `list_all()`：列出已跟踪 lane。
- `status()`：查看某个 lane 的 git status。
- `enter()`：标记准备进入某个 lane。
- `run()`：在 lane 内执行 shell 命令。
- `closeout()`：保留或移除 lane。
- `events()`：读取最近事件。

它内部持有：

```rust
repo_root
.worktrees/ dir
index_path
SharedTaskManager
EventBus
```

task manager 和 worktree manager 通过共享引用协作。创建 lane 时可以更新 task 的绑定；关闭 lane 时也可以更新 task 的 closeout 状态。

## Worktree Tools

本章新增的工具集中在 `src/tool/worktree.rs`：

- `worktree_create`
- `worktree_list`
- `worktree_enter`
- `worktree_status`
- `worktree_run`
- `worktree_closeout`
- `worktree_events`

典型流程是：

```text
task_create
  -> worktree_create(name, task_id)
  -> sub_agent(prompt, task_id)
  -> worktree_status(name)
  -> worktree_closeout(name, keep/remove)
```

这些工具让模型可以显式管理执行 lane，而不是只靠全局 `bash`。

## Subagent Target

`sub_agent` 工具新增了两个可选字段：

```json
{
  "prompt": "Implement the task and summarize changes.",
  "task_id": 1,
  "worktree": "task-1"
}
```

目标解析规则是：

- 如果给了 `task_id`，使用该 task 绑定的 worktree。
- 如果给了 `worktree`，直接使用指定 lane。
- 如果两者都给了，要求它们互相匹配。
- 如果都没给，退回根 workspace。

子代理启动后只拿到 scoped workspace 的工具集合。文件工具和 bash 都以 lane 路径为边界，防止子代理用相对路径逃回主工作区。

## worktree_run

`worktree_run` 在指定 lane 内执行命令：

```json
{
  "name": "task-1",
  "command": "cargo test -p s18_worktree_task_isolation"
}
```

执行前会做简单危险命令检查。执行时会记录：

- `last_command_at`
- `last_command_preview`
- `worktree.run.before`
- `worktree.run.after`
- `worktree.run.failed`

这让 agent 和用户都能追踪 lane 中发生过什么。

## Closeout

任务完成后，lane 需要明确收尾：

```json
{
  "name": "task-1",
  "action": "keep",
  "reason": "Need human review before merge.",
  "complete_task": true
}
```

`action` 有两种：

- `keep`：保留 worktree，适合等待人工 review 或后续合并。
- `remove`：删除 worktree，适合任务已结束且不需要保留目录。

如果 remove 时 lane 仍有未提交修改，默认会拒绝，除非传入 `force=true`。

## EventBus

`.worktrees/events.jsonl` 是一个轻量事件日志。

它记录：

- worktree 创建前后。
- 命令执行前后。
- subagent 开始和结束。
- closeout 行为。
- 失败原因。

事件日志不是模型上下文，但可以通过 `worktree_events` 工具按需读取。这样长任务调试时，不需要完全依赖聊天历史。

## Rust 实现要点

- 用 `std::process::Command` 调用 git 和 shell。
- 用 `PathBuf` 和 canonicalize 限制 scoped workspace。
- 用 JSON 文件保存 task 和 worktree index。
- 用 JSONL 保存事件流。
- 用 `Arc<Mutex<...>>` 共享 task manager 和 worktree manager。
- 用 `strum` 解析 `WorktreeState` 和 `CloseoutAction`。

## 本章的局限

- worktree isolation 不是安全沙箱，仍然共享同一台机器和权限。
- `worktree_run` 的危险命令检查很有限。
- 没有自动 merge、rebase 或冲突处理。
- `.worktrees/index.json` 没有跨进程锁。
- closeout 只记录保留或移除，不负责把结果合并回主分支。
