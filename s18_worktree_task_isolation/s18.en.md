# s18: Worktree Task Isolation

s17 lets teammates run and claim tasks, but multiple agents editing the same workspace can overwrite each other. A coding agent harness needs to separate what to do from where to do it.

This chapter introduces git worktree lanes. Tasks are managed by the task system; execution environments are managed by the worktree system. A subagent can be bound to a worktree and work inside that lane only.

## Run

```bash
cargo run -p s18_worktree_task_isolation
```

Runtime state:

```text
.tasks/
.worktrees/
.worktrees/index.json
.worktrees/events.jsonl
```

`worktree_*` tools require a git repository.

## New Capabilities

- Add `WorktreeManager`.
- Track lanes in `.worktrees/index.json`.
- Create isolated directories with `git worktree add`.
- Bind tasks to worktrees.
- Run subagents inside a task or worktree lane.
- Add `worktree_run` for commands inside a lane.
- Add `worktree_closeout` to keep or remove a lane.
- Record worktree events.

## Why Worktree Isolation

One shared workspace has risks:

- Two tasks edit the same file.
- Exploratory commands affect main work.
- Half-finished changes pollute the main directory.
- It is hard to see what belongs to which task.

git worktree gives a practical boundary:

```text
same repository
  -> multiple working directories
  -> each directory has its own branch
  -> each task works in its own lane
```

## Task Worktree Fields

`TaskRecord` gains fields such as:

- `worktree`
- `worktree_state`
- `last_worktree`
- `closeout`

Now a task can record not only what should be done, but also where it was done.

## WorktreeRecord

`.worktrees/index.json` stores:

- lane name
- path
- branch
- task id
- status
- last entered time
- last command preview
- closeout record

This is the durable index for worktree lanes.

## WorktreeManager

The manager handles:

- `create()`
- `list_all()`
- `status()`
- `enter()`
- `run()`
- `closeout()`
- `events()`

It coordinates with the task manager when lanes are created or closed.

## Worktree Tools

Tools:

- `worktree_create`
- `worktree_list`
- `worktree_enter`
- `worktree_status`
- `worktree_run`
- `worktree_closeout`
- `worktree_events`

Typical flow:

```text
task_create
  -> worktree_create(name, task_id)
  -> sub_agent(prompt, task_id)
  -> worktree_status(name)
  -> worktree_closeout(name, keep/remove)
```

## Subagent Target

`sub_agent` accepts optional `task_id` and `worktree`.

Rules:

- If `task_id` is provided, use the task's bound worktree.
- If `worktree` is provided, use that lane.
- If both are provided, they must match.
- If neither is provided, use the root workspace.

The child gets a scoped toolset whose file and shell operations are bounded to the lane path.

## Closeout

At the end, a lane should be closed out:

- `keep`: preserve the worktree for review or merge.
- `remove`: remove the worktree.

Removal refuses dirty worktrees unless `force=true`.

## EventBus

`.worktrees/events.jsonl` records:

- create events
- command execution
- subagent start/finish
- closeout events
- failures

The agent can read these events with `worktree_events`.

## Limits

- Worktree isolation is not a security sandbox.
- Command safety checks are limited.
- No automatic merge or conflict handling.
- No cross-process lock for `.worktrees/index.json`.
- Closeout does not merge results back to main.
