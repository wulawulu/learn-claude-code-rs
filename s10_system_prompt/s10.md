# s10

## 目标

`s10` 的核心目标是把 **system prompt 从“一整段硬编码字符串”提升为“可组装、可扩展的结构”**。

这一章对应 Python 原版里的核心思想是：

> system prompt should be assembled from clear sections, not written as one giant hardcoded blob

也就是：

- prompt 不应该只是一个长字符串
- 不同来源的信息应该分别建模
- 稳定信息和动态信息应该有清晰边界
- 后续要加新 section 时，不应该回到“继续拼字符串”的老路

Rust 版保留了这个目标，但没有机械翻译 Python 的 `SystemPromptBuilder` 流程，而是收敛成：

- `SystemPrompt` 负责定义“有哪些 section”
- `Prompt` 负责模板化渲染
- `LoopState` 负责从运行时状态中取出 memory / skill / CLAUDE.md / dynamic context 并注入

所以这一章的重点不是“翻译 Python 写法”，而是把 **prompt construction 变成一个清晰的系统接口**。

---

## 当前 Rust 版做了什么

当前 `s10` 主要引入了三层职责分离。

### 1. `SystemPrompt`：定义系统提示的结构

在 `src/prompt.rs` 里，`SystemPrompt` 现在已经不只是 `role / guidelines / constraints / additional` 了，还新增了：

- `skills_available`
- `memory`
- `claude_md`
- `dynamic_context`
- `memory_guidance`

这意味着 system prompt 的几个核心 section 都有了显式字段，而不是靠外部随手拼接字符串。

同时它还提供了和现有风格一致的接口：

- `with_role`
- `with_guidelines`
- `with_constraints`
- `with_skills_available`
- `with_memory`
- `with_claude_md`
- `with_dynamic_context`
- `with_memory_guidance`
- `with_additional`

这里最重要的设计点是：

- **memory 和 skills 现在是 prompt 的一等组成部分**
- 它们不再只是 `build_system_prompt()` 里临时拼进去的一段文本

这也是这一章最关键的收敛：把“prompt section”变成显式的数据结构，而不是隐式的字符串约定。

### 2. `Prompt`：负责模板化渲染，而不是业务拼接

`Prompt` 这一层没有被推翻，而是继续承担模板渲染职责。

当前流程是：

1. `SystemPrompt` 持有结构化字段
2. `Into<Prompt>` 时把这些字段写进模板上下文
3. 用 `tera` 渲染 `system_prompt_template.md`

这样做的价值是：

- section 的存在与否由模板控制
- 数据来源和展示形式分开
- 以后新增 section 时，只要加字段和模板片段即可

所以 Rust 版这里强调的是：

- **Prompt 是模板引擎层**
- **SystemPrompt 是 prompt 结构层**
- 两者职责分开

这比 Python 原版那个“builder 直接返回大字符串”的教学实现更适合扩展。

### 3. `LoopState`：从运行时 state 注入动态 section

在 `src/lib.rs` 里，`LoopState::build_system_prompt()` 现在不再手工 `join("\n\n")` 拼字符串，而是改成：

- 先构造 `SystemPrompt::builder()`
- 从 `skill_registry.describe_available()` 取技能摘要
- 从 `self.load_memory_prompt()` 取 memory prompt
- 从 `self.load_claude_md_prompt()` 取 CLAUDE.md 链
- 从 `self.load_dynamic_context()` 取动态上下文
- 再统一 `.to_prompt().render()`

另外，memory 的接入也放回到了 state 层：

- `LoopState` 里新增了 `load_memory_prompt()`
- 它负责锁住 `memory_manager` 并调用 `MemoryManager::load_memory_prompt()`
- `LoopState` 里新增了 `load_claude_md_prompt()`
- 它负责按优先级收集 `~/.claude/CLAUDE.md`、项目根 `CLAUDE.md` 和当前目录 `CLAUDE.md`
- `LoopState` 里新增了 `load_dynamic_context()`
- 它负责生成日期、工作目录、模型和平台等动态信息

所以运行时的职责边界也更清楚了：

- `state` 负责拿数据
- `SystemPrompt` 负责装结构
- `Prompt` 负责最终渲染

## 模板层做了什么

在 `src/system_prompt_template.md` 里，系统提示现在已经是 section 化模板，而不是默认把所有内容写死。

当前模板支持按条件渲染：

- `role`
- `skills_available`
- `guidelines`
- `constraints`
- `memory`
- `claude_md`
- `memory_guidance`
- `additional`
- `dynamic_context`

这很重要，因为它体现了这一章真正想教的东西：

- prompt 不是一段固定文本
- prompt 是“若干 section 的组合”
- section 可以按上下文决定是否出现
- 稳定 section 应该排在 boundary 之前，动态 section 应该排在 boundary 之后

这和 Python 版的“pipeline with boundaries”是同一个方向，但 Rust 版落在模板系统上，更利于以后继续演化。

## memory / skill / CLAUDE.md / dynamic context 在 Rust 版中的处理

这是当前 Rust 版和 Python 原版差异最大、也是更合理的地方。

### skill

Python 原版的 skill section 是自己扫描 `skills/` 目录、读 `SKILL.md`、解析 frontmatter，然后手工组装成文本。

当前 Rust 版没有在 `s10` 里重复发明一套 skill prompt 构建逻辑，而是直接复用已有能力：

- `SkillRegistry::describe_available()`

也就是说这一章不再关心：

- skills 目录怎么扫
- frontmatter 怎么解析
- 文本怎么逐行拼

它只关心：

- system prompt 需要一个 “available skills” section
- 这个 section 从 `skill registry` 拿到摘要即可

这比 Python 原版更干净，也更符合模块边界。

### memory

Python 原版会自己遍历 `.memory` 目录、解析 frontmatter、再把 memory 文本拼成 prompt section。

当前 Rust 版同样没有在 `s10` 里重新写一遍 memory prompt 构造，而是直接复用：

- `MemoryManager::load_memory_prompt()`

然后把这件事通过 `LoopState::load_memory_prompt()` 接到 state 上。

所以这里的原则也很明确：

- `memory manager` 负责 memory 的组织和格式
- `state` 负责把它暴露给 system prompt
- `SystemPrompt` 只负责把它作为一个 section 接进来

这比 Python 原版的“builder 自己懂 memory 文件结构”要更清晰。

### CLAUDE.md

Python 原版会把 `CLAUDE.md` 作为一个独立 section，并按来源链式加入 prompt。

当前 Rust 版也补上了这一点，但接法仍然和 memory / skill 一样，走 state 封装：

- `LoopState::load_claude_md_prompt()`

它负责：

- 尝试加载 `~/.claude/CLAUDE.md`
- 尝试加载项目根的 `CLAUDE.md`
- 如果当前目录和项目根不同，再尝试加载当前目录的 `CLAUDE.md`

然后统一整理成一个 `# CLAUDE.md instructions` section。

所以 Rust 版这里强调的是：

- `LoopState` 负责“从环境中取数据”
- `SystemPrompt` 负责“把这块数据接入 prompt”
- 模板负责“决定它怎么显示”

### dynamic context

Python 原版里，dynamic context 会加入：

- 当前日期
- 工作目录
- 模型
- 平台

当前 Rust 版也补上了这一节，同样通过 state 封装：

- `LoopState::load_dynamic_context()`

它负责每次构建 prompt 时生成这类运行时信息，然后作为一个独立 section 交给模板渲染。

这样做的好处是：

- 动态信息和稳定模板分开
- 以后要加更多运行时字段时，不需要改 builder 主流程
- 这一节的来源也保持清晰

### dynamic boundary

Python 原版里有一个显式的 `DYNAMIC_BOUNDARY`，用来强调：

- 上面是相对稳定的 prompt section
- 下面是每次构建都可能变化的动态内容

当前 Rust 版也把这个 marker 补上了：

- `=== DYNAMIC_BOUNDARY ===`

这里 Rust 版没有把它建模成运行时字段，而是直接把它写在模板里。

这样更合理，因为：

- boundary 本身不是动态数据
- 它只是模板结构的一部分
- `SystemPrompt` 不需要为这种固定标记增加额外字段

所以当前实现是让模板层在 `dynamic context` 出现时，自动在前面插入这个分隔符。

同时，`memory_guidance` 和 `additional` 仍然放在 boundary 之前，因为它们都属于稳定提示的一部分，不是每次构建都变化的运行时上下文。

## 与 Python 原版的对应关系

如果按教学意图来对照，当前 Rust 版和 Python 原版是对应的。

Python 原版的 section pipeline 是：

1. core instructions
2. tool listing
3. skill metadata
4. memory section
5. CLAUDE.md chain
6. dynamic context

当前 Rust 版真正保留下来的核心是：

1. role
2. guidelines
3. constraints
4. skills available
5. memory
6. CLAUDE.md instructions
7. memory guidance
8. additional
9. `=== DYNAMIC_BOUNDARY ===`
10. dynamic context

所以它们的共同点是：

- 都强调 system prompt 是由多个 section 组成
- 都强调 section 应该来源清晰
- 都强调 prompt construction 不应退化成一大段硬编码文本

## Rust 版为什么没有照搬 Python 的 6 段流水线

这里是最值得说明的地方。

Python 原版是教学代码，所以它把很多概念都放进一个 `SystemPromptBuilder` 里：

- 工具列表
- skill 元数据
- memory
- CLAUDE.md
- dynamic context

这样做的好处是：

- 很直观
- 很容易展示“prompt 是 pipeline”
- 每一段都能看见

但 Rust 版如果完全照搬，会有两个问题。

### 1. 会把已有模块边界打散

现在项目里：

- skill 已经有 `SkillRegistry`
- memory 已经有 `MemoryManager`
- prompt 已经有 `SystemPrompt + Prompt`

如果再让 `s10` 自己去扫描技能目录、自己解析 memory 文件、自己管理动态 section，就会重复已有逻辑。

这样会让代码看起来像“在这一章里重新实现半个系统”，而不是“把现有模块装配成 prompt pipeline”。

### 2. 会把“结构化 prompt”重新做回“字符串 builder”

Python 原版的 `build()` 本质上仍然是字符串拼接，只是 section 更清楚。

Rust 版这次刻意没有停在这个层面，而是继续往前走了一步：

- section 不只是代码里的拼接块
- section 是 `SystemPrompt` 的字段
- 渲染由模板负责

所以 Rust 版更强调：

- **结构先于字符串**
- **接口先于拼接**
- **复用已有模块，而不是在这一章重复实现它们**

## 当前 Rust 版和 Python 原版的相同点

### 1. 教学主题相同

两者都在讲一件事：

- system prompt 应该分段构造
- 不要把所有内容揉成一个难维护的硬编码 blob

### 2. 都把 skills 和 memory 看作 prompt 的组成部分

虽然实现不同，但两边都认为：

- skills available 应该出现在 system prompt 里
- persistent memory 应该进入 system prompt
- 这些都不是零散附加文本，而是 prompt pipeline 的一部分

### 3. 都保留了“动态内容单独处理”的方向

Python 原版通过 `DYNAMIC_BOUNDARY` 强调静态与动态分界。

Rust 版也保留了这个 marker，并且把它直接放在模板里。再配合 `LoopState -> SystemPrompt -> Prompt` 的装配流程，可以清楚看到：

- boundary 是模板结构的一部分
- 数据是运行时注入的
- 模板是稳定的
- prompt 结构和 prompt 内容是分开的

所以教学意图仍然一致。

---

## 当前 Rust 版和 Python 原版的不同点

### 1. Rust 版没有直接复刻 `SystemPromptBuilder`

Python 原版有一个显式的 builder，内部逐段调用：

- `_build_core`
- `_build_tool_listing`
- `_build_skill_listing`
- `_build_memory_section`
- `_build_claude_md`
- `_build_dynamic_context`

Rust 版没有做这个一模一样的流水线对象，而是采用：

- `SystemPrompt` 作为结构体
- `Prompt` 作为模板
- `LoopState` 作为数据注入入口

所以 Rust 版是“结构化装配”，不是“分段字符串 builder”。

### 2. Rust 版不再自己实现 skill/memory 的 prompt 组装

Python 原版是 builder 自己读目录、自己解析。

Rust 版改成复用：

- `skill_registry.describe_available()`
- `memory_manager.load_memory_prompt()`

这意味着 Rust 版更强调模块复用，而不是在 `s10` 里再实现一次底层逻辑。

### 3. Rust 版目前没有实现 tool listing

Python 示例里明确展示了：

- 工具清单
- CLAUDE.md 链
- 动态边界 `DYNAMIC_BOUNDARY`
- date/model/platform 等动态上下文

当前 Rust 版已经补上了后 3 项，只是没有把 `tool listing` 放进 prompt。

这不是漏翻译，而是取舍：

- `s10` 当前聚焦的是 system prompt 的结构化建模
- 尤其是 memory / skill / CLAUDE.md / dynamic context 如何以可扩展的方式接入
- 而不是把 Python 样例里的所有 section 一次性原样复制

这里目前只剩 `tool listing` 没有加入，而且这是有意保留：

- Rust 版已经通过 Anthropic SDK 的 `with_tools(...)` 把工具以结构化方式传给模型
- 这些工具定义不是 prompt 文本，而是 API 请求里的正式字段
- 如果再在 system prompt 里重复一份 tool listing，会产生重复信息

所以当前 Rust 版的判断是：

- `tool listing` 不再放进 system prompt
- 工具能力由 SDK 的 tool schema 负责传递

### 4. Rust 版模板层更强

Python 版最终产物仍然是 `"\n\n".join(sections)`。

Rust 版最终产物是模板渲染，这让它天然更适合：

- section 条件显示
- 新增字段
- 自定义模板替换
- 后续抽出 partial 或多模板支持

这一点是 Rust 版相对 Python 版更工程化的地方。

---

## 当前实现的价值

从教学上看，当前 `s10` 实际讲清楚了三件事：

1. `system prompt` 应该是结构，不应该只是字符串。
2. `memory` 和 `skills` 这类能力型信息，应该通过已有模块接入，而不是在 prompt builder 里重写一遍。
3. prompt 的最终文本应该交给模板层渲染，这样系统才真正具备扩展性。

所以如果用一句话总结当前 `s10`：

> Python 原版强调“system prompt 是分段拼起来的”，Rust 版进一步把它落实成“system prompt 是一个可扩展的数据结构，由 state 提供内容、由模板负责渲染”。
