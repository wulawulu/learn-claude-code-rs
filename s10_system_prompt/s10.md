# s10: System Prompt

前面的章节里，system prompt 基本都是临时字符串拼接。随着 agent 能力增加，prompt 里开始出现越来越多来源：

- agent 角色。
- 行为准则。
- 硬性约束。
- skills 摘要。
- memory 内容。
- 项目或用户的 `CLAUDE.md`。
- 当前日期、工作目录、模型、平台等动态上下文。

如果继续用一个大字符串维护，prompt 会越来越难读、难改、难测试。本章把 system prompt 变成结构化对象，并通过模板渲染。

## 运行方式

```bash
cargo run -p s10_system_prompt
```

## 本章新增能力

- 新增 `SystemPrompt` 结构体。
- 使用 builder 构造 prompt section。
- 使用 `tera` 模板渲染最终 system prompt。
- 将 skills、memory、CLAUDE.md、dynamic context 纳入统一 prompt 结构。
- 用 `=== DYNAMIC_BOUNDARY ===` 区分稳定提示和动态上下文。
- 复用前面章节的 skill registry 和 memory manager。

## 为什么要结构化 system prompt

System prompt 不是一段普通说明文字。它是 agent harness 给模型构造的运行环境说明。

一个成熟的 prompt 通常包含不同性质的信息：

- 稳定身份：你是谁。
- 软性建议：应该怎样做得更好。
- 硬性约束：必须遵守什么。
- 可用知识：有哪些 skills。
- 长期记忆：历史偏好和项目约定。
- 环境指令：项目中的 `CLAUDE.md`。
- 动态状态：当前日期、工作目录、平台。

这些信息的生命周期不同，来源不同，更新频率也不同。结构化之后，新增 section 不需要继续手工拼字符串。

## SystemPrompt

`SystemPrompt` 表达 prompt 的结构：

```rust
pub struct SystemPrompt {
    role: Option<String>,
    skills_available: Option<String>,
    memory: Option<String>,
    claude_md: Option<String>,
    dynamic_context: Option<String>,
    memory_guidance: Option<String>,
    guidelines: Vec<String>,
    constraints: Vec<String>,
    additional: Option<String>,
    template: Prompt,
}
```

这比大字符串更清楚：每个字段都代表一个明确 section。

## Builder

本章使用 `derive_builder` 生成 builder：

```rust
let prompt = SystemPrompt::builder()
    .role(format!("You are a coding agent operating in {}.", workdir.display()))
    .guidelines([
        "Try to understand how to complete the task well before completing it.",
    ])
    .constraints([
        "Think step by step",
        "Use the provided tools to interact with the system and accomplish the task",
    ])
    .skills_available(self.skill_registry.describe_available())
    .memory(self.load_memory_prompt()?)
    .claude_md(self.load_claude_md_prompt(&workdir))
    .dynamic_context(self.load_dynamic_context(&workdir))
    .memory_guidance(MEMORY_GUIDANCE.trim())
    .build()?;
```

这样 `LoopState` 只负责收集运行时数据，prompt 的结构由 `SystemPrompt` 表达，最终展示由模板决定。

## Prompt 渲染层

`Prompt` 是模板渲染层：

- 可以保存一段 one-off template。
- 可以引用已注册的 Tera template。
- 可以不断追加 context value。
- 最终调用 `render()` 输出字符串。

默认模板来自：

```rust
include_str!("system_prompt_template.md")
```

这让 system prompt 模板和 Rust 代码分离，读者可以直接打开 Markdown 模板查看最终结构。

## 模板结构

[`src/system_prompt_template.md`](./src/system_prompt_template.md) 按条件渲染 section：

```text
# Your role
# Available skills
# Guidelines you need to follow
# Constraints that must be adhered to
# Memories
# CLAUDE.md instructions
# Memory guidance
=== DYNAMIC_BOUNDARY ===
# Dynamic context
```

如果某个字段为空，对应 section 不渲染。

## Dynamic Boundary

模板中有：

```text
=== DYNAMIC_BOUNDARY ===
```

它的作用是区分：

- 上方：相对稳定的系统说明，如角色、约束、skills、memory guidance。
- 下方：每次构建可能变化的动态上下文，如日期、工作目录、模型、平台。

这种边界对调试 prompt 很有价值。你可以很快看出哪些内容应该长期稳定，哪些内容每次运行都会变化。

## Skills

s10 复用 skill registry：

```rust
self.skill_registry.describe_available()
```

System prompt 中只放技能摘要，不放所有技能正文。完整技能仍然由 `load_skill` 工具按需加载。

这延续了 s05 的原则：让知识可发现，但不要默认塞满上下文。

## Memory

s10 复用 memory manager：

```rust
self.load_memory_prompt()?
```

memory 作为独立 section 注入。这样长期偏好、项目约定和外部资源位置会在每次会话中被模型看到。

同时，`memory_guidance` 也作为单独 section 注入，提醒模型不要保存临时状态或敏感信息。

## CLAUDE.md

`load_claude_md_prompt()` 会尝试读取：

- `~/.claude/CLAUDE.md`
- 项目根目录 `CLAUDE.md`
- 当前目录 `CLAUDE.md`

如果存在，会整理成：

```text
# CLAUDE.md instructions

## From user global (~/.claude/CLAUDE.md)
...

## From project root (CLAUDE.md)
...
```

这类文件适合承载用户或项目的长期行为规范。

## Dynamic context

`load_dynamic_context()` 当前包含：

- 当前日期。
- 工作目录。
- 模型名。
- 操作系统。

这些信息每次构建 prompt 时重新生成。

## Rust 实现要点

- 用 `derive_builder` 降低结构化 prompt 的构造成本。
- 用 `tera` 让 prompt 展示逻辑从 Rust 代码中分离。
- 用 `LazyLock<RwLock<Tera>>` 支持全局模板仓库。
- 用 `Option<String>` 控制 section 是否渲染。
- 用 `Vec<String>` 表达多条 guidelines 和 constraints。

## 本章的局限

- prompt 没有快照测试。
- 没有 token 预算控制，不会根据长度自动裁剪 section。
- `CLAUDE.md` 只读取固定路径，没有向上级目录递归查找。
- dynamic context 内容仍然较少。
- memory 和 skills 仍然是直接注入摘要，没有检索排序。
