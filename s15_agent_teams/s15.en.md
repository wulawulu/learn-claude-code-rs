# s15: Agent Teams

s04 introduced one-shot subagents: the main agent starts a child, gives it a task, receives a summary, and the child exits. This chapter introduces another collaboration pattern: named teammates with persistent identity and inboxes.

An agent team is not a complex orchestration framework. It is a set of agent loops connected through mailboxes.

## Run

```bash
cargo run -p s15_agent_teams
```

Interactive commands:

```text
/team
/inbox
exit()
```

Runtime state is stored under:

```text
.team/
```

## New Capabilities

- Add a team lead.
- Add persistent teammates.
- Store team config in `.team/config.json`.
- Store inbox files in `.team/inbox/<name>.jsonl`.
- Let the lead spawn teammates.
- Let agents send messages to each other.
- Let the lead broadcast messages.
- Inject each agent's inbox into its context before model calls.

## Subagent vs Teammate

Subagent:

```text
spawn -> execute one subtask -> return summary -> exit
```

Teammate:

```text
stable name -> inbox -> can idle -> can receive more messages -> work again
```

A teammate is closer to a team member. A subagent is closer to a one-shot task runner.

## MessageBus

Each member has an inbox file:

```text
.team/inbox/lead.jsonl
.team/inbox/reviewer.jsonl
.team/inbox/implementer.jsonl
```

Message structure:

```rust
pub struct InboxMessage {
    pub message_type: MessageType,
    pub from_name: String,
    pub content: String,
    pub timestamp: f64,
}
```

An inbox is a pending queue, not a chat transcript. `read_inbox()` reads all messages and clears the file.

## Inbox Injection

`LoopState` gains:

```rust
pub message_bus: SharedMessageBus,
pub inbox_owner: String,
pub system_prompt: String,
pub max_rounds: usize,
```

Before each model request, the loop injects inbox messages:

```xml
<inbox>
[
  ...
]
</inbox>
```

Team communication still enters the model through context. It is not an out-of-band callback.

## TeamConfig

Team state is saved in:

```text
.team/config.json
```

It records:

- team name
- members
- member roles
- member status

Common statuses:

- `Idle`
- `Working`
- `Shutdown`

## spawn_teammate

Input:

```json
{
  "name": "reviewer",
  "role": "code reviewer",
  "prompt": "Review the current implementation and send findings to lead."
}
```

The tool:

- Creates or reuses a teammate record.
- Registers an inbox.
- Starts a teammate loop with `tokio::spawn`.
- Refuses to start the same working teammate twice.

## Lead Tools and Teammate Tools

Lead tools:

- `spawn_teammate`
- `list_teammates`
- `send_message`
- `read_inbox`
- `broadcast`
- file and bash tools

Teammate tools:

- `send_message`
- `read_inbox`
- file and bash tools

The tool boundary expresses role boundaries.

## send_message

```json
{
  "to": "reviewer",
  "content": "Please inspect src/lib.rs and report risks.",
  "msg_type": "message"
}
```

The message is appended to the target inbox.

## broadcast

The lead can broadcast to all teammates:

```json
{
  "content": "Stop current exploration and report status."
}
```

The sender does not receive its own broadcast.

## Rust Notes

- Use JSONL files as simple mailboxes.
- Use `Mutex<()>` to serialize inbox access in one process.
- Use `Arc` to share message bus and teammate manager.
- Use `tokio::spawn` for teammate loops.
- Persist team membership in `.team/config.json`.
- Use different toolsets for lead and teammates.

## Limits

- No cross-process file locking.
- Teammates are not fully long-running polling workers yet.
- No cancellation or forced shutdown tool.
- Multiple teammates still share one workspace.
- No full task assignment or approval protocol yet.
