# s15

## 目标

`s15` 的核心目标，是把 agent 从“只能临时拉起一个一次性 subagent”推进到“可以维护一组有名字、可重复唤醒的 persistent teammates”。

这一章对应 Python 原版里的核心思想是：

> teammates are persistent named workers with inboxes, not one-shot subagents

这里最容易混淆的点需要先分开：

- `s04` 的 subagent 是一次性子任务执行槽位
- `s15` 的 teammate 是长期存在的命名协作者

两者都可能跑工具，也都可能有自己的 loop，但语义不同。

- subagent：spawn -> 执行 -> 汇总 -> 销毁
- teammate：spawn -> 工作 -> idle -> 再次收到消息 -> 再工作

所以这一章的重点不是再做一套“并行任务执行器”，而是：

- teammate 如何拥有稳定身份
- teammate 如何通过 inbox 接收消息
- inbox 消息如何重新进入各自的 agent loop
- 团队状态如何持久化到磁盘

---

## 当前 Rust 版做了什么

当前 `s15` 可以概括成一句话：

**把 team 抽象成“持久化成员注册表 + JSONL mailbox + 可反复进入的独立 agent loop”。**

Rust 版最终收敛成了三层：

1. `SharedMessageBus` 负责 JSONL inbox 的发送、drain 和广播
2. `SharedTeammateManager` 负责 team config、成员状态和 teammate 生命周期
3. `LoopState` 负责在每轮模型调用前自动注入 inbox 消息

这样主线仍然很清楚：

1. lead 或 teammate 往某个 inbox 追加消息
2. 收件方下一轮 loop 开始前先 drain 自己的 inbox
3. 这些消息变成新的 `user` 上下文
4. 收件方继续作为一个普通 agent 处理后续工作

这就是这一章真正想教的东西：

- teammate 不是独立协议系统
- mailbox 也不是另一个 RPC 框架
- 它本质上只是“把外部消息重新注回已有 agent loop”

## 当前 Rust 版的结构

### 1. `MessageType / TeammateStatus`

Rust 版没有继续用随手写的字符串，而是把两个关键概念收敛成了显式 `enum`：

- `MessageType`
- `TeammateStatus`

其中：

- `MessageType` 表达 mailbox 里的消息种类
- `TeammateStatus` 表达成员当前的生命周期状态

它们同时派生了：

- `serde` 的序列化 / 反序列化
- `strum` 的 `Display` / `EnumString`

所以像：

- `message`
- `broadcast`
- `shutdown_request`
- `idle`
- `working`
- `shutdown`

这些概念现在都变成了有边界的类型，而不是零散约定。

这一点和前几章一样，属于很典型的 Rust 化收敛：

- Python 原版更偏教学上的直接表达
- Rust 版则尽量把状态空间明确成类型

### 2. `SharedMessageBus`：mailbox 是 append-only JSONL

Python 原版这一章最核心的实现是：

- 每个 teammate 一个 inbox 文件
- 发消息时 append 一行 JSON
- 读 inbox 时整批读出并清空

Rust 版保留了这个行为语义，没有把“通信”和“持久化”拆成两套不同真源。

当前 `SharedMessageBus` 负责三件事：

- `send`
- `read_inbox`
- `broadcast`

其中最重要的是 `read_inbox` 的 drain 语义：

- 先读出当前文件里所有消息
- 解析成 `InboxMessage`
- 再把文件清空

这意味着 inbox 不是“可回看聊天记录”，而是“待处理消息队列”。

这是这一章的关键语义边界：

- context 负责保留对话历史
- inbox 只负责暂存还没注入 loop 的外部消息

### 3. 锁：只做最小必要的进程内串行化

当前 Rust 版没有引入更复杂的文件锁协议，也没有为 mailbox 单独做跨进程协调。

现在的做法是：

- `SharedMessageBus` 内部持有一个进程内 `Mutex<()>`
- `send` 和 `read_inbox` 在进入文件临界区前先拿锁

这层锁的目标很明确：

- 防止同一进程内多个 teammate task 同时读写 JSONL 文件时交错

这里没有再往上做：

- `fs4` 文件锁
- 每个 inbox 单独一把细粒度锁
- channel + file 双缓冲

这是有意的简化。原因不是这些方案不能做，而是这一章的重点并不在 mailbox 的并发优化，而在“persistent teammates + inbox + loop reinjection”这条主线。

### 4. `TeamConfig / TeamMember`

Python 原版用 `.team/config.json` 维护团队成员列表。

Rust 版保持了同样的持久化形态，并显式建模成：

- `TeamConfig`
- `TeamMember`

其中成员记录至少包括：

- `name`
- `role`
- `status`

这让几个语义变得更清楚：

- “这个 teammate 是否已存在”
- “它当前是不是 idle”
- “再次 spawn 是重启一个已有成员，还是新建成员”

所以 Rust 版这里不是把 teammate 只当运行时 task handle，而是把它当成有身份的团队成员记录。

### 5. `SharedTeammateManager`：team 的持久状态和生命周期入口

当前 Rust 版里，`SharedTeammateManager` 是这一章最核心的聚合点。

它负责：

- 初始化 `.team/config.json`
- 维护成员状态
- `spawn` teammate
- 列出成员
- 在 teammate 退出时把状态切回 `idle`

这里最关键的行为是 `spawn`：

- 如果成员不存在，就创建一条新记录
- 如果成员已存在但状态是 `idle` 或 `shutdown`，就复用它的身份再次启动
- 如果成员还在 `working`，则拒绝重复启动

这正是 Python 原版里“persistent named workers”的语义核心：

- teammate 的 identity 是稳定的
- 每次工作不是创建一个匿名 worker
- 而是重新唤醒那个名字对应的成员

### 6. teammate loop：独立任务，但仍然是同一种 agent loop

Python 原版给每个 teammate 开一个线程，不断重复：

- 看 inbox
- 把消息塞进 context
- 调模型
- 执行工具
- 直到停止

Rust 版保留了这个行为结构，但实现方式改成了：

- `tokio::spawn(...)`
- 每个 teammate 一份独立 `LoopState`

也就是说：

- teammate 确实在独立异步任务里运行
- 但它用的仍然是和主 agent 同一套 loop 骨架

这点很重要，因为它说明：

- teammate 不是另一种 agent 实现
- 它只是同一种 loop 的不同实例
- 区别只在 system prompt、inbox owner 和工具集合

### 7. `LoopState`：在模型调用前自动注入 inbox

这一章真正把 mailbox 接进 agent 的关键，不在 manager，也不在 tool，而在 `LoopState`。

当前 Rust 版里，`LoopState::agent_loop()` 每轮在真正请求模型前都会先：

- 调 `inject_inbox_messages()`
- drain 当前 owner 的 inbox
- 把 inbox 内容包装成一条新的 `user` 消息塞进 context

这意味着：

- mailbox 不是旁路输入
- teammate 也不是收到消息后直接执行某个回调
- 一切最终都还是通过“消息进入上下文”来驱动模型

这和 Python 原版的教学意图是完全一致的。

### 8. 工具集合：lead 和 teammate 共用一套工具框架，但暴露不同能力

这一章 Rust 版没有把工具列表散落在 `main.rs` 和 manager 里，而是统一收进 `tool/mod.rs`：

- `leader_tools(...)`
- `teammate_tools(...)`

两边共享的基础工具仍然是：

- `bash`
- `read_file`
- `write_file`
- `edit_file`

在此基础上：

- lead 额外拥有 `spawn_teammate / list_teammates / send_message / read_inbox / broadcast`
- teammate 只拥有 `send_message / read_inbox` 这种团队协作最小能力

这样做的重点是把“谁能做什么”收成了清楚的工具边界，而不是靠 prompt 暗示。

## 和 Python 原版的对应关系

如果按教学主线对照，当前 Rust 版和 Python 原版是一致的：

1. 都有持久化 team config
2. 都有每个成员对应的 inbox 文件
3. 都允许 lead spawn 命名 teammate
4. 都让 teammate 通过自己的 loop 持续工作
5. 都把 inbox 消息重新注回上下文，而不是绕开 loop

所以这一章真正传达的思想仍然是：

- named worker
- mailbox
- persistent identity
- loop re-entry

而不是“如何做一个复杂的多 agent orchestration framework”。

---

## 和 Python 原版的相同点

### 1. teammate 都是“有名字的长期成员”

两边都强调：

- teammate 不是一次性 subagent
- 名字本身就是稳定身份
- 同名成员再次工作时，应该延续这份身份

### 2. inbox 都是 append + drain

两边都采用：

- 发消息时 append
- 读消息时 drain

所以 inbox 的语义始终是“待处理队列”，而不是完整聊天历史。

### 3. teammate loop 都会反复重入

两边都不是“收到一条消息，执行一次函数就结束”，而是：

- teammate 持续存在
- 在不同时间点多次重新参与工作

### 4. lead 和 teammate 都通过工具协作

两边都保留了最基础的协作模式：

- lead spawn teammate
- lead 给 teammate 发消息
- teammate 自己读 inbox 或回信
- lead 可以广播给全队

---

## 和 Python 原版的不同点

### 1. Rust 版更明确地把状态收敛成类型

Python 原版里很多状态是字典字段和字符串。

Rust 版把关键概念提升成了：

- `MessageType`
- `TeammateStatus`
- `InboxMessage`
- `TeamMember`
- `TeamConfig`

这样 mailbox、team config 和成员状态的边界都更清楚。

### 2. Rust 版没有再额外做 channel 作为消息主通道

在设计上其实有另一种常见选择：

- 通信层用 channel
- 持久化层用文件

当前 Rust 版没有走这条路，而是明确让 JSONL inbox 成为唯一消息真源。

原因是这一章最重要的语义是：

- 消息被写进 mailbox
- 收件方下一轮 drain mailbox

如果再并行维护一份 channel 队列，就会引入双缓冲一致性问题，而这不是本章重点。

### 3. Rust 版用 Tokio task，不再用线程

Python 原版用 `threading.Thread`。

Rust 版这里改成了 `tokio::spawn`，但语义仍然等价：

- 每个 teammate 有独立执行槽位
- 它们可以在主 loop 之外继续运行

这只是实现风格的 Rust/Tokio 化，不改变教学主线。

### 4. Rust 版把工具装配也做了职责分层

Python 原版这章的工具定义更偏教学直排。

Rust 版最终把工具装配分成：

- lead 的工具集合
- teammate 的工具集合

这样后面继续扩展团队能力时，不需要在多个地方同时改动。

---

## 这章真正新增的能力

从整个课程脉络看，`s15` 新增的不是“更多工具”，而是 agent 组织形式的变化。

前面章节大体还是：

- 一个主 agent
- 一些额外能力

而从 `s15` 开始，系统第一次具备了：

- 一个 lead
- 多个具名成员
- 成员间通过 mailbox 协作

这意味着接下来可以自然展开很多后续主题：

- approval protocol
- richer teammate policies
- team-level coordination
- shutdown / handoff / escalation

也就是说，`s15` 是后续多 agent 协作章节的基础地板。

---

## 当前实现的边界

虽然当前 Rust 版已经具备了 `s15` 的核心教学语义，但它仍然刻意保持了简化。

目前没有展开的部分包括：

- 跨进程级别的 mailbox 文件锁
- 更细粒度的每-inbox 锁
- teammate 自动休眠 / 唤醒协议
- 更复杂的 shutdown 协议
- 更正式的 mailbox request / response schema

这些都可以继续做，但不属于当前章节最小必要实现。

就本章而言，真正需要保住的是三件事：

1. teammate 是持久命名成员
2. mailbox 是 append + drain 的文件队列
3. mailbox 消息最终重新进入同一个 agent loop

只要这三点成立，`s15` 的教学主线就是完整的。

---

## 小结

`s15` 真正教的不是“多开几个 agent”，而是：

- 给 agent 引入稳定身份
- 用 mailbox 让它们异步协作
- 再把这些外部消息重新喂回已有 loop

当前 Rust 版用：

- `SharedMessageBus`
- `SharedTeammateManager`
- `LoopState::inject_inbox_messages()`

把这条主线落成了一个比较 Rusty、但语义上仍然和 Python 原版一致的实现。

所以这一章最重要的理解可以收成一句话：

**teammate 不是一次性任务，mailbox 不是旁路协议，它们最终都只是同一个 agent loop 的延伸。**
