# s15: Agent Teams

s04 的 subagent 是一次性的：主 agent 启动它，交给它一个子任务，拿到总结后它就结束。本章引入另一种协作形态：有名字、可重复唤醒、拥有 inbox 的 teammate。

Agent team 的核心不是复杂编排框架，而是把多个 agent loop 通过消息邮箱连接起来。

## 运行方式

```bash
cargo run -p s15_agent_teams
```

交互命令：

```text
/team
/inbox
exit()
```

运行后会使用：

```text
.team/
```

保存团队配置和 inbox 文件。

## 本章新增能力

- 新增 team lead。
- 新增 persistent teammate。
- 新增 `.team/config.json` 保存团队成员。
- 新增 `.team/inbox/<name>.jsonl` 保存成员 inbox。
- lead 可以 spawn teammate。
- agent 之间可以 `send_message`。
- lead 可以 `broadcast`。
- 每个 agent loop 在请求模型前自动注入自己的 inbox 消息。

## Subagent 和 Teammate 的区别

Subagent：

```text
spawn -> 执行一次子任务 -> 返回总结 -> 结束
```

Teammate：

```text
有稳定名字 -> 有 inbox -> 可以 idle -> 可以再次收到消息 -> 再次工作
```

teammate 更像团队成员，subagent 更像一次性任务执行者。

## MessageBus

每个成员有一个 inbox 文件：

```text
.team/inbox/lead.jsonl
.team/inbox/reviewer.jsonl
.team/inbox/implementer.jsonl
```

消息结构：

```rust
pub struct InboxMessage {
    pub message_type: MessageType,
    pub from_name: String,
    pub content: String,
    pub timestamp: f64,
}
```

消息类型包括：

```rust
Message,
Broadcast,
ShutdownRequest,
ShutdownResponse,
PlanApproval,
PlanApprovalResponse,
```

inbox 是待处理队列，不是聊天记录。`read_inbox()` 会读出所有消息并清空文件。

## LoopState 注入 inbox

s15 的 `LoopState` 新增：

```rust
pub message_bus: SharedMessageBus,
pub inbox_owner: String,
pub system_prompt: String,
pub max_rounds: usize,
```

每轮模型调用前执行：

```rust
self.inject_inbox_messages()?;
```

如果 inbox 有消息，会注入：

```xml
<inbox>
[
  ...
]
</inbox>
```

这意味着团队通信最终仍然是“消息进入上下文”，而不是绕过模型的外部回调。

## TeamConfig

团队状态保存为：

```text
.team/config.json
```

结构：

```rust
pub struct TeamConfig {
    pub team_name: String,
    pub members: Vec<TeamMember>,
}

pub struct TeamMember {
    pub name: String,
    pub role: String,
    pub status: TeammateStatus,
}
```

状态包括：

```rust
Idle,
Working,
Shutdown,
```

## spawn_teammate

工具输入：

```json
{
  "name": "reviewer",
  "role": "code reviewer",
  "prompt": "Review the current implementation and send findings to lead."
}
```

执行逻辑：

- 如果成员不存在，创建成员记录。
- 如果成员存在且是 `idle` 或 `shutdown`，复用这个名字重新启动。
- 如果成员正在 `working`，拒绝重复启动。
- 为成员注册 inbox。
- 用 `tokio::spawn` 启动 teammate loop。

## lead tools 和 teammate tools

lead 拥有：

- `spawn_teammate`
- `list_teammates`
- `send_message`
- `read_inbox`
- `broadcast`
- 基础文件和 bash 工具

teammate 拥有：

- `send_message`
- `read_inbox`
- 基础文件和 bash 工具

工具边界决定了不同角色能做什么。lead 负责组织团队，teammate 负责完成被分配的任务并沟通。

## send_message

输入：

```json
{
  "to": "reviewer",
  "content": "Please inspect src/lib.rs and report risks.",
  "msg_type": "message"
}
```

消息会被追加到对方 inbox JSONL 文件。

## broadcast

lead 可以向所有 teammate 广播：

```json
{
  "content": "Stop current exploration and report status."
}
```

发送者自己不会收到广播。

## Rust 实现要点

- 用 JSONL 文件实现简单 mailbox。
- 用 `Mutex<()>` 串行化同一进程内 inbox 读写。
- 用 `Arc` 共享 message bus 和 teammate manager。
- 用 `tokio::spawn` 为 teammate 启动独立 loop。
- 用 `.team/config.json` 持久化成员身份和状态。
- 用不同 toolset 表达 lead 和 teammate 的能力差异。

## 本章的局限

- inbox 没有跨进程文件锁。
- teammate 没有长期常驻轮询 loop，当前是按 spawn prompt 运行一段任务。
- 没有取消或强制 shutdown 工具。
- 多个 teammate 共享同一工作区，文件修改可能冲突。
- 没有任务分配协议和审批协议的完整实现。
