# s04: Subagent

s03 lets the agent maintain a todo list, but all work still happens in one context. For complex tasks, exploration can fill the main context with noisy details.

This chapter adds a `sub_agent` tool. The main agent can delegate a focused subtask to a fresh agent loop and receive a concise summary.

## Run

```bash
cargo run -p s04_subagent
```

## New Capabilities

- Add a `sub_agent` tool.
- Spawn a fresh `LoopState` for delegated work.
- Give the subagent its own context and toolset.
- Return the subagent's final summary to the main agent.
- Keep exploratory context out of the parent conversation.

## Why Subagents

Some tasks are useful but noisy:

- Inspecting many files.
- Searching for a pattern.
- Trying a small implementation approach.
- Gathering context before the main agent decides.

If all of that happens in the main context, the final reasoning context becomes large and unfocused.

A subagent gives the harness a simple isolation pattern:

```text
main agent
  -> calls sub_agent(prompt)
  -> child agent works in fresh context
  -> child returns summary
  -> main agent continues with only the summary
```

## Subagent Tool

The tool input is:

```json
{
  "prompt": "Inspect the tool module and summarize the dispatch flow."
}
```

The tool creates:

- A new LLM client.
- A new `LoopState`.
- A fresh context containing only the delegated prompt.
- A child toolset.

Then it runs the normal agent loop and extracts the final assistant message as the result.

## Fresh Context

The child agent does not inherit the parent context.

This is the key point. The parent chooses what to delegate through the prompt, and the child returns a compact summary.

That gives the parent:

- Less context growth.
- A cleaner summary of exploration.
- A way to separate parallel-looking concerns even in a simple CLI.

## Tool Boundary

The subagent gets a limited toolset:

- `bash`
- `read_file`
- `write_file`
- `edit_file`

It does not get `sub_agent` again. This prevents recursive uncontrolled delegation in the teaching version.

## Result Extraction

After the child loop finishes, the tool finds the most recent assistant text message and returns it:

```text
subagent final response -> parent tool_result
```

The parent sees the subagent as just another tool call, even though a whole agent loop ran behind it.

## Rust Notes

- A subagent is just another `LoopState`.
- The same loop can be reused for parent and child.
- The tool result is a summary string, not a shared context.
- The child toolset is intentionally smaller than the parent toolset.

## Limits

- Subagents run sequentially in this teaching version.
- No cancellation.
- No streaming output back to the parent.
- No structured result schema.
- No worktree isolation yet; parent and child share the same workspace.
