# s04: Subagent

s03 解决了“复杂任务需要计划”的问题，但还没有解决另一个常见问题：探索过程会污染主上下文。

比如用户让 agent 修改一个功能。主 agent 可能需要先搜索代码、阅读多个文件、尝试理解模块关系。这些探索信息对完成任务有用，但不一定都应该留在主对话里。上下文越长，模型越容易被细节干扰，也越容易浪费 token。

本章加入 `task` 工具，让主 agent 可以启动一个子代理。子代理拥有独立上下文，完成任务后只把总结返回给主 agent。

## 运行方式

```bash
cargo run -p s04_subagent
```

## 本章新增能力

- 新增 `task` 工具，用于启动子代理。
- 子代理和主代理共享文件系统，但不共享对话历史。
- 子代理使用受限工具集，避免无限递归创建子代理。
- 将 agent loop 抽到 `lib.rs`，方便主代理和子代理复用。
- 给 `Tool` trait 增加 `Send + Sync` 约束。
- 改进写文件路径处理，允许写入尚不存在的新文件。

## 为什么需要子代理

子代理的核心价值是上下文隔离。

主 agent 需要保留的是：

- 用户目标。
- 已经做出的关键决策。
- 最终需要执行的修改。
- 子任务的结论。

子 agent 可以承担的是：

- 搜索代码。
- 阅读大量文件。
- 比较多个实现方案。
- 总结某个模块的行为。

完成后，子代理只返回摘要。主上下文不需要保存所有中间噪声。

## 复用 LoopState

s04 把 agent loop 从 `main.rs` 移到 `lib.rs`：

```rust
pub struct LoopState {
    pub client: AnthropicClient,
    pub context: Vec<Message>,
    pub tools: HashMap<String, Box<dyn Tool>>,
    pub system_prompt: String,
    pub max_round: usize,
}
```

这样主代理和子代理都可以使用同一个循环，只需要传入不同的：

- system prompt
- toolset
- max_round
- 初始上下文

这一步是从“单程序 demo”走向“可复用 harness”的开始。

## 主代理工具集与子代理工具集

[`src/tool/mod.rs`](./src/tool/mod.rs) 里定义了两个工具集合：

```rust
pub fn agent_tools() -> HashMap<String, Box<dyn Tool>>
pub fn subagent_tools() -> HashMap<String, Box<dyn Tool>>
```

主代理拥有：

- `bash`
- `read_file`
- `write_file`
- `edit_file`
- `task`

子代理拥有：

- `bash`
- `read_file`
- `write_file`
- `edit_file`

子代理没有 `task` 工具。这样可以避免子代理继续无限创建子代理，使执行树失控。

## task 工具

`task` 工具的 schema：

```json
{
  "prompt": "Find where configuration is loaded and summarize it.",
  "description": "Inspect config loading"
}
```

实现流程：

```text
主 agent 调用 task
  -> 创建新的 LLM client
  -> 创建子代理工具集
  -> 构造子代理 system prompt
  -> 把 prompt 作为子代理第一条用户消息
  -> 运行子代理 loop
  -> 提取子代理最后一条 assistant 文本
  -> 作为 tool_result 返回主 agent
```

核心函数是：

```rust
async fn sub_agent_loop(prompt: &str, description: Option<&str>) -> Result<String>
```

## 子代理的上下文隔离

子代理启动时只收到一条用户消息：

```rust
state.context.push(Message::new_text(Role::User, prompt));
```

它看不到主代理之前的完整对话。这是刻意设计的。

如果子任务需要背景信息，主代理必须把必要信息写进 `prompt`。这会促使主代理做信息压缩和任务澄清，而不是把整段历史丢给子代理。

## max_round

子代理创建时使用：

```rust
let mut state = LoopState::new(client, tools, system_prompt, 30);
```

`max_round` 限制子代理最多运行 30 轮，避免子任务陷入无限工具调用。

主代理则使用：

```rust
usize::MAX
```

教学版本里主代理不限制轮数，后续章节会引入更细的权限和任务控制。

## Send + Sync

s04 中的 `Tool` trait 改成：

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    async fn invoke(&mut self, input: &Value) -> Result<String>;
    fn name(&self) -> Cow<'_, str>;
    fn tool_spec(&self) -> ToolSpec;
}
```

这是为更复杂的异步执行做准备。工具被放进 trait object 后，如果未来要跨线程、放入异步任务或共享到更复杂的调度结构中，`Send + Sync` 会成为必要约束。

## 写文件路径修正

s02/s03 的 `safe_path()` 对新文件不友好，因为 `canonicalize()` 要求路径已经存在。

s04 增加了：

```rust
fn safe_path_allow_missing(path: &str) -> Result<std::path::PathBuf>
```

它会先 canonicalize 父目录，再拼回文件名。这样既允许创建新文件，又能继续阻止路径逃逸 workspace。

## 本章的局限

- 子代理类型还没有细分，例如 explore/code/review/test。
- 子代理权限和主代理权限差别不大，只是去掉了 `task`。
- 子代理结果只是文本摘要，没有结构化产物。
- 子代理和主代理共享同一工作区，修改仍可能互相影响。

这些问题会在后续章节里继续演进，比如权限系统、多 agent 协议和 worktree 任务隔离。
