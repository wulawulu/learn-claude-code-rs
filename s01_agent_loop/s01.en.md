# s01: Minimal Agent Loop

This chapter builds the smallest runnable coding agent. It has only one tool: `bash`. Even with such a small capability set, it already contains the core loop of an agent harness:

```text
user input -> model reasoning -> tool call -> execute tool -> return tool result -> model continues -> final response
```

After you understand this chapter, later features such as tools, task planning, subagents, skill loading, and permissions are all extensions around this loop.

## Run

Configure `.env` at the repository root:

```bash
ANTHROPIC_API_KEY=your_api_key
ANTHROPIC_BASE_URL=your_anthropic_compatible_base_url
```

Run:

```bash
cargo run -p s01_agent_loop
```

Exit the CLI:

```text
exit()
```

## Goals

- Build an interactive CLI agent.
- Call the model API and keep multi-turn message context.
- Expose `bash` as a model-callable tool.
- Detect `tool_use`, execute the tool, and write `tool_result` back into context.
- Print the final text response when the model stops calling tools.

## Core Structure

The entry point is [`src/main.rs`](./src/main.rs).

### 1. System Prompt

```rust
const SYSTEM: &str = r#"You are a coding agent.
Use bash to inspect and change the workspace. Act first, then report clearly.
"#;
```

The prompt tells the model three things:

- It is a coding agent.
- It can use `bash` to inspect and change the current workspace.
- It should act first, then report clearly.

Early in a harness, keep the prompt small. First give the model a clear capability boundary, then observe what the runtime still needs.

### 2. LoopState

```rust
struct LoopState {
    client: AnthropicClient,
    pub context: Vec<Message>,
    turn_count: usize,
    transition_reason: Option<String>,
}
```

`LoopState` stores runtime state:

- `client`: the model client.
- `context`: the full conversation history, including user messages, assistant messages, and tool results.
- `turn_count`: current loop count.
- `transition_reason`: why the loop continued, currently used for tool result transitions.

In an agent harness, `context` is the most important state. Every model call depends on it to understand what has happened.

### 3. Tool Definition

`get_tools()` returns the tool schema visible to the model:

```rust
Tool {
    name: "bash".to_string(),
    description: Some("Run a shell command in the current workspace.".to_string()),
    input_schema: serde_json::json!({
        "type": "object",
        "properties": {
            "command": { "type": "string" }
        },
        "required": ["command"]
    }),
}
```

A tool has two parts:

- A schema shown to the model: name, description, and input shape.
- Local execution logic: the Rust code that actually runs the tool.

The model only emits structured `tool_use`; it never executes commands directly. The harness decides whether and how to execute them, and what result to return.

### 4. bash Execution

`run_bash()` uses `tokio::process::Command` to run shell commands asynchronously:

- Blocks obviously dangerous commands with a small denylist.
- Uses `sh -c` so shell syntax works.
- Captures stdout and stderr.
- Sets a 120 second timeout.
- Truncates long output to avoid filling the context.

This is not a full sandbox. It is the minimum safety boundary for a teaching version. Later chapters add permission checks and isolation.

### 5. One Reasoning Turn

`run_one_turn()` is the key function:

```text
build request
  -> call model
  -> append assistant message to context
  -> if there is no tool request, stop
  -> if there is a tool request, execute it
  -> append tool_result as a user message
  -> continue
```

Why is `tool_result` written as a `User` message? In the message protocol, a tool result is feedback from the outside environment. The model sees that feedback in the next turn and decides what to do next.

### 6. Agent Loop

```rust
async fn agent_loop(state: &mut LoopState) -> Result<()> {
    while run_one_turn(state).await? {}
    Ok(())
}
```

This loop is the core of the minimal harness.

As long as the model returns `tool_use`, the harness executes tools and returns results. When the model returns normal text, the loop ends and prints the final response.

## Rust Notes

- `tokio` supports async model calls and async command execution.
- `anyhow::Result` keeps error propagation simple.
- `dotenvy` loads API configuration from `.env`.
- `inquire` provides simple interactive input.
- `serde_json::json!` builds the tool input schema.

## Limits

- Only one `bash` tool, hard-coded in the main file.
- File operations rely on shell commands instead of structured tools.
- Tool dispatch only supports `bash`, so adding tools will make the loop messy.
- Safety is coarse and has no real permission prompt.
- Context grows forever and is not compacted.
