# s01: 最小 Agent Loop

本章实现一个最小可运行的 coding agent。它只有一个工具：`bash`。虽然能力很少，但已经包含 agent harness 最核心的闭环：

```text
用户输入 -> 模型推理 -> 工具调用 -> 执行工具 -> 工具结果回填 -> 模型继续推理 -> 最终回复
```

理解这一章之后，后面的工具系统、任务规划、子代理、技能加载都只是围绕这个循环继续扩展。

## 运行方式

在仓库根目录配置 `.env`：

```bash
ANTHROPIC_API_KEY=your_api_key
ANTHROPIC_BASE_URL=your_anthropic_compatible_base_url
```

运行：

```bash
cargo run -p s01_agent_loop
```

退出交互：

```text
exit()
```

## 本章目标

- 建立一个交互式 CLI agent。
- 调用模型 API，并维护多轮消息上下文。
- 将 `bash` 暴露为模型可调用的工具。
- 识别模型返回的 `tool_use`，执行工具，再把 `tool_result` 写回上下文。
- 在模型停止调用工具时，把最终文本回复给用户。

## 核心结构

入口在 [`src/main.rs`](./src/main.rs)。

### 1. System Prompt

```rust
const SYSTEM: &str = r#"You are a coding agent.
Use bash to inspect and change the workspace. Act first, then report clearly.
"#;
```

这里的 prompt 很短，只告诉模型三件事：

- 它是一个 coding agent。
- 它可以使用 `bash` 检查和修改当前 workspace。
- 它应该优先行动，完成后清楚汇报。

早期阶段不要把 prompt 写得太复杂。一个好的 harness 应该先让模型拥有清晰的能力边界，再观察真实行为中缺什么。

### 2. LoopState

```rust
struct LoopState {
    client: AnthropicClient,
    pub context: Vec<Message>,
    turn_count: usize,
    transition_reason: Option<String>,
}
```

`LoopState` 保存 agent 运行过程中的状态：

- `client`：模型客户端。
- `context`：完整对话历史，包括用户消息、模型消息、工具结果。
- `turn_count`：当前循环轮数。
- `transition_reason`：本轮为什么继续，当前主要用于记录工具结果回填。

在 agent harness 中，`context` 是最重要的状态。模型每次推理都依赖它来理解“已经发生了什么”。

### 3. 工具定义

`get_tools()` 返回模型可见的工具 schema：

```rust
Tool {
    name: "bash".to_string(),
    description: Some("Run a shell command in the current workspace.".to_string()),
    input_schema: serde_json::json!({
        "type": "object",
        "properties": {
            "command": {
                "type": "string"
            }
        },
        "required": ["command"]
    }),
}
```

工具由两部分组成：

- 给模型看的 schema：名称、描述、参数结构。
- 本地执行逻辑：真正运行命令的 Rust 代码。

模型只会生成结构化的 `tool_use`，不会直接执行命令。是否执行、如何执行、返回什么结果，都由 harness 控制。

### 4. bash 执行

`run_bash()` 使用 `tokio::process::Command` 异步执行 shell 命令：

- 通过简单黑名单拦截明显危险命令。
- 使用 `sh -c` 让命令支持 shell 语法。
- 捕获 stdout 和 stderr。
- 设置 120 秒超时。
- 截断过长输出，避免把上下文撑爆。

这不是完整沙箱，只是教学版本里的最小安全边界。后续章节会继续引入权限系统和隔离机制。

### 5. 单轮推理

`run_one_turn()` 是本章最关键的函数：

```text
构造请求
  -> 调用模型
  -> 把 assistant 消息加入 context
  -> 如果模型没有请求工具，结束
  -> 如果模型请求工具，执行工具
  -> 把 tool_result 作为 user 消息加入 context
  -> 继续下一轮
```

为什么工具结果要作为 `User` 消息写回？因为从消息协议的角度看，工具结果是外部环境对模型动作的反馈。模型下一轮看到这个反馈后，再决定是否继续行动。

### 6. Agent Loop

```rust
async fn agent_loop(state: &mut LoopState) -> Result<()> {
    while run_one_turn(state).await? {}
    Ok(())
}
```

这个循环就是最小 agent harness 的核心。

只要模型继续返回 `tool_use`，harness 就继续执行工具并回填结果；当模型返回普通文本时，循环结束，把最终回复展示给用户。

## Rust 实现要点

- 使用 `tokio` 支持异步模型调用和异步命令执行。
- 使用 `anyhow::Result` 简化错误传播。
- 使用 `dotenvy` 从 `.env` 加载 API 配置。
- 使用 `inquire` 做简单交互式输入。
- 使用 `serde_json::json!` 构造工具 input schema。

## 本章的局限

- 只有一个 `bash` 工具，工具逻辑硬编码在主文件里。
- 文件读写依赖 shell 命令，缺少结构化接口。
- 工具调度只支持 `bash`，扩展新工具会让主循环越来越乱。
- 安全策略很粗糙，没有真正的权限确认。
- 上下文没有压缩，长任务会持续增长。

这些问题会在后续章节逐步解决。
