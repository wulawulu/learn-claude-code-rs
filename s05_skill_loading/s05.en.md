# s05: Skill Loading

s04 lets the agent delegate work to subagents. The next problem is specialized knowledge: it should not all be stuffed into the system prompt.

If every checklist, convention, workflow, and domain rule is loaded up front, the agent starts with a lot of irrelevant context. A better pattern is to show only short skill summaries first, then load full skill content when the task needs it.

This chapter adds a `load_skill` tool and a `SkillRegistry`.

## Run

```bash
cargo run -p s05_skill_loading
```

## New Capabilities

- Scan `skills/` for `SKILL.md`.
- Parse frontmatter and extract `name` and `description`.
- List available skills in the system prompt.
- Add a `load_skill` tool to load full skill content by name.
- Share `SkillRegistry` between agent state and the tool through `Arc`.

## Why Skills

Tools answer: what can the agent do?

Skills answer: how should the agent behave for a class of tasks?

Examples:

- Code review checklist.
- MCP server workflow.
- PDF processing commands and library choices.
- Team-specific code style.
- Business-domain procedures.

These are not always needed. With skills, the model sees short summaries and loads full content only when relevant.

## Skill File Format

A skill is a `SKILL.md` file with optional YAML frontmatter:

```markdown
---
name: code-review
description: Perform thorough code reviews with security, performance, and maintainability analysis.
---

# Code Review Skill

...
```

The frontmatter generates the short skill summary. The body is loaded into context when requested. If `name` is missing, the directory name is used.

## SkillRegistry

The core type is in [`src/skill.rs`](./src/skill.rs):

```rust
pub struct SkillRegistry {
    skills_dir: PathBuf,
    skills: HashMap<String, SkillDocument>,
}
```

Loading flow:

```text
walk skills_dir
  -> find SKILL.md files
  -> read content
  -> parse frontmatter
  -> build SkillDocument
  -> insert by name
```

The implementation uses `walkdir`, so each skill directory can also contain references, scripts, and other local files. This chapter only loads `SKILL.md`.

## Frontmatter Parsing

`parse_frontmatter()` supports LF and CRLF:

```rust
let text = text.replace("\r\n", "\n");
```

Then it parses common Markdown frontmatter:

```text
---
name: ...
description: ...
---

body
```

The frontmatter is deserialized with `serde_yaml`.

## System Prompt Summaries

At startup, the agent builds a system prompt containing skill summaries:

```rust
let system = format!(
    r#"You are a coding agent at {}.
Use load_skill when a task needs specialized instructions before you act.

Skills available:
    {}
"#,
    std::env::current_dir()?.display(),
    self.skill_registry.describe_available()
);
```

Only summaries are included, not full skill bodies.

## load_skill Tool

Input:

```json
{
  "name": "code-review"
}
```

The tool returns the skill content wrapped in an XML-like block:

```xml
<skill name="code-review">
...
</skill>
```

This makes the loaded content easy for the model to distinguish from ordinary user text.

## Why Arc

`SkillRegistry` is needed by:

- `LoopState`, to render skill summaries into the system prompt.
- `LoadSkillTool`, to load full skill content.

The registry is loaded once at startup and then read-only, so `Arc<SkillRegistry>` is a simple shared ownership model.

## Tool vs Skill

Tool: executable capability.

```text
read_file, write_file, bash, edit_file
```

Skill: loadable knowledge.

```text
code review guide, MCP workflow, PDF processing guide
```

A mature agent harness usually needs both.

## Limits

- Skills are scanned only at startup.
- `load_skill` only loads `SKILL.md`, not references or scripts.
- The harness does not track whether a skill was already loaded.
- No skill permissions, versions, or dependencies.
- Skill content still enters ordinary context and will later need compaction.
