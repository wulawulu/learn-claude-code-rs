# s11

## 目标

`s11` 的核心目标是把 agent 从“报错就停”推进到“先识别失败类型，再决定怎么继续”。

这一章对应 Python 原版里的核心思想是：

> 错误不是例外，而是主循环必须预留出来的一条正常分支。

到了 `s10`，系统已经不再是一个只会聊天的最小 demo，而是一个真的会调用工具、读写文件、持续推进任务的程序。

这时候真正的问题就出现了：

- 模型输出写到一半被截断
- 上下文过长，请求装不进窗口
- API 临时抖动，超时、限流或连接失败

如果没有恢复逻辑，主循环会在第一个错误上直接停住。

所以这一章的目标不是“隐藏错误”，而是：

- 先判断失败属于哪一类
- 如果可以恢复，就执行有限次数的补救动作
- 如果恢复也失败，再明确结束

---

## 当前 Rust 版做了什么

当前 `s11` 做的事情可以概括成一句话：

**把错误恢复接进主循环，但不让主循环失去可读性。**

对应到实现上，Rust 版保留了 Python 原版的三条恢复路径：

1. 输出被截断时续写
2. 上下文过长时压缩再试
3. 临时 API / 连接错误时退避重试

同时又做了两层收敛：

- 恢复预算只在当前 query 内生效，不跨整个 session 累积
- 恢复逻辑拆成 helper，主循环仍然保留清晰的执行主线

---

## 当前主循环的形状

在最终实现里，`agent_loop()` 仍然保持了很清楚的骨架：

1. 调用模型
2. 如果请求失败，尝试恢复
3. 如果响应被截断，尝试续写
4. 如果是正常 `tool_use`，执行工具
5. 如果上下文太长，主动 compact

也就是说，主循环现在不是“所有错误都堆在一起判断”，而是：

- 正常执行路径仍然可读
- 恢复只是主循环里的正常分支之一
- 真正复杂的恢复动作放在独立 helper 里

这和 Python 原版的教学重点是一致的：

- 重点不是异常类名本身
- 而是系统在失败后怎么继续往下走

---

## 恢复状态做了什么

Rust 版保留了和 Python 原版一致的核心想法：
每条恢复路径都要有自己的预算。

当前实现里这部分是 `RecoveryState`：

- `continuation_attempts`
- `compact_attempts`
- `transport_attempts`

它的作用很明确：

- 防止无限循环
- 让续写、压缩、退避三条路径各算各的次数

这里还有一个很重要的取舍：

- 预算是 **per query**
- 每次 `agent_loop()` 开始都会重新初始化

这意味着：

- 同一个用户 query 里，恢复次数会被限制
- 但不会因为前一轮对话已经触发过 compact，就把下一轮正常 query 直接判死

这个选择比把预算做成全局 session 状态更合理，也更符合教学目标。

---

## 三条恢复路径分别怎么落地

### 1. 输出截断后续写

当响应的 `stop_reason == max_tokens` 时，Rust 版会：

- 增加 `continuation_attempts`
- 注入一条明确的 continuation message
- 回到主循环继续请求

这条 continuation message 保留了 Python 原版的核心要求：

- 直接继续
- 不要 recap
- 不要重复
- 必要时从半句中间接着写

这里最重要的不是“继续”两个字，而是：

- 明确告诉模型不要重来
- 明确告诉模型不要重复之前已经输出的内容

### 2. prompt 太长时先 compact

如果 API 错误文本表明：

- prompt 太长
- overlong prompt
- context length 超限

Rust 版会：

- 增加 `compact_attempts`
- 调用已有的 `compact_history()`
- 然后回到主循环继续

这里有一个关键点：

- `s11` 没有重新实现 compact 逻辑
- 而是直接复用当前 chapter 里已经存在的 `compact_history()`

也就是说，这一章不再重复讲“怎么总结历史”，而是强调：

- 当错误表明上下文装不下时
- 主循环知道可以通过 compact 来恢复

### 3. 临时传输错误时退避

如果错误文本看起来是临时 transport / API 抖动，例如：

- timeout
- rate limit
- unavailable
- connection
- overloaded

Rust 版会：

- 增加 `transport_attempts`
- 根据 attempt 计算 backoff delay
- `sleep` 一小段时间
- 再回到主循环重试

这里保留了 Python 原版的指数退避思路：

- 第一次稍等
- 后面等待时间逐步变长
- 同时附带一点 jitter

重点不是公式本身，而是：

- 不要立刻原地狂打
- 给临时拥堵一点恢复时间

---

## 主动 compact 仍然保留

除了“请求失败后再 compact”这条恢复路径，当前 Rust 版还保留了一条主动 compact：

- 每轮工具执行完成后估算上下文大小
- 如果超过阈值，就直接 `compact_history()`

这里刻意保持了和 Python 原版一致的最小逻辑：

- 只要超过阈值就压缩
- 不再引入额外分类判断

这么做反而更清楚，因为这一条的目的不是“错误恢复分类”，而是：

- 在问题真正发生前，主动降低上下文继续膨胀的风险

也可以理解成：

- reactive compact：因为请求已经失败，所以压缩
- proactive compact：因为上下文已经明显过大，所以先压缩

这两种 compact 在语义上不同，但在实现上都复用了同一个 `compact_history()`。

---

## 为什么 Rust 版没有照搬 Python 的写法

Python 原版的 teaching demo 很强调：

- 先写一个统一的恢复选择器
- 再根据 `kind` 去决定 continue / compact / backoff / fail

这对教学很有帮助，因为它把“分类”和“动作”拆得很直观。

但 Rust 版最终没有完全照抄这个组织方式，而是做了一个更贴近当前代码结构的收敛：

- 保留同样的三条恢复路径
- 保留同样的恢复预算思想
- 但不强行把所有恢复都塞进一个统一分类器

原因很简单：

- `max_tokens` 来自正常响应里的 `stop_reason`
- `prompt too long` / `timeout` 之类来自 `create_message()` 的错误

它们虽然都属于“恢复”，但触发位置并不相同。

所以 Rust 版最终更强调：

- 对外呈现的恢复模型是统一的
- 但代码内部允许按照更自然的入口组织

这不是偏离原版，而是一次实现层面的收敛。

---

## 和 Python 原版的相同点

### 1. 都保留了三条最小恢复路径

两边都明确支持：

- `max_tokens` 后续写
- prompt 太长后 compact
- 临时 API / transport 错误后 backoff

### 2. 都要求每条路径有独立预算

这一点非常重要，因为没有预算就可能无限恢复。

### 3. 都把恢复看成主循环的一部分

恢复不是外围补丁，不是简单 `try/except`。

两边都在强调：

- 主循环必须知道失败后怎么续下去

---

## 和 Python 原版的不同点

### 1. Rust 版直接复用了已有的 `compact_history()`

Python 原版在这一章自己定义了 `auto_compact()`。

Rust 版没有重复发明同一套东西，而是直接复用当前 chapter 已有的 compact 能力。

这让这一章更聚焦在：

- 什么时候触发 compact

而不是再讲一遍：

- compact 本身怎么实现

### 2. Rust 版把恢复预算限定在单次 query 内

这是当前 Rust 版很重要的一个工程取舍。

如果把 compact / continuation / transport 预算做成整个 session 共享状态，会出现一个很糟糕的问题：

- 上一轮 query 的恢复次数，影响下一轮本来正常的新 query

所以当前实现把恢复状态限定在一次 `agent_loop()` 调用里。

### 3. Rust 版更强调“保持主循环骨架清晰”

Python 教学版更强调展示恢复分支本身。

Rust 版则多做了一步收敛：

- 模型调用仍然直接留在 `agent_loop()`
- 错误恢复动作抽到 helper
- 主循环仍然保留“调用模型 -> 处理响应 -> 工具执行”的主线

这使得代码既能讲恢复，也不会把主线打散。

---

## 当前实现的价值

从教学上看，当前 `s11` 讲清楚了三件事：

1. 错误恢复不是把异常吞掉，而是有限预算下的继续执行。
2. 不同失败类型应该走不同恢复路径，而不是一把抓。
3. 恢复逻辑应该接进主循环，但不能把主循环本身写乱。

所以如果用一句话总结当前 `s11`：

> Python 原版强调“错误先分类，恢复再执行”，Rust 版则把这件事落实成“主循环知道什么时候该继续，并把恢复动作收敛成可读的控制流”。
