# s11: Error Recovery

到 s10 为止，agent 已经有工具、memory、skill 和结构化 prompt。但真实运行中还会遇到很多失败：模型输出被截断、上下文太长、API 临时不可用、网络超时。

如果 agent loop 遇到任何错误就直接退出，长任务体验会很差。本章把错误恢复接入主循环，让 agent 能对常见可恢复错误做有限次数的补救。

## 运行方式

```bash
cargo run -p s11_error_recovery
```

## 本章新增能力

- 识别 `max_tokens` 截断并请求模型续写。
- 识别 prompt/context 过长错误并压缩历史后重试。
- 识别临时传输错误并指数退避重试。
- 为每类恢复路径设置独立预算，避免无限循环。
- 在工具执行后主动估算上下文大小，超过阈值时 compact。

## 为什么错误恢复属于主循环

Agent harness 的主循环不只是“调用模型”。它还要管理失败后的走向。

典型恢复路径：

```text
模型输出太长 -> 追加 continuation message -> 继续请求
上下文太长 -> compact history -> 继续请求
临时网络/API 错误 -> backoff -> 继续请求
不可恢复错误 -> 明确停止
```

这些不是外围补丁，而是 agent loop 的正常分支。

## RecoveryState

```rust
struct RecoveryState {
    continuation_attempts: u32,
    compact_attempts: u32,
    transport_attempts: u32,
}
```

每类恢复路径都有自己的计数器。当前最大恢复次数是：

```rust
const MAX_RECOVERY_ATTEMPTS: u32 = 3;
```

恢复预算只在一次 `agent_loop()` 调用内生效。也就是说，每个用户 query 都有自己的恢复预算，不会因为上一轮触发过恢复而影响下一轮。

## 输出截断续写

当模型响应的 `stop_reason` 是 `MaxTokens` 时：

```rust
if matches!(response.stop_reason, Some(StopReason::MaxTokens))
    && self.handle_max_tokens_recovery(&mut recovery)
{
    continue;
}
```

恢复动作是向上下文追加：

```text
Output limit hit. Continue directly from where you stopped.
No recap, no repetition. Pick up mid-sentence if needed.
```

这条消息的重点是避免模型重新总结或重复已经输出的内容。

## 上下文过长 compact

当 API 错误文本看起来像 prompt/context 过长时：

```rust
is_prompt_too_long_error(&error_text)
```

会调用：

```rust
self.compact_history().await
```

`compact_history()` 会把当前对话总结成一条新的 continuation context，然后替换掉原来的长历史。

这是一种 reactive recovery：请求已经失败，所以先压缩再试。

## 主动 compact

工具执行后还会调用：

```rust
self.maybe_auto_compact().await?;
```

如果上下文估算超过：

```rust
const CONTEXT_THRESHOLD_CHARS: usize = 50_000;
```

就主动压缩。

这是一种 proactive recovery：在真正报错前降低上下文继续膨胀的风险。

## 临时错误退避

当错误文本包含：

- `timeout`
- `rate limit`
- `unavailable`
- `connection`
- `overloaded`
- `broken pipe`

等特征时，会进入 transport recovery。

等待时间由 `backoff_delay()` 计算：

```rust
base * 2^attempt + jitter
```

并限制最大等待：

```rust
const BACKOFF_MAX_DELAY_SECS: f64 = 30.0;
```

这样可以避免 API 暂时拥堵时立刻重复请求。

## LoopControl

恢复 helper 返回：

```rust
enum LoopControl {
    Continue,
    Stop,
}
```

这让主循环可以明确知道恢复动作之后应该继续还是结束。

## Rust 实现要点

- 把恢复状态限制在单次 query 内。
- 用 helper 函数识别不同错误类型。
- 用 `thread::sleep` 做简单退避等待。
- 用 `StopReason::MaxTokens` 处理正常响应里的截断。
- 用错误字符串分类处理 API 请求失败。

## 本章的局限

- 错误分类依赖字符串匹配，不够精确。
- `thread::sleep` 会阻塞当前线程，后续可改成异步 sleep。
- compact 摘要质量依赖模型。
- 没有把恢复事件写入审计日志。
- 没有针对工具失败做自动重试策略。
