# s02: Modular Tool Use

s01 has one hard-coded `bash` tool. It demonstrates the basic agent loop, but it does not scale. Once we add `read_file`, `write_file`, `edit_file`, search, network calls, or database queries, putting all tool logic in the main loop quickly becomes unmaintainable.

This chapter turns tools into a Rust abstraction that can be registered, dispatched, and extended.

## Run

```bash
cargo run -p s02_tool_use
```

## New Capabilities

- Define a unified `Tool` trait.
- Manage tools with `HashMap<String, Box<dyn Tool>>`.
- Add structured file tools:
  - `read_file`
  - `write_file`
  - `edit_file`
- Move tool execution out of the main loop.
- Add `normalize_messages()` so message history better satisfies the model API requirements for tool use.

## Tool Trait

The core abstraction is in [`src/tool/mod.rs`](./src/tool/mod.rs):

```rust
#[async_trait]
pub trait Tool {
    async fn invoke(&self, input: &Value) -> Result<String>;
    fn name(&self) -> Cow<'_, str>;
    fn tool_spec(&self) -> ToolSpec;
}
```

A tool provides:

- `invoke()`: execute local logic with model-generated JSON input.
- `name()`: return the tool name.
- `tool_spec()`: return the schema shown to the model.

Rust traits cannot directly use `async fn` as trait objects in this shape, so the chapter uses `async_trait`. This lets different tools live behind `Box<dyn Tool>`.

## Tool Registration

```rust
pub fn toolset() -> HashMap<String, Box<dyn Tool>> {
    HashMap::from([
        ("bash".to_string(), bash_tool()),
        ("read_file".to_string(), read_file_tool()),
        ("write_file".to_string(), write_file_tool()),
        ("edit_file".to_string(), edit_file_tool()),
    ])
}
```

The main loop no longer cares about concrete tool types. It stores:

```rust
tools: HashMap<String, Box<dyn Tool>>
```

When the model returns `tool_use`, the harness looks up the tool by name and calls it:

```rust
let Some(tool) = self.tools.get_mut(name) else {
    return format!("Unknown tool: {name}");
};

match tool.invoke(input).await {
    Ok(output) => output,
    Err(e) => format!("Error invoking tool {}: {}", name, e),
}
```

To add a new tool, implement `Tool` and register it in `toolset()`. The core loop does not need to change.

## File Tools

### read_file

Reads a UTF-8 text file and supports an optional `limit`:

```json
{
  "path": "src/main.rs",
  "limit": 120
}
```

Returned content is truncated to 50,000 characters to avoid polluting context with huge files.

### write_file

Writes file content:

```json
{
  "path": "notes/demo.md",
  "content": "# Demo\n"
}
```

In this chapter, `safe_path()` uses `canonicalize()`, which is not friendly to missing file paths. Later chapters fix this with `safe_path_allow_missing()`.

### edit_file

Performs an exact replacement:

```json
{
  "path": "src/main.rs",
  "old_text": "old",
  "new_text": "new"
}
```

It replaces only the first match. This is more controlled than asking the model to rewrite a whole file.

## Path Boundary

`safe_path()` restricts file tools to the current workspace:

```rust
let cwd = std::env::current_dir()?;
let full = cwd.join(path).canonicalize()?;

if !full.starts_with(&cwd) {
    return Err(anyhow::anyhow!("Path escapes workspace"));
}
```

The model may suggest a path, but the harness decides whether it is allowed.

## Message Normalization

The model API expects `tool_use` and `tool_result` messages to match. If execution is interrupted or context is stitched together, history may contain an orphan `tool_use`.

`normalize_messages()`:

- Adds a `(cancelled)` `tool_result` for a `tool_use` without a result.
- Merges consecutive messages with the same role.

This kind of bookkeeping is small but important for long-running agents.

## Rust Notes

- Use a trait to express the tool interface.
- Use trait objects for heterogeneous tool collections.
- Use `HashMap` for name-based dispatch.
- Use `serde_json::Value` for dynamic model input.
- Use `Cow<'_, str>` to avoid unnecessary string allocation.

## Limits

- No permission checks.
- No lifecycle hooks.
- `Tool::invoke(&self, ...)` is not ideal for stateful tools.
- Creating missing files is awkward.
- No task planning yet.
