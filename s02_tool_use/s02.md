# s02: 模块化 Tool Use

s01 只有一个硬编码的 `bash` 工具。它能说明 agent loop 的基本形态，但不适合继续扩展。一旦我们加入 `read_file`、`write_file`、`edit_file`、搜索、网络请求、数据库查询等能力，所有工具逻辑都堆在主循环里会很快失控。

本章的目标是把“工具”变成一个可注册、可派发、可扩展的 Rust 抽象。

## 运行方式

```bash
cargo run -p s02_tool_use
```

## 本章新增能力

- 定义统一的 `Tool` trait。
- 用 `HashMap<String, Box<dyn Tool>>` 管理工具集合。
- 新增结构化文件工具：
  - `read_file`
  - `write_file`
  - `edit_file`
- 将工具执行从主循环中拆出来。
- 增加 `normalize_messages()`，让消息历史更符合模型 API 对 tool use 的要求。

## Tool trait

核心抽象在 [`src/tool/mod.rs`](./src/tool/mod.rs)：

```rust
#[async_trait]
pub trait Tool {
    async fn invoke(&self, input: &Value) -> Result<String>;
    fn name(&self) -> Cow<'_, str>;
    fn tool_spec(&self) -> ToolSpec;
}
```

一个工具需要提供三件事：

- `invoke()`：收到模型生成的 JSON 参数后，执行本地逻辑。
- `name()`：工具名称。
- `tool_spec()`：返回给模型看的工具 schema。

Rust trait 里不能直接写 `async fn` 并作为 trait object 使用，所以这里使用 `async_trait`。这让我们可以把不同工具装进 `Box<dyn Tool>`。

## 工具注册

```rust
pub fn toolset() -> HashMap<String, Box<dyn Tool>> {
    HashMap::from([
        ("bash".to_string(), bash_tool()),
        ("read_file".to_string(), read_file_tool()),
        ("write_file".to_string(), write_file_tool()),
        ("edit_file".to_string(), edit_file_tool()),
    ])
}
```

主循环不再关心具体工具类型，只保存：

```rust
tools: HashMap<String, Box<dyn Tool>>
```

当模型返回 `tool_use` 时，根据 `name` 找到对应工具并调用：

```rust
let Some(tool) = self.tools.get_mut(name) else {
    return format!("Unknown tool: {name}");
};

match tool.invoke(input).await {
    Ok(output) => output,
    Err(e) => format!("Error invoking tool {}: {}", name, e),
}
```

这个设计的价值是：新增工具时，只需要实现 `Tool`，再注册到 `toolset()`，不用改 agent loop 的核心流程。

## 文件工具

本章新增三个文件工具。

### read_file

读取 UTF-8 文本文件，并支持可选 `limit` 限制返回行数：

```json
{
  "path": "src/main.rs",
  "limit": 120
}
```

工具返回内容会被截断到 50000 个字符，避免过长文件污染上下文。

### write_file

写入文件内容：

```json
{
  "path": "notes/demo.md",
  "content": "# Demo\n"
}
```

注意：本章的 `safe_path()` 依赖 `canonicalize()`，对不存在的文件路径不够友好。这个问题会在后续章节通过 `safe_path_allow_missing()` 修正。

### edit_file

对文件做精确替换：

```json
{
  "path": "src/main.rs",
  "old_text": "old",
  "new_text": "new"
}
```

它只替换第一次匹配的文本。这样比直接让模型重写整个文件更可控，也更适合小范围修改。

## 路径边界

`safe_path()` 的作用是限制文件工具只能访问当前 workspace 内部：

```rust
let cwd = std::env::current_dir()?;
let full = cwd.join(path).canonicalize()?;

if !full.starts_with(&cwd) {
    return Err(anyhow::anyhow!("Path escapes workspace"));
}
```

这是 agent 工程里非常重要的边界：模型可以建议访问路径，但 harness 必须决定是否允许访问。

## 消息规范化

模型 API 对 tool use 的消息顺序有要求：`tool_use` 需要对应的 `tool_result`。如果中间异常退出、用户打断或上下文被拼接，就可能出现孤立的 `tool_use`。

`normalize_messages()` 做两件事：

- 给没有结果的 `tool_use` 补一个 `(cancelled)` 的 `tool_result`。
- 合并连续相同角色的消息，减少不必要的消息边界。

这类逻辑看起来不显眼，但对长时间运行的 agent 很关键。真实系统里，很多稳定性问题都来自消息历史不满足协议约束。

## Rust 实现要点

- 用 trait 表达工具接口。
- 用 trait object 管理异构工具集合。
- 用 `HashMap` 做工具名到工具实现的派发。
- 用 `serde_json::Value` 接收模型生成的动态参数。
- 用 `Cow<'_, str>` 避免不必要的字符串分配。

## 本章的局限

- 工具没有权限控制，模型请求工具就会执行。
- 工具之间没有生命周期 hook。
- `Tool::invoke(&self, ...)` 不适合保存工具内部状态。
- 文件写入对新文件路径支持不完善。
- agent 还没有任务规划能力，复杂任务容易丢失进度。
