# s03: Todo Planning

前两章里的 agent 已经可以调用工具，也能读写文件。但它还缺少一个关键能力：在复杂任务中维护自己的进度。

当任务只有一步时，模型可以直接完成；当任务包含多个文件、多轮探索、多次修改和验证时，仅靠上下文里的自然语言很容易丢失当前状态。本章加入 `todo` 工具，让 agent 把任务拆成明确的计划，并在推进过程中持续更新。

## 运行方式

```bash
cargo run -p s03_todo_write
```

## 本章新增能力

- 新增 `todo` 工具，用结构化列表记录任务计划。
- 每个计划项支持三种状态：
  - `pending`
  - `in_progress`
  - `completed`
- 限制同一时间最多只能有一个 `in_progress`。
- 为长期未更新计划的情况加入 reminder。
- 将 `Tool::invoke` 从 `&self` 改成 `&mut self`，支持工具内部状态。

## 为什么 agent 需要 todo

Agent harness 的核心循环本身不理解“任务”。它只知道：

```text
模型请求工具 -> 执行工具 -> 回填结果 -> 继续推理
```

但复杂任务需要额外的过程状态，例如：

- 当前正在做哪一步。
- 哪些步骤已经完成。
- 下一步应该做什么。
- 是否长时间没有刷新计划。

`todo` 工具就是把这些状态显式化。它不是为了展示给用户看的装饰，而是给模型一个可以反复读取和更新的外部工作记忆。

## Todo 数据模型

```rust
#[derive(EnumProperty, PartialEq, Eq, Clone, Debug, Hash, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum PlanItemStatus {
    Pending,
    InProgress,
    Completed,
}
```

每个计划项由 `PlanItem` 表示：

```rust
pub struct PlanItem {
    content: String,
    status: PlanItemStatus,
    #[serde(rename = "activeForm")]
    active_form: Option<String>,
}
```

`content` 是任务描述，`status` 是当前状态，`activeForm` 用来给正在执行的任务提供一个更自然的进行时标签。

示例输入：

```json
{
  "items": [
    {
      "content": "Inspect the project structure",
      "status": "completed"
    },
    {
      "content": "Implement the parser",
      "status": "in_progress",
      "activeForm": "Implementing the parser"
    },
    {
      "content": "Run tests",
      "status": "pending"
    }
  ]
}
```

## TodoManager

`TodoManager` 是一个有状态工具：

```rust
#[derive(Default)]
pub struct TodoManager {
    items: Vec<PlanItem>,
}
```

它的 `update()` 会做两个约束检查：

- 计划最多 12 项，避免模型生成过长 checklist。
- 同一时间最多一个 `in_progress`，避免并行状态混乱。

```rust
if in_progress_count > 1 {
    return Err(anyhow::anyhow!("Only one plan item can be in_progress"));
}
```

这个约束很重要。好的 agent harness 不应该替模型写死工作流，但应该用少量规则帮助模型保持清晰。

## Tool trait 的变化

s02 中的工具接口是：

```rust
async fn invoke(&self, input: &Value) -> Result<String>;
```

s03 改成：

```rust
async fn invoke(&mut self, input: &Value) -> Result<String>;
```

原因是 `todo` 需要修改自身状态。Rust 在这里迫使我们把状态变化表达清楚：如果工具会改变内部数据，就需要可变引用。

这也是 Rust 写 agent harness 的一个优点。很多架构边界会通过类型系统提前暴露出来。

## Reminder 机制

`LoopState` 新增：

```rust
todo_rounds_since_update: usize
```

每轮工具执行后，如果本轮没有调用 `todo`，计数增加；如果调用了 `todo`，计数清零。

当计数超过阈值时，harness 会插入一条 reminder：

```text
<reminder>Refresh your current plan before continuing.</reminder>
```

这条 reminder 会作为工具结果的一部分回到上下文里，提醒模型更新计划。

## 这里为什么还没有抽象 Hook

从架构上看，“每轮结束后检查 todo 是否更新”已经有点像生命周期 hook。它不是一个普通工具调用，而是 agent loop 在特定阶段触发的系统行为。

本章为了保持代码简单，直接把计数器放在 `LoopState` 里。这样读者可以先看到问题本身：

- 有些逻辑不是模型主动调用的工具。
- 它们需要在 agent 生命周期中自动运行。
- 当这类逻辑变多时，就需要 hook 系统。

后面的章节会继续把这种模式抽象出来。

## Rust 实现要点

- 用 `serde` 反序列化模型传入的 JSON。
- 用 enum 表达有限状态集合。
- 用 `strum` 给状态枚举绑定展示 marker。
- 用 `&mut self` 明确工具状态变更。
- 用 `LoopState` 保存跨轮次过程状态。

## 本章的局限

- reminder 逻辑还写在 `LoopState` 中，没有通用 hook 抽象。
- todo 状态只存在内存中，程序退出后丢失。
- 模型仍然可能不按预期更新 todo，只能通过 prompt 和 reminder 引导。
- 计划是单会话级别，还不是长期任务系统。
