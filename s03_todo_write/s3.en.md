# s03: Todo Planning

s02 gives the agent structured tools, but complex work still has no durable plan. If a task contains many steps, the model can forget what has been done or jump between subtasks.

This chapter adds a `todo` tool. The agent can create, update, and inspect a task list while it works.

## Run

```bash
cargo run -p s03_todo_write
```

## New Capabilities

- Add a stateful `todo` tool.
- Store todo items in the tool itself.
- Support `add`, `update`, `list`, and `clear` actions.
- Change `Tool::invoke` from `&self` to `&mut self`.
- Let the agent maintain visible progress during multi-step tasks.

## Why Todo Matters

An agent loop gives the model a way to act, but not a stable plan.

A todo list gives the agent:

- A place to record intended steps.
- A way to mark progress.
- A compact summary of the current task state.
- A tool result that can be returned to context and reused in later turns.

This is not a full project management system. It is a minimal planning surface inside the harness.

## Stateful Tool

In s02, tools used:

```rust
async fn invoke(&self, input: &Value) -> Result<String>;
```

s03 changes it to:

```rust
async fn invoke(&mut self, input: &Value) -> Result<String>;
```

That lets a tool mutate internal state.

The todo tool keeps a vector of todo items:

```rust
pub struct TodoTool {
    todos: Vec<TodoItem>,
}
```

This state lives for the current process. It is not yet persisted to disk.

## TodoItem

A todo item contains:

- `id`
- `content`
- `status`

Typical statuses are:

- `pending`
- `in_progress`
- `completed`

The model updates these through tool calls, and the result is returned as text.

## Tool Input

The todo tool uses an `action` field:

```json
{
  "action": "add",
  "content": "Inspect the current tool system"
}
```

Update an item:

```json
{
  "action": "update",
  "id": 1,
  "status": "completed"
}
```

List items:

```json
{
  "action": "list"
}
```

This keeps the teaching version compact. Later chapters split larger domains into separate typed tools.

## Agent Behavior

The system prompt tells the agent to use todo planning for non-trivial work. A typical flow becomes:

```text
user asks for a task
  -> agent creates todos
  -> agent performs one step with tools
  -> agent updates todo status
  -> agent continues
  -> final response summarizes completed work
```

The harness does not force the model to use the todo tool. It exposes the capability and describes when to use it.

## Rust Notes

- `&mut self` enables tool-local state.
- `HashMap<String, Box<dyn Tool>>` still works with mutable dispatch.
- `serde_json::Value` keeps the tool input flexible.
- Tool output remains plain text returned as `tool_result`.

## Limits

- Todo state is in memory only.
- The schema uses an action-based API instead of separate typed tools.
- No dependency graph between todo items.
- No persistence across sessions.
- The harness does not enforce todo usage.
