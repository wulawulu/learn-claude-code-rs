# s07: Permission System

In earlier chapters, every model-requested tool call is executed. That is convenient for demos, but unsafe for a real coding agent. The model may ask to modify files, run shell commands, or perform dangerous operations.

This chapter adds a permission system. Tool calls are checked against policy before the harness allows, denies, or asks the user.

## Run

```bash
cargo run -p s07_permission_system
```

Startup asks for a permission mode:

```text
default - ask by default
plan - read only
auto - allow reads, ask for writes
```

Interactive commands:

```text
/rules
/mode default
/mode plan
/mode auto
```

## New Capabilities

- Add `PermissionManager`.
- Support three modes: `Default`, `Plan`, and `Auto`.
- Support `allow`, `deny`, and `ask`.
- Match rules by tool name, path, and command content.
- Add regex-based bash safety checks.
- Let the user allow once, deny, or always allow a tool.
- Suggest plan mode after repeated denials.

## What Permissions Solve

Tool use lets the model affect the outside world. If an agent can write files, run commands, or access external systems, it needs boundaries.

The permission system answers:

```text
Should this tool call execute under the current mode and rules?
```

Possible decisions:

- `Allow`: execute directly.
- `Deny`: reject and return the reason as a tool result.
- `Ask`: pause and ask the user.

## PermissionMode

```rust
pub enum PermissionMode {
    Default,
    Plan,
    Auto,
}
```

Modes:

- `Default`: ask when no rule matches.
- `Plan`: allow read-only operations and deny writes. Useful for analysis and planning.
- `Auto`: allow reads automatically; writes still need a rule or user confirmation.

These modes cover common trust levels for a coding agent.

## PermissionDecision

```rust
pub struct PermissionDecision {
    pub behavior: PermissionBehavior,
    pub reason: String,
}
```

The permission layer returns both the decision and the reason. The reason is shown to the user and returned to the model if denied.

## Rule Matching

`PermissionRule` can match:

- `tool`: tool name.
- `path`: path glob.
- `content`: bash command glob.

Default rules include:

```rust
vec![
    PermissionRule::deny_tool_content("bash", "rm -rf /"),
    PermissionRule::deny_tool_content("bash", "sudo *"),
    PermissionRule::allow_tool("read_file").with_path("*"),
]
```

The chapter uses `wildmatch` for simple glob matching.

## Bash Safety Check

`BashSecurityValidator` checks risky command patterns:

- Shell metacharacters such as `;`, `&`, `|`, backticks, `$`.
- `sudo`.
- `rm -r` / `rm -rf`.
- Command substitution `$(...)`.
- `IFS=`.

Severe matches are denied. Lower-risk matches ask the user.

This is not a shell sandbox. It demonstrates that bash needs a separate permission layer before execution.

## Check Order

`PermissionManager::check()` follows this order:

```text
bash safety check
  -> deny rules
  -> mode logic
  -> allow rules
  -> ask
```

Dangerous actions are blocked early; modes provide global behavior; rules provide local exceptions.

## Agent Loop Integration

Before tool execution, [`src/lib.rs`](./src/lib.rs) checks:

```rust
let decision = self.permission_manager.check(name, input);
```

If denied, the model receives:

```text
Permission denied: <reason>
```

The model can then adapt, for example by doing read-only analysis or asking the user.

## User Confirmation

When the decision is `Ask`, the CLI shows a preview:

```text
[Permission] edit_file: {"path":"src/lib.rs",...}
```

The user can choose:

- `allow once`
- `deny`
- `always allow this tool`

`always allow` adds an in-memory allow rule for the current process.

## Rust Notes

- Use enums for modes and decisions.
- Use `strum` for parsing commands like `/mode plan`.
- Use `regex` for bash risk detection.
- Use `wildmatch` for simple glob matching.
- Use `inquire::Select` for interactive prompts.

## Limits

- Rules are not persisted.
- The bash validator is not a sandbox.
- Read/write classification is still coarse.
- Sensitive workspace areas are not separated.
- The permission system is not yet integrated with hooks.

## Next
