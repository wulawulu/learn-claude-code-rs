# s07: Permission System

前几章里，只要模型请求工具，harness 就会执行。对于教学 demo 这很方便，但对真实 coding agent 来说不够安全。模型可能请求修改文件、运行 shell 命令，甚至执行危险操作。

本章加入权限系统，让工具调用先经过策略判断，再决定允许、拒绝或询问用户。

## 运行方式

```bash
cargo run -p s07_permission_system
```

启动时会选择权限模式：

```text
default - ask by default
plan - read only
auto - allow reads, ask for writes
```

交互中还支持：

```text
/rules
/mode default
/mode plan
/mode auto
```

## 本章新增能力

- 引入 `PermissionManager`。
- 支持三种权限模式：`Default`、`Plan`、`Auto`。
- 支持 allow / deny / ask 三种决策。
- 支持基于工具名、路径、命令内容的规则匹配。
- 对 bash 命令做正则安全检查。
- 用户可以选择 allow once、deny、always allow this tool。
- 连续拒绝后提示切换到 plan mode。

## 权限系统解决什么问题

Tool use 的本质是让模型影响外部世界。只要 agent 能写文件、跑命令、访问网络，就必须有边界。

权限系统的职责不是替模型完成任务，而是回答：

```text
这个 tool call 在当前模式和规则下是否应该执行？
```

它把工具调用分成三类：

- `Allow`：直接执行。
- `Deny`：拒绝执行，并把拒绝原因作为 tool result 返回模型。
- `Ask`：暂停，询问用户。

## PermissionMode

```rust
pub enum PermissionMode {
    Default,
    Plan,
    Auto,
}
```

三种模式的含义：

- `Default`：没有命中规则时询问用户。
- `Plan`：只允许读操作，写操作直接拒绝。适合让 agent 先分析和制定方案。
- `Auto`：读操作自动允许，写操作仍需规则或用户确认。

这三个模式覆盖了 coding agent 常见的信任层级。

## PermissionDecision

```rust
pub struct PermissionDecision {
    pub behavior: PermissionBehavior,
    pub reason: String,
}
```

权限检查不仅返回结果，也返回原因。原因会用于日志和 tool result，让模型知道为什么工具调用被拒绝。

## 规则匹配

`PermissionRule` 支持按三种维度匹配：

- `tool`：工具名。
- `path`：文件路径 glob。
- `content`：bash command glob。

默认规则：

```rust
vec![
    PermissionRule::deny_tool_content("bash", "rm -rf /"),
    PermissionRule::deny_tool_content("bash", "sudo *"),
    PermissionRule::allow_tool("read_file").with_path("*"),
]
```

规则使用 `wildmatch` 做简单 glob 匹配。它不追求完整策略语言，但足够表达教学版本中的常见边界。

## bash 安全检查

`BashSecurityValidator` 使用正则检查危险命令模式：

- shell metacharacters：`;`、`&`、`|`、反引号、`$`
- `sudo`
- `rm -r` / `rm -rf`
- command substitution：`$(...)`
- `IFS=`

严重命中如 `sudo`、`rm_rf` 会直接拒绝；较轻命中如管道符会进入询问。

这不是完整 shell sandbox。它的目标是展示：bash 工具不能只靠简单字符串黑名单，权限层应该独立存在，并且在执行前统一处理。

## 权限检查顺序

`PermissionManager::check()` 的顺序是：

```text
bash 安全检查
  -> deny rule
  -> mode logic
  -> allow rule
  -> ask
```

这个顺序有两个意义：

- 明确危险操作优先拦截。
- 模式可以提供全局策略，规则提供局部例外。

## 接入 agent loop

在 [`src/lib.rs`](./src/lib.rs) 中，工具执行前先检查权限：

```rust
let decision = self.permission_manager.check(name, input);
```

如果拒绝：

```text
Permission denied: <reason>
```

会作为 `tool_result` 返回给模型。这样模型可以基于拒绝原因调整行为，比如换成只读分析或请求用户授权。

## 用户确认

当决策是 `Ask` 时，CLI 会展示工具名和参数预览：

```text
[Permission] edit_file: {"path":"src/lib.rs",...}
```

用户可以选择：

- `allow once`
- `deny`
- `always allow this tool`

`always allow` 会在当前进程内追加一条 allow rule。这个规则不会持久化，重启后消失。

## Rust 实现要点

- 用 enum 表达模式和决策，避免字符串状态散落。
- 用 `strum` 支持 `/mode plan` 这类字符串解析。
- 用 `regex` 做 bash 风险识别。
- 用 `wildmatch` 做简单 glob 匹配。
- 用 `inquire::Select` 实现交互式确认。

## 本章的局限

- 规则不持久化。
- bash validator 不是沙箱，无法覆盖所有 shell 风险。
- read/write 工具分类仍然比较粗。
- 没有区分工作区内不同敏感目录。
- 权限系统还没有接入 hook 机制。
