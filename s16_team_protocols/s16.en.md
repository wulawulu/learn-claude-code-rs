# s16: Team Protocols

s15 lets agents communicate through inbox messages. Free-form messages are useful, but some workflows need explicit confirmation, queryable state, and recovery.

This chapter adds structured protocols on top of team mailboxes. Some actions become requests: create a record, send a request message, wait for a response, and persist the outcome.

## Run

```bash
cargo run -p s16_team_protocols
```

Interactive commands:

```text
/team
/inbox
exit()
```

Runtime state:

```text
.team/
.team/inbox/
.team/requests/
```

## New Capabilities

- Add a durable request store.
- Correlate request and response with `request_id`.
- Support shutdown request / response.
- Support plan approval submit / review.
- Keep protocol messages on the same inbox surface.
- Store request state outside model context.
- Centralize mailbox, request store, and teammate lifecycle in `SharedTeammateManager`.

## Why Protocols

Ordinary message:

```text
send text -> receiver interprets it
```

Protocol request:

```text
create request -> send message with request_id -> receiver responds -> update durable state
```

The difference is state. If a workflow needs to be checked later, it should have a durable request record.

## InboxMessage

Protocol messages still use:

```text
.team/inbox/<member>.jsonl
```

Message types include:

```rust
Message,
Broadcast,
ShutdownRequest,
ShutdownResponse,
PlanApproval,
PlanApprovalResponse,
```

Protocol metadata such as `request_id`, `approve`, `plan`, and `feedback` is carried in `extra`.

## RequestRecord

Core models:

```rust
pub enum RequestKind {
    Shutdown,
    PlanApproval,
}

pub enum RequestStatus {
    Pending,
    Approved,
    Rejected,
}
```

Each request is written to:

```text
.team/requests/<request_id>.json
```

This makes protocol status durable even if context is compacted.

## SharedRequestStore

The request store:

- Generates request ids.
- Creates pending records.
- Reads request records.
- Updates approval results.
- Renders status.

The message bus transports events. The request store keeps workflow state.

## SharedTeammateManager

By s16, the teammate manager becomes a team protocol facade. It provides:

- spawn teammate
- list teammates
- send message
- broadcast
- read inbox
- create shutdown request
- respond shutdown
- submit plan
- review plan
- query request status

High-level code talks to the manager instead of directly coordinating bus and store.

## Agent Loop Integration

Protocols do not change the main loop:

```text
inbox event -> context -> model -> tool_use -> manager -> inbox/request update
```

The model still sees protocol events through inbox injection, then chooses the right tool response.

## Shutdown Protocol

```text
lead creates shutdown request
  -> request = pending
  -> lead sends shutdown_request to teammate inbox
  -> teammate calls shutdown_response
  -> request becomes approved / rejected
  -> if approved, manager sets stop signal
```

Protocol response and actual loop exit are separate concepts.

## Plan Approval Protocol

```text
teammate submits plan
  -> pending request is stored
  -> lead receives plan_approval
  -> lead approves or rejects
  -> request record is updated
  -> lead sends plan_approval_response
```

This lets the team enforce explicit approval before important work.

## Role-Specific Tools

Lead side:

- `shutdown_request`
- `shutdown_response` for querying status
- `plan_approval` for reviewing plans
- team organization tools

Teammate side:

- `shutdown_response` for responding
- `plan_approval` for submitting plans
- messaging tools

The same tool name may represent different sides of a protocol depending on role.

## Limits

- No cross-process request locking.
- No protocol timeout or retry.
- No generic protocol engine trait.
- Plan approval is a collaboration protocol, not a hard scheduler policy.
- Shutdown is cooperative, not forceful.
