# s16: Team Protocols

s15 已经让多个 agent 可以通过 inbox 收发消息。但自由文本消息只适合一般协作，不适合表达必须被确认、可查询、可恢复的工作流。

本章在团队 mailbox 之上加入结构化协议：某些动作不再只是“发一条消息”，而是创建 request、发送 request message、等待 response，并把状态落盘。

## 运行方式

```bash
cargo run -p s16_team_protocols
```

交互命令：

```text
/team
/inbox
exit()
```

运行后会使用：

```text
.team/
.team/inbox/
.team/requests/
```

保存团队成员、inbox 消息和协议请求状态。

## 本章新增能力

- 增加 durable request store。
- 用 `request_id` 关联协议请求和响应。
- 支持 shutdown request / response。
- 支持 plan approval submit / review。
- 协议消息继续通过 inbox 进入 agent loop。
- request 状态独立落盘，不依赖模型上下文记忆。
- `SharedTeammateManager` 统一编排 mailbox、request store 和 teammate 生命周期。

## 为什么需要 Protocol

普通 message 的语义是：

```text
发送一段内容 -> 对方看到后自行理解
```

协议 request 的语义是：

```text
创建请求 -> 发送带 request_id 的消息 -> 对方响应 -> 更新 durable 状态
```

两者的差异在于状态边界。

普通消息适合：

- 分派说明。
- 状态同步。
- 临时沟通。

协议请求适合：

- shutdown 这类需要明确确认的控制流。
- plan approval 这类需要审批结果的协作步骤。
- 后续可查询、可恢复、可审计的团队工作流。

在 agent harness 中，模型上下文不是可靠数据库。只要一个协作动作需要“之后还能查到结果”，它就应该有独立的 durable record。

## InboxMessage

协议消息仍然使用和普通团队消息相同的 inbox 文件：

```text
.team/inbox/lead.jsonl
.team/inbox/reviewer.jsonl
```

消息类型包括：

```rust
Message,
Broadcast,
ShutdownRequest,
ShutdownResponse,
PlanApproval,
PlanApprovalResponse,
```

协议字段通过 `extra` 携带，例如：

- `request_id`
- `approve`
- `plan`
- `feedback`

这样可以保持两层边界：

- `InboxMessage` 是传输 envelope。
- `RequestRecord` 是协议状态。

## RequestRecord

本章新增的核心模型是 request：

```rust
pub enum RequestKind {
    Shutdown,
    PlanApproval,
}

pub enum RequestStatus {
    Pending,
    Approved,
    Rejected,
}
```

每个 request 都会写入：

```text
.team/requests/<request_id>.json
```

request record 解决的问题是：协议状态不再藏在聊天文本里。

例如 shutdown request 创建后，即使模型上下文被压缩，系统仍然可以通过 request id 查询：

- 谁发起了请求。
- 请求给了谁。
- 当前是 pending、approved 还是 rejected。
- 响应内容是什么。

## SharedRequestStore

`SharedRequestStore` 负责 durable request 的文件读写：

- 创建 request id。
- 写入 pending request。
- 读取 request record。
- 更新审批结果。
- 渲染 request 状态。

它和 message bus 的职责不同：

- message bus 负责把事件送进对方 inbox。
- request store 负责保存协议状态。

这也是本章最重要的设计边界。

## SharedTeammateManager

到 s16 为止，`SharedTeammateManager` 不再只是 teammate registry。它变成团队协议的统一门面：

- spawn teammate
- list teammates
- send message
- broadcast
- read inbox
- create shutdown request
- respond shutdown
- submit plan
- review plan
- query request status

高层代码只和 manager 交互，不直接同时操作 message bus 和 request store。这样 request 创建、inbox 发送、状态更新这些步骤不会散落在多个工具里。

## 接入 Agent Loop

协议系统没有改变 agent loop 的基本形态。

每轮请求模型前，`LoopState` 仍然读取当前 agent 的 inbox，并注入上下文：

```xml
<inbox>
[
  ...
]
</inbox>
```

也就是说，request store 不会直接驱动模型。模型仍然通过 inbox 看到协议事件，再选择调用对应工具响应。

这保持了一个统一循环：

```text
inbox event -> context -> model -> tool_use -> manager -> inbox/request update
```

## Shutdown Protocol

shutdown 不再是一条普通消息，而是一个 handshake：

```text
lead 创建 shutdown request
  -> request record = pending
  -> lead 发送 shutdown_request 到 teammate inbox
  -> teammate 调用 shutdown_response
  -> request record 更新为 approved / rejected
  -> 如果 approved，manager 设置 stop signal
```

这里要区分两件事：

- 协议响应：teammate 是否同意 shutdown。
- loop 退出：teammate 是否已经真的停止运行。

本章把 stop signal 收到 manager 里，工具层只负责提交响应。

## Plan Approval Protocol

plan approval 使用同样的 request-response 模式，但语义是审批计划：

```text
teammate 提交计划
  -> 创建 pending plan request
  -> lead inbox 收到 plan_approval
  -> lead 审批 approved / rejected
  -> request record 更新
  -> lead 发送 plan_approval_response
```

这让团队协作多了一层约束：teammate 可以工作，但某些大步骤需要先提交计划并等待明确结果。

## 角色工具差异

lead 和 teammate 看到的工具集合不同。

lead 侧：

- `shutdown_request`：发起 shutdown。
- `shutdown_response`：查询 shutdown request 状态。
- `plan_approval`：审批 teammate 的计划。
- `spawn_teammate` / `list_teammates` / `broadcast` 等组织工具。

teammate 侧：

- `shutdown_response`：响应 shutdown request。
- `plan_approval`：提交计划等待审批。
- `send_message` / `read_inbox` 等沟通工具。

同名协议工具在不同角色下可以代表协议的不同一侧。关键不是名字是否完全唯一，而是 request / response 能否通过 request id 闭合。

## Rust 实现要点

- 用 enum 表达 request kind 和 request status。
- 用 JSON 文件保存 durable request state。
- 用 `request_id` 做 correlation id。
- 用 manager facade 集中协议编排。
- 用 `Arc` 和内部锁共享 message bus、request store 和 teammate runtime。
- 让 protocol message 和普通 message 共用 inbox surface。

## 本章的局限

- request file 没有跨进程锁。
- 没有协议超时和重试。
- 没有统一 protocol engine trait。
- plan approval 只是协作协议，还不是强制执行策略。
- shutdown 是协作式停止，不是强制终止。
