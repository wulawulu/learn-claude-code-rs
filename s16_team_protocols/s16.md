# s16

## 目标

`s16` 的核心目标，是把 `s15` 的“自由收发 mailbox 消息”推进到“某些团队协作必须走结构化 request-response protocol”。

这一章对应 Python 原版里的核心思想是：

> protocol workflows are durable handshakes layered on top of the same inbox surface

这里最重要的区分要先讲清楚：

- `s15` 的 message 主要是自由文本协作
- `s16` 的 protocol request 是带 request_id 的结构化握手

两者都还是通过 inbox 进入 agent loop，但语义不同。

- 普通消息：发出去就结束，是否被理解主要靠上下文
- 协议消息：有请求、有响应、有 request_id、有 durable 状态

所以这一章的重点不是“再做一套新通信系统”，而是：

- 哪些工作流应该从 free-form chat 升级成 protocol
- request / response 如何通过同一个 inbox surface 传递
- 请求状态如何脱离上下文，单独 durable 落盘
- lead 和 teammate 如何通过 correlation id 对齐同一件事

---

## 当前 Rust 版做了什么

当前 `s16` 可以概括成一句话：

**在 `s15` 的 mailbox 团队协作之上，加了一层 durable request-response protocol。**

Rust 版最终收敛成了三层：

1. `InboxMessage` 仍然负责 mailbox envelope
2. `RequestRecord` + `SharedRequestStore` 负责 durable 协议状态
3. `SharedTeammateManager` 负责把 message bus、request store 和 teammate 生命周期编排成完整握手

这样 shutdown 和 plan approval 两条协议线都变成了同一种模式：

1. 先创建 request record
2. 再往 inbox 发送带 `request_id` 的协议消息
3. 对方处理后发回 response message
4. request record 更新为 `approved` 或 `rejected`

这一章真正想教的就是：

- protocol 不是另一套 agent
- request record 也不是 task record
- 它只是把“有状态的团队握手”从纯聊天里显式抽出来

## 当前 Rust 版的结构

### 1. `InboxMessage`：协议仍然走同一个 inbox surface

Rust 版没有把 protocol 消息拆到另一条传输通道。

当前所有团队消息仍然都通过 `.team/inbox/*.jsonl`：

- 普通 `message`
- `broadcast`
- `shutdown_request`
- `shutdown_response`
- `plan_approval`
- `plan_approval_response`

也就是说，协议没有绕开 mailbox。

这点和 Python 原版的教学主线一致：

- inbox surface 还是唯一入口
- 不同之处只在于某些消息带了更强的结构和更强的状态约束

当前 `InboxMessage` 的关键字段是：

- `type`
- `from`
- `content`
- `timestamp`
- `extra`

这里的 `extra` 是显式 `Option<HashMap<...>>`，只在协议消息里出现。这样 envelope 和附加协议字段的边界会更清楚：

- 普通消息不带 `extra`
- 协议消息可以通过 `extra` 带 `request_id / approve / plan / feedback`

### 2. `RequestKind / RequestStatus / RequestRecord`

如果说 `s15` 教的是“named workers + mailboxes”，那 `s16` 真正新增的就是 durable request state。

Rust 版没有让请求状态继续藏在消息文本里，而是明确建模成：

- `RequestKind`
- `RequestStatus`
- `RequestRecord`

其中：

- `RequestKind` 区分 shutdown 和 plan approval
- `RequestStatus` 区分 `pending / approved / rejected`
- `RequestRecord` 负责真正落盘的协议状态

这一层的重点不是“多定义几个 struct”，而是把 protocol 的状态边界从上下文里拿出来。

这样：

- request_id 不只是消息中的一个字符串
- 它还能在 `.team/requests/<request_id>.json` 里找到完整状态

### 3. `SharedRequestStore`：request 是 durable workflow object

Python 原版把 request file 定义成：

- `.team/requests/{request_id}.json`

Rust 版保持了同样的思路，并收成了 `SharedRequestStore`。

它负责：

- 生成 request id
- 创建 request record
- 读取 request record
- 更新 request record
- 输出 request status 的 JSON 展示

这让 request 和普通 inbox message 的角色明确分开：

- inbox message：传输协议事件
- request record：保存协议状态

所以这一章最重要的边界就是：

- message 是瞬时事件
- request record 是 durable workflow state

### 4. `RequestRecord` 辅助方法：协议状态更新被收成显式语义

当前 Rust 版里，`RequestRecord` 还带了一些很小但很有用的辅助方法：

- `pending_shutdown(...)`
- `pending_plan_approval(...)`
- `resolve_shutdown(...)`
- `review_plan(...)`

这样 manager 里就不需要每次都手写一整串字段拼装。

这也是一个典型的 Rust 化收敛：

- Python 原版更偏直接改字典
- Rust 版更倾向把状态变迁抽成显式小方法

它带来的好处不是“抽象更高级”，而是：

- pending request 长什么样
- approve / reject 会改哪些字段

这些规则都更集中，也更不容易写漏。

### 5. `SharedTeammateManager`：现在更像 team protocol facade

到 `s16` 为止，`SharedTeammateManager` 已经不只是一个 teammate registry 了。

它现在对外统一提供：

- spawn teammate
- list teammates
- send / broadcast
- read inbox
- create shutdown request
- respond shutdown
- submit plan
- review plan
- query shutdown status

而 `SharedMessageBus` 和 `SharedRequestStore` 则都退回成 manager 的内部依赖。

这层设计在当前 Rust 版里非常重要，因为它把调用层级收清楚了：

- `main` 只和 manager 打交道
- `LoopState` 只和 manager 打交道
- team/protocol tools 也只和 manager 打交道

这样 mailbox 和 request store 仍然存在，但已经不再直接泄漏到高层调用面。

### 6. `LoopState`：协议消息仍然通过 inbox 注回上下文

这一章虽然加入了 request store，但并没有改变主 loop 的基本模式。

`LoopState::agent_loop()` 在每轮真正请求模型前仍然会：

- 通过 manager 读当前 owner 的 inbox
- 把 inbox 内容包装成 `<inbox>...</inbox>`
- 注入成新的 `user` 消息

这说明一件很关键的事：

- request store 不是直接驱动模型
- 真正进入上下文的仍然是 inbox message

所以协议系统仍然服从同一个 loop：

- mailbox 提供事件
- loop 吸收事件
- 模型基于上下文决定下一步

### 7. shutdown protocol：请求和退出是两回事

shutdown 在这一章里不是一句普通“请你停下”的聊天消息，而是一个明确的 handshake。

Rust 版当前流程是：

1. lead 创建 `shutdown` 类型 request
2. request record 初始状态是 `pending`
3. lead 往 teammate inbox 发 `shutdown_request`
4. teammate 用 `shutdown_response` 工具响应
5. request record 更新为 `approved` 或 `rejected`
6. 如果批准，manager 再触发该 teammate 的 stop signal
7. teammate loop 退出后，成员状态落成 `shutdown`

这里很关键的一点是：

- 协议响应
- loop 停止

在 Rust 版里已经都统一收口到了 manager。

所以 `ShutdownResponseTool` 不再自己操心：

- request record 怎么更新
- 该不该停 loop

它只负责把输入交给 manager。

### 8. plan approval protocol：先提交计划，再继续大工作

plan approval 的主线和 shutdown 类似，但语义不同：

1. teammate 先提交计划
2. request record 以 `pending` 状态落盘
3. lead 在 inbox 里看到这条 protocol message
4. lead 审批这份计划
5. request record 更新为 `approved` 或 `rejected`
6. lead 再发 `plan_approval_response`

这条协议没有直接控制 loop 退出，但它表达了一种更强的团队约束：

- 不是所有重要工作都可以直接开做
- 某些步骤必须先经过明确审批

从教学角度说，这就是这一章比 `s15` 多出来的东西：

- 协作从“能发消息”推进到“某些事情必须完成握手”

### 9. 工具集合：同名协议工具在 lead / teammate 两边语义不同

这一章的 Rust 版继续沿用 `tool/mod.rs` 的统一装配方式。

lead 工具集合里：

- `shutdown_request` 表示“发起 shutdown”
- `shutdown_response` 表示“查询 shutdown request 状态”
- `plan_approval` 表示“审批 teammate 的计划”

teammate 工具集合里：

- `shutdown_response` 表示“真正响应 shutdown request”
- `plan_approval` 表示“提交计划等待 lead 审批”

也就是说，同名工具在不同角色下代表的是 protocol 的不同一侧。

这和 Python 原版的教学表达是一致的：

- 重要的不是工具名完全唯一
- 而是 request / response 两侧的工作流要成对闭合

## 和 Python 原版的对应关系

如果按教学主线对照，当前 Rust 版和 Python 原版是一致的：

1. protocol 消息仍然走 mailbox
2. request 有单独的 durable request file
3. shutdown 和 plan approval 都通过 request_id 关联 request / response
4. teammate loop 仍然基于 inbox re-entry，而不是另一套执行系统

所以当前 Rust 版保住了这章最关键的教学意图：

- protocol 是 mailbox 之上的结构化握手
- request file 是 durable 状态，不是聊天记录

---

## 和 Python 原版的相同点

### 1. shutdown 和 plan approval 都走同一种 correlation pattern

两边都采用：

- 创建 request_id
- 发 request message
- 对方发 response message
- 根据 request_id 回写 durable record

所以虽然是两个不同协议，但底层 pattern 是统一的。

### 2. request state 都是 durable 的

两边都把协议状态写到：

- `.team/requests/<request_id>.json`

这样 request 不依赖“模型还记不记得上下文里之前说了什么”。

### 3. 协议消息仍然和普通团队消息共用 inbox

两边都没有额外创建另一条 transport，而是沿用同一个 mailbox surface。

---

## 和 Python 原版的不同点

### 1. Rust 版把 request state 和状态流转显式做成类型

Python 原版主要靠字典字段和字符串约定。

Rust 版这里把核心概念收敛成了：

- `RequestKind`
- `RequestStatus`
- `RequestRecord`
- `ProtocolResponse`

这样协议状态空间会更稳定，也更容易看出边界。

### 2. Rust 版把编排逻辑更集中地收口到 manager

Python 原版更偏教学式直排：

- tool handler 里直接写 request store 更新
- 直接写 mailbox response

Rust 版当前没有继续把这些规则散在 tool 里，而是尽量收口到 `SharedTeammateManager`：

- tool 负责参数解析
- manager 负责真正的 protocol orchestration

这让当前实现更像“team coordination service”，而不是一组零散 handler。

### 3. Rust 版把 stop signal 也并入了 manager 的协议收口

Python 原版里 teammate loop 会在处理 `shutdown_response approve=true` 后自行退出。

Rust 版当前进一步把：

- 更新 shutdown request
- 触发对应 teammate 的 stop signal

都收到了 manager 里。

这样工具层就不会再跨层知道 loop 退出控制细节。

### 4. Rust 版让 `main` 和 `LoopState` 也只依赖 manager facade

当前 Rust 版里：

- `main` 不再直接构造 bus/store
- `LoopState` 也不再直接拿 `message_bus`

它们都只依赖 `SharedTeammateManager`。

这意味着当前架构已经形成比较完整的 facade 形状：

- 高层只知道 manager
- 低层 bus/store 都是内部实现细节

## 这章真正新增的能力

从整套课程脉络看，`s16` 真正新增的不是“更多消息类型”，而是：

- 团队里第一次出现了明确的 handshake workflow

从 `s15` 到 `s16` 的跃迁可以概括成：

- `s15`：teammates can talk
- `s16`：some workflows must be acknowledged, correlated, and durably tracked

这会给后面很多更复杂的团队协作打基础，比如：

- work claiming
- approval gates
- resumable coordination
- autonomous task negotiation

也就是说，`s16` 的真正价值在于：

- 把“自由聊天协作”推进成“带 durable state 的协议协作”

---

## 当前实现的边界

虽然当前 Rust 版已经把 `s16` 的主线落完整了，但它仍然保持了刻意简化。

目前还没有展开的部分包括：

- 更复杂的 protocol schema 校验
- 跨进程级别的 request file 协调
- protocol 超时 / 重试
- 多阶段审批流
- 一个统一的 protocol engine trait

这不是缺失，而是刻意没有把本章做成完整工作流引擎。

当前真正必须保住的，是这四件事：

1. protocol 消息仍走 inbox
2. request state 单独 durable 落盘
3. request / response 通过 request_id 关联
4. handshake 编排被集中表达，而不是散在聊天上下文里

只要这四点成立，`s16` 的教学主线就是完整的。

---

## 小结

`s16` 真正教的不是“给团队加几个审批工具”，而是：

- 把普通团队消息和协议消息区分开
- 把协议状态从聊天上下文里抽出来
- 让团队协作第一次具备 durable handshake 的形状

当前 Rust 版通过：

- `InboxMessage`
- `RequestRecord`
- `SharedRequestStore`
- `SharedTeammateManager`

把这条主线落成了一个比较清楚的实现。

所以这一章最重要的理解可以收成一句话：

**protocol 不是新的 transport，而是 mailbox 之上的 durable handshake。**
