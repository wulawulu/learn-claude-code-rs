# s12: Task System

s03 的 todo 是会话内计划，适合当前任务的短期进度管理。但 todo 仍然活在对话上下文里，一旦上下文压缩或程序退出，任务状态就可能丢失。

本章把任务提升成持久化 work item，保存到 `.tasks/` 目录。Agent 可以通过工具创建、读取、更新和列出任务，并维护任务之间的依赖关系。

## 运行方式

```bash
cargo run -p s12_task_system
```

运行后会在当前工作目录创建或使用：

```text
.tasks/
```

## 本章新增能力

- 新增 `TaskRecord` 持久任务结构。
- 新增 `.tasks/task_<id>.json` 任务文件。
- 支持任务状态：`pending`、`in_progress`、`completed`、`deleted`。
- 支持任务依赖：`blockedBy` 和 `blocks`。
- 支持 owner 字段，为后续 team 协作做准备。
- 新增四个 task 工具：
  - `task_create`
  - `task_get`
  - `task_list`
  - `task_update`

## Task 和 Todo 的区别

Todo 是当前会话的工作清单：

```text
短期、轻量、主要帮助模型保持当前步骤清晰
```

Task 是磁盘上的持久工作项：

```text
长期、可恢复、可以跨压缩和跨进程继续存在
```

二者都能帮助 agent 管理工作，但层级不同。

## TaskRecord

```rust
pub struct TaskRecord {
    pub id: u64,
    pub subject: String,
    pub description: Option<String>,
    pub status: TaskStatus,
    pub blocked_by: Vec<u64>,
    pub blocks: Vec<u64>,
    pub owner: String,
}
```

字段含义：

- `id`：任务编号。
- `subject`：一句话标题。
- `description`：可选详细说明。
- `status`：当前状态。
- `blocked_by`：当前任务被哪些任务阻塞。
- `blocks`：当前任务阻塞哪些任务。
- `owner`：任务负责人，后续 agent teams 会用到。

## TaskStatus

```rust
pub enum TaskStatus {
    Pending,
    InProgress,
    Completed,
    Deleted,
}
```

状态不再是散落的字符串，而是 Rust enum。通过 `strum` 给每个状态绑定展示 marker：

```text
[ ] pending
[>] in_progress
[x] completed
[-] deleted
```

## TaskManager

`TaskManager` 负责 `.tasks/` 里的持久化：

- `new()`：创建目录，扫描已有任务，计算下一个 id。
- `create()`：创建 `task_<id>.json`。
- `get()`：读取单个任务。
- `update()`：更新状态、owner 和依赖。
- `list_all()`：输出任务摘要。

任务文件示例：

```json
{
  "id": 1,
  "subject": "Implement background task manager",
  "status": "pending",
  "blockedBy": [],
  "blocks": [],
  "owner": ""
}
```

## 依赖维护

任务依赖有两个方向：

- `blockedBy`: 当前任务依赖谁。
- `blocks`: 当前任务阻塞谁。

当 `task_update` 添加 `addBlocks` 时，系统会同步更新被阻塞任务的 `blockedBy`。

当任务状态变成 `completed` 时，`clear_dependency()` 会从其他任务的 `blockedBy` 中移除当前任务 id。

这样 `.tasks/` 目录就变成一个简单的持久 work graph。

## SharedTaskManager

多个 task 工具需要共享同一个 manager，所以本章使用：

```rust
Arc<Mutex<TaskManager>>
```

并包成：

```rust
pub struct SharedTaskManager {
    inner: Arc<Mutex<TaskManager>>,
}
```

工具只持有 `SharedTaskManager`，不直接处理锁细节。

## task 工具

### task_create

```json
{
  "subject": "Write README",
  "description": "Draft the first version"
}
```

### task_get

```json
{
  "task_id": 1
}
```

### task_list

无参数，返回所有任务摘要。

### task_update

```json
{
  "task_id": 1,
  "status": "in_progress",
  "owner": "lead",
  "addBlockedBy": [2],
  "addBlocks": [3]
}
```

## Rust 实现要点

- 用 JSON 文件作为任务真实状态。
- 用 enum 表达有限状态集合。
- 用 serde 的 `rename` 在 Rust 字段名和 JSON 字段名之间转换。
- 用 `Arc<Mutex<_>>` 共享可变 task manager。
- 用 `merge_unique()` 保证依赖列表去重。

## 本章的局限

- 没有删除任务文件，只是支持 `deleted` 状态。
- 没有循环依赖检测。
- 没有任务搜索和过滤。
- 没有并发写入的跨进程文件锁。
- task 只是状态记录，不会自己执行。
