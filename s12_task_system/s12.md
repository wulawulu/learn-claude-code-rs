# s12

## 目标

`s12` 的核心目标是把 agent 从“任务只活在当前对话里”推进到“任务可以作为持久工作状态留在磁盘上”。

这一章对应 Python 原版里的核心思想是：

> task state survives compression because it lives on disk, not only inside the conversation

也就是：

- task 不是一段临时聊天上下文
- task 应该能跨上下文压缩继续存在
- task 之间可以有依赖关系
- agent 应该能通过工具读写这份持久 work graph

这一章最容易混淆的地方也正是 Python 原版反复强调的那一点：

- **task record 是 durable work item**
- **它不是线程，不是 worker，不是后台槽位**

所以 Rust 版这次的重点不是“实现一个并发调度器”，而是：

- 定义清楚什么叫 task record
- 让这份 task graph 落盘
- 把 task 操作接进同一个 tool loop

---

## 当前 Rust 版做了什么

当前 `s12` 可以概括成一句话：

**把任务系统从 prompt 里的临时计划，提升成 `.tasks/` 目录里的持久数据。**

对应到实现上，Rust 版主要做了三层收敛：

1. `TaskRecord / TaskStatus / TaskUpdate` 定义 durable task graph 的数据结构
2. `TaskManager` 负责磁盘上的 CRUD 和依赖维护
3. `task_create / task_get / task_list / task_update` 作为正常工具接回 agent loop

---

## 当前 Rust 版的结构

### 1. `TaskRecord`：任务是磁盘上的记录，不是运行时槽位

在 `src/task.rs` 里，Rust 版把任务定义成了显式的数据结构：

- `id`
- `subject`
- `description: Option<String>`
- `status`
- `blocked_by`
- `blocks`
- `owner`

这里最重要的两个点是：

- `description` 是可选字段，而不是用空字符串占位
- `blockedBy` 在 Rust 里用 `blocked_by` 表示，再通过 serde 映射回磁盘格式

这让 JSON 结构和 Rust 类型都比较自然。

同时，状态不再是散落的字符串判断，而是 `TaskStatus` enum：

- `Pending`
- `InProgress`
- `Completed`
- `Deleted`

这里按你的要求用了 `strum` 和 `strum_macros`：

- 状态字符串解析走 `EnumString`
- 列表 marker 走 enum property

所以像 `[ ] / [>] / [x] / [-]` 这样的显示逻辑，不再需要写成一串手工匹配字符串的分支。

### 2. `TaskManager`：把 durable task graph 落到 `.tasks/`

Python 原版的 `TaskManager` 重点讲的是：

- `task_*.json` 是真实状态
- `_next_id` 从现有文件推出来
- `create / get / update / list_all`
- 完成任务时要清掉其他任务里的 `blockedBy`

当前 Rust 版在 `src/task.rs` 里保留了同样的语义，但实现风格更 Rust：

- `TaskManager::new()` 启动时创建 `.tasks/`，并扫描现有文件计算 `next_id`
- `create()` 把新任务写成 `task_<id>.json`
- `get()` 读单个任务
- `update()` 统一处理状态、owner 和依赖更新
- `list_all()` 输出任务摘要列表

其中依赖图的语义和 Python 版保持一致：

- `add_blocked_by` 会补充当前任务依赖谁
- `add_blocks` 会补充当前任务会阻塞谁
- 当 `add_blocks` 出现时，也会同步回填被阻塞任务的 `blocked_by`
- 当任务变成 `Completed` 时，会从所有其他任务的 `blocked_by` 里移除它

也就是说，这里维护的是一个持久的 work graph，而不是“谁现在正在运行”的执行状态。

### 3. `SharedTaskManager`：共享 manager，但不把状态逻辑塞进主循环

在当前 `Tool` trait 下，工具是 `Box<dyn Tool>`，并且 `invoke(&mut self, ...)` 会被主循环统一调用。

这意味着 task 工具需要共享同一份 manager。

当前 Rust 版采用的是：

- `Arc<Mutex<TaskManager>>`
- 外面包一层 `SharedTaskManager`

这样做的好处是：

- 工具模块不需要自己处理锁细节
- `task_create / task_get / task_list / task_update` 都只依赖一个可克隆的共享句柄
- 主循环不需要知道任务系统内部怎么同步

这比把任务逻辑直接塞进 `agent_loop()` 可读性更好。

### 4. task 工具：让持久任务进入同一个 agent loop

当前 Rust 版在 `src/tool/` 下新增了四个工具：

- `task_create`
- `task_get`
- `task_list`
- `task_update`

它们和原有的 `bash / read_file / write_file / edit_file` 一样，都是普通工具。

也就是说，这一章的重要设计不是“另起一个任务系统入口”，而是：

- task 操作仍然通过 tool use 完成
- 模型不需要学习另一套控制流
- durable work state 和普通文件/命令工具一起进入同一个循环

在 `src/main.rs` 里，`SharedTaskManager` 会先初始化到当前工作目录下的 `.tasks/`，然后再把四个 task 工具一起装进 tool map。

所以这一章真正完成的是：

- task 数据落盘
- task 工具暴露给模型
- 主循环继续复用现有骨架

## 和 Python 原版的对应关系

如果按教学意图来对照，当前 Rust 版和 Python 原版是对应的。

Python 原版这章主要讲三件事：

1. `TaskManager` 定义磁盘上的 task record
2. task 工具进入和普通工具相同的 loop
3. task state 因为在磁盘上，所以能跨上下文压缩继续存在

当前 Rust 版这三点都保留了：

1. `TaskRecord` 和 `TaskManager` 明确表示 durable task graph
2. `task_create / task_get / task_list / task_update` 作为工具接入 loop
3. `.tasks/task_<id>.json` 是真实状态来源，不依赖当前对话是否还保留原文

所以这一章和 Python 原版在教学主线上是一致的：

- 都在讲“任务状态要脱离短期上下文”
- 都在讲“依赖关系要成为持久数据的一部分”
- 都在讲“task 是 durable work item，不是运行时 worker”

---

## 和 Python 原版的相同点

### 1. 都把任务存成磁盘上的记录

两边都采用：

- `.tasks/`
- `task_<id>.json`

所以压缩上下文时，task state 不会一起丢掉。

### 2. 都支持一个最小依赖图

两边都保留了：

- `blockedBy`
- `blocks`

并且都支持：

- 任务完成时解锁后继任务
- 增加 `blocks` 时同步回填被阻塞方

### 3. 都通过工具操作任务

两边都没有把 task 系统做成“主循环外的特殊命令模式”，而是：

- 继续通过 tool use 进入 agent loop

这让 task 系统仍然属于 agent 的正常能力，而不是旁路逻辑。

---

## 和 Python 原版的不同点

### 1. Rust 版把状态定义成 enum，而不是字符串

这是最明显的一处 Rust 风格收敛。

Python 原版里：

- `status` 是字符串

当前 Rust 版里：

- `status` 是 `TaskStatus` enum

这样做的好处是：

- 状态值受类型约束
- tool 输入解析更明确
- marker 这样的显示元数据也能和状态定义放在一起

### 2. Rust 版把共享 manager 包成了单独类型

Python 原版里，`TASKS` 是一个全局对象。

当前 Rust 版没有直接用全局可变状态，而是：

- `TaskManager` 负责核心逻辑
- `SharedTaskManager` 负责共享访问
- task 工具各自持有一份共享句柄

这更符合 Rust 的所有权和并发模型，也让代码边界更清晰。

### 3. Rust 版把 task 工具拆成独立模块

Python 原版为了教学紧凑，把 task handler 和 base tools 一起写在一个文件里。

当前 Rust 版没有照搬这个布局，而是拆成：

- `task.rs`
- `tool/task_create.rs`
- `tool/task_get.rs`
- `tool/task_list.rs`
- `tool/task_update.rs`

这让每个工具的输入解析和行为都更集中，也更便于后面继续扩展。

### 4. Rust 版把 `description` 做成了真正的可选字段

Python 原版在创建任务时即使没描述，也会写一个空字符串。

当前 Rust 版把它收敛成：

- `description: Option<String>`

这样磁盘上的 JSON 更干净，类型语义也更明确。

---

## 当前实现的价值

从教学上看，当前 `s12` 讲清楚了三件事：

1. task state 应该脱离当前对话，成为 durable work state。
2. 依赖关系不是 prompt 里的说明文字，而是应该进入持久数据结构。
3. task 系统不需要另起一套执行框架，也可以通过现有 tool loop 稳定接入。

所以如果用一句话总结当前 `s12`：

> Python 原版强调“任务要活在磁盘上而不是只活在上下文里”，Rust 版则把这件事落实成“用类型化的 task record、持久化 manager 和共享 task tools 来维护一个 durable work graph”。
