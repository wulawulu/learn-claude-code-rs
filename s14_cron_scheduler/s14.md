# s14

## 目标

`s14` 的核心目标是把 agent 从“只能响应眼前这轮输入”推进到“可以记住未来某个时间点要做的事”。

这一章对应 Python 原版里的核心思想是：

> scheduling remembers future work, then hands it back to the same main loop when the time arrives

也就是：

- agent 可以安排未来任务
- 任务到点后，不是走另一套执行系统
- 而是重新回到同一个主循环
- 最终仍然表现成一次普通对话里的用户消息注入

所以这一章最重要的不是 cron 语法本身，而是：

- 未来工作怎么被记住
- 到时间后怎么被唤回
- 调度结果怎么重新进入 agent loop

---

## 当前 Rust 版做了什么

当前 `s14` 可以概括成一句话：

**把“未来某个时间要做的提示”做成了可持久化、可后台检查、可重新注入主循环的调度系统。**

对应到实现上，Rust 版主要收敛成了三层：

1. `ScheduledTaskRecord` 负责表示调度记录
2. `SharedCronScheduler` 负责调度、持久化、锁和通知队列
3. `LoopState` 负责在每轮 LLM 请求前注入已触发的调度消息

这里没有照搬 Python 原版的全局对象和手写 cron 匹配，而是更偏 Rust 风格地把职责拆开。

---

## 当前 Rust 版的结构

### 1. `ScheduledTaskRecord`：未来任务是持久记录，不是瞬时回调

在 `src/cron.rs` 里，调度任务被建模成了显式结构：

- `id`
- `cron`
- `prompt`
- `mode`
- `persistence`
- `created_at`
- `last_fired_at`
- `jitter_offset_minutes`

这里最重要的两个类型是：

- `ScheduleMode`
- `PersistenceMode`

它们都用 `enum` 表达，并且通过 `strum` 派生了 `Display`。

所以像：

- `recurring`
- `one_shot`
- `session`
- `durable`

这些概念不再是零散布尔值的解释约定，而是变成了明确的类型。

这也是这一章的一个关键 Rust 化收敛：

- Python 原版里很多语义靠布尔值和注释维持
- Rust 版里尽量把这些概念提升成可读的类型

### 2. `SharedCronScheduler`：调度器负责“记住、检查、触发”

Python 原版里，`CronScheduler` 同时承担：

- 调度记录管理
- durable 加载/保存
- 后台检查
- 通知入队

当前 Rust 版保留了这四项职责，但收进了 `SharedCronScheduler` 这一个统一入口里。

它内部负责：

- `.claude/scheduled_tasks.json` 的持久化
- `.claude/cron.lock` 的文件锁协调
- 后台检查 loop
- 通知队列
- missed task 检测

这里的设计重点是：

- scheduler 自己知道怎样持久化
- scheduler 自己知道怎样尝试拿 cron lock
- scheduler 自己知道怎样在后台按时间检查

而外层不需要理解这些细节。

### 3. cron 匹配：直接复用现成 crate

Python 原版为了教学可读性，自己写了一个最小 cron 匹配器。

当前 Rust 版没有照搬这部分，而是直接使用：

- `cron` crate

也就是说，Rust 版这里不再试图“教学性手写 cron 解析”，而是明确做了工程化取舍：

- cron 表达式匹配交给现成库
- 当前章节重点放在调度系统如何接入 agent

这比继续维护手写表达式匹配更符合这版 Rust 代码的目标。

### 4. 后台检查：用 `tokio::interval` 驱动

Python 原版是：

- 后台线程
- 每秒 sleep 一次
- 每分钟只检查一次

当前 Rust 版保留了这个行为语义，但实现方式改成了：

- `tokio::spawn(...)`
- `tokio::time::interval(...).tick().await`

也就是说：

- 仍然是后台独立运行
- 仍然是定期检查
- 但现在和整个异步运行时模型更一致

这也是这一章一个比较明显的 Rust 化差异：

- Python 版强调“能看懂的最小线程 loop”
- Rust 版更自然地贴近 Tokio 生态

### 5. 通知注入：仍然回到同一个 agent loop

这部分是 `s14` 最重要的主线，Rust 版和 Python 原版完全一致。

在 `src/lib.rs` 里，`LoopState::agent_loop()` 每次真正发 LLM 请求前，都会先：

- drain 已触发的 cron notifications
- 把这些 notification 注入成新的 `user` 消息

也就是说，到点触发的任务不会绕开主循环，也不会自己直接调模型。

它做的是：

- 先变成一条普通消息
- 然后和其他对话上下文一起进入下一次模型调用

这就是这一章真正想教的东西：

- scheduling 不是另一套 agent
- 它只是把未来工作重新喂回现有 loop

### 6. 锁协调：现在是文件锁，不是 PID 文件协议

Python 原版用了：

- lock file
- PID 写入
- stale lock 检查

当前 Rust 版最终没有保留这套协议，而是用：

- `fs4`
- 基于锁文件的独占文件锁

并且锁逻辑没有再单独抽成一个 `CronLock` 结构，而是直接收进 scheduler 内部：

- `try_acquire_cron_lock(&self) -> Result<Option<File>>`

这让当前实现更高内聚，因为：

- lock file 路径本来就是 scheduler 的派生属性
- 锁只服务于 scheduler 自己的后台检查
- 没必要再为了这一层薄包装单独造一个类型

所以现在的语义是：

- 多个进程共享同一工作目录时
- 只有拿到 `.claude/cron.lock` 文件锁的那一个会真正执行 cron 检查

这能避免多个 session 同时重复触发同一轮 schedule。

## 和 Python 原版的对应关系

如果按教学意图来对照，当前 Rust 版和 Python 原版是对应的。

Python 原版这章主要讲的是：

1. agent 可以登记未来任务
2. 后台检查会在时间命中时把提示放进队列
3. 主循环会在下一次运行前把这些提示重新注入上下文

当前 Rust 版这三点都保留了：

1. `cron_create / cron_delete / cron_list` 负责调度工具入口
2. 后台 tick loop 会检查任务并产生 notification
3. `agent_loop()` 在真正请求模型前注入 notification

所以教学主线仍然是同一条：

- 记住未来工作
- 到点后唤醒
- 再交回同一个对话循环

---

## 和 Python 原版的相同点

### 1. 都支持 session / durable 两种持久化模式

两边都区分：

- 只在当前会话里存在的任务
- 持久化到磁盘的任务

所以 durable 调度任务在进程退出后仍然能重新加载。

### 2. 都支持 recurring / one-shot 两种调度模式

两边都支持：

- 周期性任务
- 触发一次后自动删除的任务

并且都保留了 recurring 自动过期的概念。

### 3. 都通过 notification queue 回到主循环

这是两边最核心的共同点。

调度器不是直接替主循环工作，而是：

- 把命中的任务变成通知
- 再由主循环把它们注入成新的消息

---

## 和 Python 原版的不同点

### 1. Rust 版不再手写 cron 匹配器

Python 原版自己实现了一个最小 cron 匹配函数，因为这更适合教学展示。

Rust 版当前没有保留这层，而是直接用 `cron` crate。

这意味着：

- Rust 版更工程化
- 也更少一段需要自己维护的表达式解析逻辑

### 2. Rust 版用 enum 收敛调度语义

Python 原版里：

- `recurring` / `durable` 是布尔值

Rust 版里：

- `ScheduleMode`
- `PersistenceMode`

是显式枚举类型。

这让代码里“这个任务属于哪种模式”更直观，不必每次都反向解释布尔值。

### 3. Rust 版后台检查更贴近 Tokio 运行时

Python 版用线程 + sleep。

Rust 版改成：

- `tokio::spawn`
- `interval.tick().await`

这让后台任务管理和 `stop().await` 的控制语义更自然。

### 4. Rust 版把锁逻辑并回 scheduler

Python 原版有一个单独的 `CronLock` 类型。

Rust 版最终把这层收掉了：

- 不再单独保留 `CronLock`
- 也不再保留单独的 guard 类型
- 直接在 scheduler 内部用 `Option<File>` 表示“当前是否持有锁”

这让代码更集中，也更符合当前这版实现的真实依赖关系。

---

## 当前实现的价值

从教学上看，当前 `s14` 讲清楚了三件事：

1. 未来任务不必马上执行，也可以作为持久状态被登记下来。
2. 到点后的工作不需要另起一套 agent，而是可以重新进入现有主循环。
3. 调度系统除了任务记录本身，还需要处理后台检查、持久化和多进程协调这些现实问题。

所以如果用一句话总结当前 `s14`：

> Python 原版强调“未来任务到点后会被重新注入主循环”，Rust 版则把这件事落实成“用类型化调度记录、Tokio 后台检查、文件锁协调和通知注入来维护一个可持续运行的 scheduler”。
