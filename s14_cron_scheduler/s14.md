# s14: Cron Scheduler

s13 解决了“慢命令不要阻塞当前 loop”的问题。本章解决另一个时间维度的问题：agent 如何记住未来某个时间点要做的事。

Cron scheduler 允许 agent 创建调度任务。任务到点后，不会绕开主循环直接执行，而是把预设 prompt 重新注入对话，让同一个 agent loop 继续处理。

## 运行方式

```bash
cargo run -p s14_cron_scheduler
```

启动后会看到：

```text
[Cron scheduler running. Background checks every second.]
[Commands: /cron to list tasks, /test to fire a test notification]
```

交互命令：

```text
/cron
/test
exit()
```

## 本章新增能力

- 新增 `cron_create` 工具创建调度任务。
- 新增 `cron_list` 工具列出任务。
- 新增 `cron_delete` 工具删除任务。
- 支持 recurring 和 one-shot。
- 支持 session-only 和 durable。
- durable 任务保存到 `.claude/scheduled_tasks.json`。
- 使用 `.claude/cron.lock` 避免多个进程重复触发。
- 调度命中后通过 notification queue 注入主循环。

## Scheduler 的核心语义

调度系统不直接替 agent 完成工作。它只做三件事：

```text
记住未来 prompt
  -> 到时间后生成 notification
  -> 在下一次模型请求前注入上下文
```

也就是说，未来任务最终仍然回到普通 agent loop。

## ScheduledTaskRecord

```rust
pub struct ScheduledTaskRecord {
    pub id: String,
    pub cron: String,
    pub prompt: String,
    pub mode: ScheduleMode,
    pub persistence: PersistenceMode,
    pub created_at: i64,
    pub last_fired_at: Option<i64>,
    pub jitter_offset_minutes: u64,
}
```

`mode` 表示触发方式：

```rust
pub enum ScheduleMode {
    Recurring,
    OneShot,
}
```

`persistence` 表示保存方式：

```rust
pub enum PersistenceMode {
    Session,
    Durable,
}
```

## cron_create

输入示例：

```json
{
  "cron": "0/5 * * * * * *",
  "prompt": "Check the background build status and summarize it.",
  "recurring": true,
  "durable": false
}
```

字段说明：

- `cron`：cron 表达式，由 `cron` crate 解析。
- `prompt`：触发时注入的提示。
- `recurring`：`true` 表示重复触发，`false` 表示触发一次后删除。
- `durable`：`true` 表示保存到磁盘，重启后加载。

## 后台检查

`SharedCronScheduler::start()` 会启动一个 Tokio 后台任务：

```rust
tokio::spawn(async move {
    scheduler.check_loop().await;
});
```

`check_loop()` 每秒 tick，但每分钟只实际检查一次任务，避免同一分钟重复触发。

## cron lock

多个 agent session 可能共享同一个工作目录。如果每个进程都检查 cron，就可能重复触发。

本章使用：

```text
.claude/cron.lock
```

并通过 `fs4` 尝试独占文件锁。只有拿到锁的进程会真正执行 cron 检查。

## Durable tasks

durable 任务保存到：

```text
.claude/scheduled_tasks.json
```

启动时 `start()` 会先调用 `load_durable()`。当 durable 任务创建、删除、触发或过期时，会重新保存。

## Notification 注入

调度命中后会发送：

```rust
CronNotification {
    task_id,
    prompt,
}
```

主循环在请求模型前调用：

```rust
self.inject_scheduled_notifications();
```

每条 notification 被作为 `User` 消息写入上下文：

```text
[Scheduled task <id>]: <prompt>
```

## one-shot、recurring 和过期

- `OneShot`：触发一次后自动删除。
- `Recurring`：每次匹配 cron 都会触发。
- recurring 任务超过 7 天会自动过期，避免长期无人维护的调度一直存在。

## jitter

对于分钟字段是 `0` 或 `30` 的 recurring cron，系统会计算一个 1 到 4 分钟的 jitter offset。

这可以避免大量任务都在整点或半点同时触发。

## Rust 实现要点

- 用 `cron::Schedule` 解析和匹配 cron 表达式。
- 用 `tokio::time::interval` 驱动后台检查。
- 用 `std::sync::mpsc` 保存 scheduler 到主循环的通知队列。
- 用 `fs4` 文件锁做跨进程协调。
- 用 enum 明确调度模式和持久化模式。

## 本章的局限

- 没有复杂时区配置，主要使用本地时间检查。
- durable 任务只有一个 JSON 文件，没有数据库。
- 没有调度任务编辑工具。
- 没有权限系统限制谁可以创建 durable cron。
- 后台 worker 只在当前进程运行。
