# s20: Tool Refactor

In earlier chapters, adding a tool requires a lot of repeated code: a struct, constructor, `Tool` trait implementation, JSON parsing, and a handwritten input schema.

This chapter refactors the tool system. Business tools can be written as async handlers with typed inputs. `schemars` generates schemas, `serde` parses inputs, `ToolRouter` dispatches calls, and a proc macro can generate wrappers.

## Run

```bash
cargo run -p s20_tool_refactor
```

The macro crate is:

```text
s20_tool_refactor_macros/
```

## New Capabilities

- Add `ToolRouter`.
- Add `ToolContext`.
- Generate `input_schema` from Rust types with `schemars`.
- Parse model JSON input into typed structs with `serde`.
- Add a `#[tool(...)]` proc macro.
- Separate agent runtime state from tool-accessible context.
- Write business tools as async handlers.

## Why Refactor Tools

Old tool implementations mix:

- metadata
- dynamic dispatch
- JSON parsing
- schema definition
- business logic

The goal is to make the business logic look like this:

```rust
#[derive(Debug, Deserialize, JsonSchema)]
pub struct LoadSkillInput {
    pub name: String,
}

#[tool(
    name = "load_skill",
    description = "Load the full body of a named skill into the current context."
)]
pub async fn load_skill(ctx: ToolContext, input: LoadSkillInput) -> anyhow::Result<String> {
    Ok(ctx.skill_registry.load_full_text(&input.name))
}
```

## AgentRuntime and ToolContext

The chapter separates two kinds of state.

`AgentRuntime` stores execution state:

- LLM client
- conversation context
- system prompt
- max round
- future permission / compact / recovery state

`ToolContext` stores business dependencies available to tools:

- `SkillRegistry`
- work directory
- future task, memory, worktree, or team managers

Tools should not receive the full `Agent`.

## Tool Trait

The trait remains as an internal dynamic dispatch interface:

```rust
#[async_trait::async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &'static str;
    fn description(&self) -> &'static str;
    fn input_schema(&self) -> serde_json::Value;

    async fn call(
        &self,
        context: ToolContext,
        input: serde_json::Value,
    ) -> anyhow::Result<String>;
}
```

Business code should not handwrite this implementation when a macro can generate it.

## ToolRouter

`ToolRouter` stores tool wrappers:

```rust
pub struct ToolRouter {
    tools: HashMap<String, Box<dyn Tool>>,
}
```

Registration:

```rust
let router = ToolRouter::new()
    .route(LoadSkillTool)
    .route(ReadFileTool)
    .route(EditFileTool);
```

Dispatch:

```text
Agent::agent_loop()
  -> model returns tool_use
  -> ToolRouter::call(ctx, name, input)
  -> wrapper parses typed input
  -> async handler runs
  -> tool_result returns
```

The router does not own context. The `Agent` owns `ToolContext` and passes a cheap clone at call time.

## serde and schemars

Responsibilities:

- `serde`: parse `serde_json::Value` into a Rust input struct.
- `schemars`: generate JSON Schema from that Rust input struct.

Example:

```rust
#[derive(Debug, Deserialize, JsonSchema)]
pub struct ReadFileInput {
    pub path: String,
    pub limit: Option<u64>,
}
```

`path` becomes required; `limit` is optional.

## #[tool] Macro

The macro generates a zero-state wrapper:

- tool name
- description
- input schema
- JSON deserialization
- handler call
- result conversion

Handlers that need shared dependencies receive `ToolContext`.

Pure function tools can expose function parameters directly as input fields.

## Agent

```rust
pub struct Agent {
    runtime: AgentRuntime,
    tool_context: ToolContext,
    tools: ToolRouter,
}
```

This avoids cycles:

```text
Agent owns runtime
Agent owns ToolContext
Agent owns ToolRouter
ToolRouter owns wrappers
wrappers do not own Agent
ToolContext does not own ToolRouter
```

## Limits

- Only one `ToolContext` type.
- No extractor system yet.
- `ToolRouter` is not generic over context type yet.
- Root agent, subagent, and teammate contexts are not separated yet.
- The macro supports only the handler shapes needed by this tutorial.
