# s20: Tool Refactor

前面的章节里，每新增一个工具都要手写 struct、构造函数、`Tool` trait 实现、JSON 参数解析和 input schema。工具数量变多后，这些模板代码会掩盖真正的业务逻辑。

本章重构工具系统：业务工具写成普通 async 函数，输入使用强类型 struct，schema 由 `schemars` 自动生成，注册和动态派发交给 `ToolRouter`，重复 wrapper 代码交给 proc macro。

## 运行方式

```bash
cargo run -p s20_tool_refactor
```

本章还包含一个宏 crate：

```text
s20_tool_refactor_macros/
```

## 本章新增能力

- 新增 `ToolRouter`。
- 新增 `ToolContext`，显式注入工具可访问依赖。
- 用 `schemars` 从 Rust 输入类型生成 `input_schema`。
- 用 `serde` 把模型传入的 JSON 解析成强类型 input。
- 新增 `#[tool(...)]` proc macro 自动生成 tool wrapper。
- 把 agent runtime state 和 tool context 分开。
- 业务工具从 trait impl 改成 async handler。

## 为什么要重构 Tool

旧写法中，一个简单工具也需要写很多模板代码：

```text
Tool struct
  -> constructor
  -> impl Tool
  -> 手动解析 serde_json::Value
  -> 手写 input_schema
  -> 调用业务逻辑
```

这些代码真正有价值的部分只有最后一步。

更理想的写法是：

```rust
#[derive(Debug, Deserialize, JsonSchema)]
pub struct LoadSkillInput {
    pub name: String,
}

#[tool(
    name = "load_skill",
    description = "Load the full body of a named skill into the current context."
)]
pub async fn load_skill(ctx: ToolContext, input: LoadSkillInput) -> anyhow::Result<String> {
    Ok(ctx.skill_registry.load_full_text(&input.name))
}
```

业务代码只描述输入和行为，框架负责解析、schema 和派发。

## AgentRuntime 和 ToolContext

本章把 agent 状态拆成两类。

`AgentRuntime` 保存运行时控制状态：

- LLM client。
- 对话上下文。
- system prompt。
- max round。
- 后续可以放 permission、compact、recovery 等执行策略。

`ToolContext` 保存工具 handler 可以访问的业务依赖：

- `SkillRegistry`
- work dir
- 后续可以加入 task manager、memory manager、worktree manager 等。

这个拆分很重要：工具不应该拿到完整 agent state。工具只需要完成业务动作所需的依赖。

## Tool Trait

trait 仍然存在，但它变成框架内部接口：

```rust
#[async_trait::async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &'static str;
    fn description(&self) -> &'static str;
    fn input_schema(&self) -> serde_json::Value;

    async fn call(
        &self,
        context: ToolContext,
        input: serde_json::Value,
    ) -> anyhow::Result<String>;
}
```

模型返回的仍然是工具名和 JSON input，所以运行时仍然需要 object-safe trait 做动态派发。区别是：业务代码不再手写 `impl Tool`，而是由宏生成。

## ToolRouter

`ToolRouter` 只负责保存工具表和动态调用：

```rust
pub struct ToolRouter {
    tools: HashMap<String, Box<dyn Tool>>,
}
```

注册方式接近路由表：

```rust
let router = ToolRouter::new()
    .route(LoadSkillTool)
    .route(ReadFileTool)
    .route(EditFileTool);
```

调用链路是：

```text
Agent::agent_loop()
  -> 模型返回 tool_use
  -> ToolRouter::call(ctx, name, input)
  -> 宏生成 wrapper 解析 input
  -> 执行业务 async handler
  -> 返回 tool_result
```

router 不持有 context。`Agent` 持有 `ToolContext`，调用工具时 clone 一份传进去。`ToolContext` 内部主要是 `Arc` 和 `PathBuf`，clone 成本很低。

## serde 和 schemars

本章使用两个 crate 分工：

- `serde`：把 `serde_json::Value` 解析成 Rust struct。
- `schemars`：从 Rust struct 生成 JSON Schema。

示例：

```rust
#[derive(Debug, Deserialize, JsonSchema)]
pub struct ReadFileInput {
    pub path: String,
    pub limit: Option<u64>,
}
```

`path` 会成为 required 字段，`limit` 会成为可选字段。这样 schema 和真实解析类型来自同一个 Rust 定义，不再需要手写两份。

## #[tool] 宏

`#[tool(...)]` 宏负责生成零状态 wrapper。

业务函数：

```rust
#[tool(name = "add", description = "Add two integers.")]
pub async fn add(a: i64, b: i64) -> i64 {
    a + b
}
```

宏生成的 wrapper 会完成：

- 返回工具名和描述。
- 生成 input schema。
- 反序列化 JSON input。
- 调用原始 async 函数。
- 把结果转成字符串。

需要共享依赖的工具显式接收 `ToolContext`：

```rust
pub async fn load_skill(ctx: ToolContext, input: LoadSkillInput) -> anyhow::Result<String>
```

纯函数工具可以直接把函数参数作为 input 字段。

## Agent

本章的 Agent 结构是：

```rust
pub struct Agent {
    runtime: AgentRuntime,
    tool_context: ToolContext,
    tools: ToolRouter,
}
```

这个结构避免循环依赖：

```text
Agent 拥有 runtime
Agent 拥有 ToolContext
Agent 拥有 ToolRouter
ToolRouter 拥有 tool wrappers
tool wrapper 不拥有 Agent
ToolContext 不拥有 ToolRouter
```

权限、恢复、压缩这类控制策略属于 runtime；skill、task、worktree 这类业务依赖属于 tool context。

## Rust 实现要点

- 用 proc macro 降低工具模板代码。
- 用 `schemars::JsonSchema` 生成模型可见 schema。
- 用 `serde::Deserialize` 做强类型输入解析。
- 用 object-safe trait 保留动态工具派发。
- 用零状态 wrapper 避免每个工具持有一份 context。
- 用 router 链式注册工具。

## 本章的局限

- 当前只有一个 `ToolContext` 类型。
- 还没有引入 Axum 风格 extractor。
- `ToolRouter` 还没有泛型化成 `ToolRouter<C>`。
- root agent、subagent、teammate 还没有各自独立的 context 类型。
- 宏只覆盖当前教程需要的 handler 形状。
