# s8

## 目标

s08 的核心目标是引入 hook system，让我们可以在**不改 agent loop 主体结构**的前提下，向系统中插入扩展行为。

这一章对应 Python 原版里的这句核心思想：

> Extend the agent without touching the loop.

Rust 版保留了这个目标，但在实现方式上做了几次重要收敛。最终版本不再机械复刻 Python，而是选择了：

- 保留同样的扩展点
- 保留同样的前后置控制思想
- 但用更符合 Rust 和 Anthropic tool-calling 约束的语义来表达

---

## 与 Python 原版的对应关系

Python 原版有 3 个事件：

- `SessionStart`
- `PreToolUse`
- `PostToolUse`

当前 Rust 版也保持了同样的 3 个事件名字：

- `SessionStart`
- `PreToolUse`
- `PostToolUse`

所以从教学上看，两者是一一对应的。

另外，Python 原版的 hook manager 做三件事：

1. 找到当前事件对应的 hooks
2. 依次执行
3. 聚合 block / inject / continue 结果

Rust 版最终也保留了这个流程，只是把实现介质从“外部命令 + json 配置”换成了“进程内闭包 + trait + 宏”。

---

## 为什么 Rust 版不直接照抄 Python

Python 原版的 hook 是通过 `.hooks.json` 配置外部命令来实现的。这个设计很适合教学，因为：

- 容易看懂
- 直观展示“系统如何开放扩展点”
- 不需要先引入类型系统

但 Rust 版如果直接照抄，会引入很多与这章主旨无关的复杂度：

- 配置文件加载
- 子进程执行
- 环境变量注入
- hook timeout
- workspace trust
- stdout/stderr 协议

这些东西都是真实系统里会出现的问题，但会把“hook 语义”本身冲淡。

所以 Rust 版做的第一个重要取舍是：

- **不先实现外部 hook runner**
- **先实现进程内 hook 语义**

这样这一章的注意力就能集中在：

- hook 在什么时机被调用
- hook 能拿到什么上下文
- hook 如何影响控制流

---

## 闭包 Hook 是一种优化

当前 Rust 版用的是闭包 hook：

- `SessionStartFn`
- `PreToolUseFn`
- `PostToolUseFn`

并且通过 blanket impl 让普通闭包自动变成对应的 hook trait。

这样做的好处是：

- 不需要外部命令执行器
- 不需要额外的协议层
- hook 逻辑就在 Rust 代码里
- 更方便讲清楚“扩展点”本身

这不是偏离原版，而是有意识的收敛。

可以理解为：

- Python 原版强调“hook system 是开放的”
- Rust 版先强调“hook system 的语义是什么”

也就是说，闭包 hook 并不是退化，而是把重点放回了教学目标。

---

## ToolUse / ToolResult 可变上下文是另一种优化

Python 原版里，hook 通过 `context: dict` 来读写当前阶段信息。

例如：

- pre hook 可以修改 `tool_input`
- post hook 可以读取 `tool_output`

这在 Python 里很自然，因为 dict 很便宜，也很灵活。

Rust 版没有继续使用“字符串 key + 动态 dict 协议”，而是改成了更直接的可变上下文：

- `PreToolUse` 拿到 `&mut ToolUse`
- `PostToolUse` 拿到 `&mut ToolResult`

这个设计的好处非常明显：

- 类型更明确
- 修改点更直接
- 不需要约定 `"tool_input"` / `"tool_output"` 这种字符串键
- hook 的职责更好理解

最终可以把 hook 分成两种能力：

1. 修改当前阶段上下文
2. 控制当前阶段是否继续

其中第 1 点是 Rust 版相对 Python 版非常重要的一次表达优化。

---

## 最开始为什么考虑过 Inject

Python 原版的 teaching contract 是：

- `0 -> continue`
- `1 -> block`
- `2 -> inject a message`

所以最开始 Rust 版也尝试过类似语义：

- `HookControl::Continue`
- `HookControl::Block(String)`
- `HookControl::Inject(String)`

并且还设计过 `HookOutcome` 来聚合：

- `blocked`
- `messages`

这在一开始看起来很自然，因为它和 Python 的“聚合 block/messages”非常接近。

---

## 为什么后来删掉了 Inject

这里的关键不是“Rust 写不出来”，而是 **Anthropic 的 tool-calling 结构约束更严格**。

我们讨论后的结论是：

- Anthropic 不适合让一个 `tool_use` 在消息流里对应很多额外插入的普通消息
- 尤其不适合在 `tool_use` 和唯一合法的 `tool_result` 之间夹入很多别的 block

Python 原版里：

- `PreToolUse` inject 会生成多个 `tool_result`
- `PostToolUse` inject 会把说明文字拼进最终 output

但在 Rust 版当前这套约束里，我们决定进一步收紧：

- `PreToolUse` 不再负责向 messages 注入内容
- `PostToolUse` 也不再通过返回值追加消息

原因是这会让 hook 的职责更清楚，也更贴合 Anthropic 的结构化工具调用。

最终保留下来的控制语义只有：

- `Continue`
- `Block(String)`

于是 `HookOutcome` 也就失去了意义，因为已经没有需要聚合的 `messages` 了。

---

## 当前最终语义

现在的 hook system 语义非常清楚：

### `SessionStart`

职责：

- 在 session 启动时运行一次
- 可以决定继续还是终止

返回值：

- `HookControl::Continue`
- `HookControl::Block(reason)`

如果 block，当前 session 直接停止。

### `PreToolUse`

职责：

- 可以修改 `ToolUse`
- 可以决定这个工具调用是否继续

它是一个“输入阶段 gate”：

- 改输入
- 决定是否允许工具执行

如果 block：

- 工具不会执行
- 框架会生成唯一一个 `tool_result`
- 内容类似：
  `Tool blocked by PreToolUse hook: ...`

这和 Python 原版“pre 阶段可以阻断执行”的思想一致，只是 Rust 版把消息流收得更严格。

### `PostToolUse`

职责：

- 可以修改 `ToolResult`
- 可以决定原始结果是否继续暴露给模型

它是一个“输出阶段 gate”：

- 改输出
- 决定这个结果是否需要被拦截/替换

如果 block：

- 工具已经执行了
- 但结果不会原样继续返回
- 框架会把 `ToolResult.content` 替换成：
  `Tool blocked by PostToolUse hook: ...`

这和 `PreToolUse` 的 block 在语义上不同：

- pre block：阻止执行
- post block：执行已发生，但结果被拦截

---

## 为什么说修改上下文也是 hook 的一等职责

这一点是当前 Rust 版非常重要的设计判断。

hook 不应该只被理解成“做一些 side effect 的回调”。

在这章里，它的核心职责其实有两个：

1. **transform**
   修改当前阶段的数据

2. **control**
   决定当前阶段是否继续

所以：

- `PreToolUse` 的 `&mut ToolUse`
  不是附带能力，而是核心能力

- `PostToolUse` 的 `&mut ToolResult`
  也不是附带能力，而是核心能力

这比“额外往消息流注入说明文字”更本质。

可以说，Rust 版在这里把 hook 的真正职责表达得比 Python 原版更清楚。

---

## 为什么 `invoke_hooks!` 宏仍然保留

虽然当前实现已经不再有 `HookOutcome`，但 `invoke_hooks!` 宏仍然保留。

原因不是为了“用宏而用宏”，而是它确实在抽象一类重复模板：

- 根据 `HookTypes` 取对应 hooks
- downcast 到正确的 enum variant
- 传入不同参数列表
- await 执行
- 处理 `Continue / Block`
- 在 `Block` 时短路

`SessionStart / PreToolUse / PostToolUse` 的主要区别就是：

- 事件种类不同
- 参数形态不同

而其执行模板是相同的。

因此这里保留宏是合理的，它解决的是：

- “不同 hook，重复模板代码”

而不是：

- “隐藏主流程控制”

主流程控制仍然清楚地留在：

- `main.rs` 的 SessionStart 处理
- `execute_tool_call` 的 pre-block / execute / post-block 处理

所以这一层抽象是克制的。

---

## 为什么 Block 要短路后续 Hook

当前实现中，一旦某个 hook 返回 `Block(reason)`，当前阶段剩余 hooks 会立即停止。

这是有意的设计，而不是实现细节。

原因：

- `Block` 表示这个阶段已经得到明确控制结论
- 再继续跑后续同阶段 hook，会让语义变模糊
- 后续 hook 即使继续修改上下文，也通常没有意义
- 还可能出现多个 hook 同时想 block / mutate，最终语义不清

所以现在的规则是：

- `Continue`：继续执行后续 hook
- `Block`：立即短路当前阶段剩余 hook

这比“全部执行完再决定是否 block”更清楚，也更容易推理。

---

## 现在这版和 Python 原版的异同

### 相同点

- 都有三个清晰的 hook 时机
- 都允许 hook 影响主循环而不需要改 loop 结构
- 都有前置和后置两个最重要的 tool 扩展点
- 都支持“继续 / 阻断”这类控制语义

### Rust 版的不同点

- 使用闭包而不是外部命令
- 使用 trait + enum 而不是 exit code + dict
- 使用 `&mut ToolUse` / `&mut ToolResult` 而不是动态 context
- 删除了 inject 语义
- 删除了 messages 聚合
- 将 block 严格收敛到唯一 `tool_result` 结构

### 这些不同点背后的判断

这些差异不是“没做完”，而是有意的教学取舍：

- Python 原版强调 hook 机制本身可以开放到外部
- Rust 版强调 hook 的控制语义和上下文变换语义

并且在 Anthropic 工具调用约束下，Rust 版进一步选择：

- 保证一个工具调用只走一个清晰的 `tool_result` 返回路径
- 不让 hook 再去任意向 messages 注入内容

这让系统更稳，也让 hook 的职责边界更清楚。

---

## 当前实现的一个简洁总结

如果只用一句话描述当前 `s08`，那就是：

> Hook 不再是“附加几条消息”，而是“在输入和输出两个关键阶段上提供可变上下文和显式控制”。

也就是说：

- `PreToolUse` = 修改输入 + 决定是否继续
- `PostToolUse` = 修改输出 + 决定是否允许原结果继续暴露

这就是当前 Rust 版相对 Python teaching version 最重要的收敛。

---

## 这一章的最终收获

s08 之后，agent 不再只是：

- LLM
- tools
- loop

而是开始具备了真正的“扩展面”：

- 不用改主循环
- 就能接入横切逻辑
- 并且扩展点有明确时机和明确语义

这是后面继续做：

- permission
- policy
- audit
- guardrail
- plugin

这些系统的基础。
