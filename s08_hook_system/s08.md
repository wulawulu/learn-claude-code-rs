# s08: Hook System

s03 的 todo reminder、s07 的权限检查都暴露出同一个问题：有些逻辑不是模型主动调用的工具，但它们需要参与 agent 生命周期。

比如：

- session 启动时初始化环境。
- 工具执行前检查、改写或阻断输入。
- 工具执行后记录日志、过滤结果或替换输出。

如果这些逻辑都写进 agent loop，主循环会越来越难维护。本章引入 hook system，让扩展行为可以挂到明确的生命周期节点上。

## 运行方式

```bash
cargo run -p s08_hook_system
```

当前 demo 在 `main.rs` 注册了三个简单 hook：

- session start 时打印初始化信息。
- tool use 前打印工具调用。
- tool use 后打印工具结果。

## 本章新增能力

- 定义 `SessionStart`、`PreToolUse`、`PostToolUse` 三类 hook。
- 用闭包注册 hook。
- 用 `HookControl` 控制继续或阻断。
- `PreToolUse` 可以修改 `ToolUse`。
- `PostToolUse` 可以修改 `ToolResult`。
- 用宏统一触发 hook。

## Hook 和 Tool 的区别

Tool 是模型主动调用的能力：

```text
模型 -> tool_use -> harness 执行工具
```

Hook 是 harness 主动触发的生命周期扩展：

```text
harness 到达某个阶段 -> 执行 hook -> 决定是否继续
```

简单判断标准：

- 如果这件事应该由模型决定是否调用，它更像 tool。
- 如果这件事应该在某个阶段自动发生，它更像 hook。

## HookControl

```rust
pub enum HookControl {
    Continue,
    Block(String),
}
```

hook 的控制结果只有两种：

- `Continue`：继续当前流程。
- `Block(reason)`：阻断当前阶段，并提供原因。

本章没有设计“向消息流注入额外消息”的能力，是为了保持 tool calling 消息结构简单：一次 `tool_use` 对应一个明确的 `tool_result`。

## ToolUse 和 ToolResult

```rust
pub struct ToolUse {
    pub id: String,
    pub name: String,
    pub input: serde_json::Value,
}

pub struct ToolResult {
    pub tool_use_id: String,
    pub content: String,
}
```

这两个结构是 hook 可以观察和修改的阶段上下文。

`PreToolUse` 拿到 `&mut ToolUse`，所以它可以：

- 改写工具名。
- 修改工具参数。
- 阻断工具执行。

`PostToolUse` 拿到 `&mut ToolResult`，所以它可以：

- 截断过长输出。
- 隐藏敏感内容。
- 添加审计信息。
- 阻断原始结果继续暴露给模型。

## 三类 hook

### SessionStart

在 session 开始时运行一次。

适合做：

- 初始化日志。
- 检查环境。
- 根据 workspace 状态决定是否允许启动。

如果返回 `Block`，当前 session 直接停止。

### PreToolUse

在每次工具执行前运行。

适合做：

- 权限检查。
- 参数规范化。
- 路径重写。
- 拦截危险调用。

如果返回 `Block`，工具不会执行，harness 会生成一个 tool result：

```text
Tool blocked by PreToolUse hook: <reason>
```

### PostToolUse

在工具执行后运行。

适合做：

- 输出截断。
- 敏感信息过滤。
- 记录审计日志。
- 把底层错误翻译成更清楚的提示。

如果返回 `Block`，工具已经执行，但结果会被替换为：

```text
Tool blocked by PostToolUse hook: <reason>
```

## 闭包注册

`LoopState` 提供了三个注册方法：

```rust
state.session_start(|_| { ... });
state.pre_tool(|_, tool_use| { ... });
state.post_tool(|_, tool_use, tool_result| { ... });
```

这些方法接收闭包，内部包装成 `Hook` enum 存入 `state.hooks`。

这种设计适合教学阶段：不需要引入配置文件、外部进程或插件协议，就能先把生命周期语义讲清楚。

## invoke_hooks! 宏

[`src/utils.rs`](./src/utils.rs) 定义了：

```rust
invoke_hooks!(PreToolUse, self, &mut tool_use)
```

宏负责：

- 按 hook 类型筛选。
- 顺序执行。
- 遇到 `Block` 后停止后续 hook。
- 返回最终控制结果。

这让 agent loop 里的 hook 调用保持简洁。

## Rust 实现要点

- 用 trait 表达不同 hook 签名。
- 用 blanket impl 让闭包自动实现 hook trait。
- 用 `Pin<Box<dyn Future<...>>>` 表达异步闭包返回值。
- 用 enum 包装不同 hook 类型。
- 用宏减少重复的分发代码。

## 本章的局限

- hook 只存在进程内，不能通过配置文件动态加载。
- hook 没有优先级。
- hook 没有命名、启停和持久化配置。
- `SessionStart` 只在 `main.rs` 手动触发，还没有完全纳入统一生命周期管理。
- 权限系统还没有真正迁移成 hook。
