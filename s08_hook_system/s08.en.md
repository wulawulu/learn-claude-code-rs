# s08: Hook System

s07 adds permission checks before tool execution. That solves one important safety problem, but a mature agent harness usually needs more lifecycle extension points:

- Log every tool call.
- Inspect or rewrite tool input.
- Filter tool output.
- Block calls based on external policy.
- Run startup logic.

This chapter introduces a hook system so these behaviors can be attached to the agent loop without hard-coding every feature into the loop itself.

## Run

```bash
cargo run -p s08_hook_system
```

## New Capabilities

- Define hook types for agent lifecycle events.
- Support `SessionStart`, `PreToolUse`, and `PostToolUse`.
- Let hooks continue or block execution.
- Allow `PreToolUse` to inspect or modify tool input.
- Allow `PostToolUse` to inspect or modify tool output.
- Keep hook logic outside tool implementations.

## Why Hooks

The agent loop should stay small:

```text
model -> tool_use -> execute tool -> tool_result -> model
```

But real systems need extra behavior around that loop. If every feature is written directly into the loop, it becomes hard to reason about.

Hooks provide a controlled extension surface:

```text
PreToolUse hook
  -> permission / logging / filtering
  -> execute tool
  -> PostToolUse hook
```

## Hook Types

The chapter defines three hook points:

- `SessionStart`: when a session starts.
- `PreToolUse`: before a tool call executes.
- `PostToolUse`: after a tool call returns.

`PreToolUse` receives a mutable `ToolUse`, so it can inspect or rewrite input.

`PostToolUse` receives the original `ToolUse` and a mutable `ToolResult`, so it can inspect or rewrite output.

## HookControl

Hooks return a control decision:

```rust
pub enum HookControl {
    Continue,
    Block(String),
}
```

- `Continue`: keep going.
- `Block(reason)`: stop this operation and return the reason.

This keeps hook behavior explicit. A hook does not need to panic or throw a special error to stop execution.

## ToolUse and ToolResult

The hook layer uses lightweight structs:

```rust
pub struct ToolUse {
    pub id: String,
    pub name: String,
    pub input: Value,
}

pub struct ToolResult {
    pub tool_use_id: String,
    pub content: String,
}
```

These are not the model SDK types. They are harness-level data structures, easier to pass to hook functions.

## Running Hooks

The agent loop calls hooks around tool execution:

```text
PreToolUse
  -> execute tool
  -> PostToolUse
```

If a `PreToolUse` hook blocks, the tool is not executed.

If a `PostToolUse` hook blocks, the tool output can be replaced by a blocked message.

## Example Uses

Hooks can implement:

- Audit logging.
- Command deny rules.
- Result redaction.
- Tool call metrics.
- Workspace policy checks.
- Debug traces.

The point is not that every one of these exists in this chapter. The point is that the runtime now has a clean place to add them.

## Rust Notes

- Use boxed function traits for hook callbacks.
- Use explicit structs instead of passing raw SDK message blocks.
- Keep hook registration on `Agent` / `LoopState`.
- Keep the agent loop responsible for ordering hook execution.

## Limits

- Hooks are registered in code, not loaded from config.
- No hook priority.
- No async hook pipeline beyond the current shape.
- `SessionStart` is defined but only lightly used.
- No persistent hook audit log yet.
