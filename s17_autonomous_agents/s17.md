# s17

## 目标

`s17` 的核心目标，是把 `s16` 的“带协议的团队协作”推进到“teammate 可以脱离 lead 持续驱动、空闲轮询、并自动认领任务”。

如果说：

- `s15` 解决的是 mailbox 协作
- `s16` 解决的是 durable protocol handshake

那么 `s17` 真正新增的是：

- teammate 不再只是被 lead 一次次显式驱动
- teammate 可以在 `work -> idle polling -> work / shutdown` 之间自己循环
- task 也不再只是手工分配，而是可以按 role 自动认领

这一章最重要的建模，不是“再加几个工具”，而是把团队成员的运行方式从：

- 一次 request，一次响应

推进到：

- 一个长期存活的 autonomous worker

---

## 当前 Rust 版做了什么

当前 `s17` 可以概括成一句话：

**在 `s16` 的协议团队之上，把 teammate 变成了会自己 idle、自己轮询 inbox、自己 auto-claim task 的长期 worker。**

Rust 版目前收敛成了四层：

1. `LoopState` 负责 agent 自己的主 loop 和 autonomous teammate loop
2. `SharedTeammateManager` 负责 team/message/request/runtime 基础设施
3. `SharedTaskManager` 负责 durable task state 和 claim/update
4. tools 负责把 lead / teammate 可做的动作暴露给模型

其中最关键的变化是：

- `lead` 仍然是 orchestration 入口
- 但 teammate 一旦被 `spawn`，就不再依赖 lead 每轮显式调度

它会自己：

- 先工作
- 没有工具调用或主动 `idle` 后进入 idle polling
- 周期性检查 inbox
- 自动认领匹配 role 的 ready task
- 在长时间空闲时自己 shutdown

## 当前 Rust 版的结构

### 1. `LoopState`：现在不只是一轮 agent loop

`s16` 里的 `LoopState` 主要负责：

- 拉取 inbox
- 请求模型
- 执行 tool call

到了 `s17`，`LoopState` 仍然保留这部分职责，但又新增了一层 autonomous teammate lifecycle：

- `agent_loop()`：单次 work phase
- `run_autonomous_teammate_loop()`：长期 teammate loop
- `run_work_phase()`：把 work 子循环结果映射成外部状态
- `run_idle_polling_phase()`：空闲轮询 inbox 和 auto-claim

也就是说：

- `agent_loop()` 还是“工作回合”
- 但 autonomous teammate 已经变成了“工作回合之上的生命周期状态机”

这一步很关键，因为它把两层逻辑分开了：

- agent 如何思考和用工具
- teammate 何时工作、何时 idle、何时 shutdown

### 2. `TeammateStatus`：三态语义已经稳定下来

当前 Rust 版把 teammate 的外部持久状态收成了三个状态：

- `Working`
- `Idle`
- `Shutdown`

这三个状态的语义是：

- `Working`：worker 仍在主动工作回合里跑模型和工具
- `Idle`：worker 仍然存活，但处于 polling inbox / auto-claim 的空闲期
- `Shutdown`：loop 已经结束，runtime 已被回收

这里最重要的不是“状态数量变少”，而是：

- `Idle` 明确表示“还活着，只是不在当前工作回合”
- `Shutdown` 明确表示“生命周期结束”

这比把“没在工作”和“已经退出”混在一起要清楚得多。

### 3. `agent_loop()` 的退出语义：工作结束并不等于 shutdown

当前 `agent_loop()` 的返回值是：

- `AgentLoopExit::Idle`
- `AgentLoopExit::Shutdown`

这里的关键边界是：

- 如果模型不再发 `ToolUse`
- 或显式调用了 `idle`
- 或达到单次工作回合的 `max_rounds`

都只是结束这次 work phase，返回 `Idle`

而不是直接 `Shutdown`。

真正的 `Shutdown` 只来自两类情况：

- 收到 stop signal / shutdown request
- idle polling 超时后仍然没有新消息、也没有可认领任务

这一点是 `s17` 很重要的行为语义：

- work phase 负责“做事”
- idle phase 负责“等事”
- 只有 idle phase 长时间等不到事，才真正退出

### 4. idle polling：`Idle` 不是静止，而是轻量轮询

当前 teammate 的 idle phase 不是纯粹什么都不做，而是：

1. sleep 一个 polling interval
2. 检查 stop signal
3. 读 inbox
4. 如果 inbox 有消息，就恢复成 `Working`
5. 如果 inbox 没消息，就扫描可 auto-claim 的任务
6. 如果有可认领任务，也恢复成 `Working`
7. 如果持续到 timeout 仍然没有任何事，就 `Shutdown`

所以 `Idle` 更准确地说是：

- 低频存活状态
- 不是工作态
- 但也不是死亡态

这和传统 worker 的 long-poll / periodic poll 很像。

### 5. identity context：idle 恢复后仍能记住“我是谁”

一个 autonomous teammate 从 idle 恢复到 work 时，不能丢失自己的身份。

当前 Rust 版在 `LoopState` 里显式保存了：

- `name`
- `role`
- `team_name`

并通过：

- `with_identity(...)`
- `ensure_identity_context()`
- `resume_from_idle_with_inbox(...)`
- `resume_from_idle_with_auto_claim(...)`

保证 idle 恢复后上下文里仍然有稳定的 identity block。

这一层的意义是：

- teammate 不是匿名工具调用器
- 它是带角色的长期 worker

所以当它从 idle 恢复时，应该继续以“我是这个 teammate”的视角工作，而不是像一次全新的 agent 调用。

### 6. `SharedTaskManager`：task 从“记录”升级成了“可自治调度的工作项”

`s17` 里的 task 不只是能创建和更新，还新增了 claim 相关语义。

当前 `TaskRecord` 增加了：

- `owner`
- `claimed_at`
- `claim_source`
- `claim_role`
- `required_role`

同时有了几条关键判断逻辑：

- `claim_role()`
- `allows_role(...)`
- `is_claimable(...)`

这意味着 task 的含义已经从：

- “一条可持久化的任务记录”

变成：

- “一个可由团队成员按条件认领的工作对象”

当前 claim 的条件包括：

- `status == pending`
- 还没有 owner
- 没有 `blocked_by`
- role 匹配

所以 `s17` 新增的不只是 task API，而是一个最小可用的 pull-based work queue 语义。

### 7. auto-claim：任务分配第一次脱离 lead 手工操作

`SharedTeammateManager::auto_claim_task(...)` 是这一章最关键的新能力之一。

它做的事很简单：

1. 扫描当前 role 可认领的 unclaimed task
2. 拿第一条 ready task
3. 以 `ClaimSource::Auto` 认领
4. 把认领结果和完整 task record 返回给 teammate loop

然后 idle 中的 teammate 会：

- 把这条 task 注回上下文
- 恢复成 `Working`
- 继续处理它

这说明一个重要的系统变化：

- 之前任务主要靠 lead 主动分派
- 现在 teammate 已经能从 shared backlog 里自己“拉任务”

这也是 `autonomous_agents` 这个名字真正开始成立的地方。

### 8. protocol 还在，但不再是主循环的全部

`s16` 的重点是 protocol：

- shutdown request / response
- plan approval submit / review

到 `s17`，这些 protocol 仍然存在，而且仍然 durable：

- `.team/requests/*.json`
- `RequestRecord`
- `SharedRequestStore`

但它们在系统里的地位发生了变化：

- 不再是 teammate 唯一重要的行为
- 而是被嵌入 autonomous teammate lifecycle 的协作约束

比如：

- teammate 可以自己工作，但大工作仍然要 `plan_approval`
- teammate 可以自己 idle，但收到 `shutdown_request` 仍然要回应

所以 `s17` 的重点不是替代 protocol，而是把 protocol 放进一个更真实的长期 worker 生命周期里。

### 9. shutdown：请求停止和真正退出被重新分开

当前 Rust 版里，shutdown 语义已经比较清楚：

- lead 通过 `shutdown_request` 发出请求
- teammate 在自己的 loop 中读到请求
- teammate 通过 `shutdown_response` 表示 approve / reject
- `respond_shutdown(..., approve=true)` 只负责把 stop signal 设上
- 真正退出仍由 teammate 自己在 loop 中检查 signal 后返回 `Shutdown`
- teammate task 返回后，由外层 spawn 包装层统一 cleanup

这一点非常重要：

- manager 不再主导“把 teammate 关掉”
- manager 只负责发 stop signal 和最终清理注册表
- 真正决定退出的是 teammate 自己的 lifecycle

这让 shutdown 语义和 idle-timeout shutdown 保持一致：

- 都是 teammate 自己走到 `Shutdown`
- 只是触发原因不同

### 10. cleanup：manager 只在 task 真正结束后回收 runtime

当前 `team/cleanup.rs` 里的职责已经收得比较单纯：

- `request_teammate_stop(name)`：设置 stop signal
- `cleanup_finished_teammate(name)`：task 结束后移除 handle / signal，并落 `Shutdown`

这里最重要的收敛是：

- 不再由 manager 在 shutdown approval 时主动起一条 reaper 主流程
- runtime cleanup 统一回到 spawn 包装层

也就是说：

- stop 是协作式的
- cleanup 是 task 结束后的善后

这比把“请求停止”和“回收现场”缠在一条 manager 流程里要干净很多。

### 11. `SharedTeammateManager`：现在更像 autonomous team runtime facade

到 `s17` 为止，`SharedTeammateManager` 已经不再只是：

- teammate registry
- message facade
- protocol facade

它现在更像整个 autonomous team runtime 的统一门面，负责：

- teammate spawn / list / status
- mailbox message bus
- durable request workflow
- runtime stop signal / cleanup
- task auto-claim 的 team-facing 编排

不过当前实现已经把它拆到了几个文件里：

- `message.rs`
- `requests.rs`
- `cleanup.rs`
- `storage.rs`
- `teammate.rs`

这一步很重要，因为它让 manager 仍然是一个 facade，但不再是一整个难读的大文件。

### 12. tools：lead 和 teammate 的工具面已经开始分化

当前 `tool/mod.rs` 很清楚地分成了两套工具：

- `leader_tools(...)`
- `teammate_tools(...)`

其中 lead 侧可以：

- `spawn_teammate`
- `shutdown_request`
- `plan_approval` review
- `task_create / task_list / task_update`
- `claim_task`
- 普通 message / broadcast / inbox 操作

而 teammate 侧更偏执行者能力：

- `send_message`
- `read_inbox`
- `shutdown_response`
- `plan_approval` submit
- `claim_task`
- `idle`
- 文件与 shell 工具

这说明一件事：

- lead 和 teammate 不再只是“同一个 agent 用不同 prompt”
- 它们的可用动作面已经开始显式分化

## 和 s16 的关系

从教学主线上看，`s17` 不是在 `s16` 上“多加一个 task API”，而是把协作模型推进了一大步。

`s16` 的重点是：

- 团队成员之间可以用协议做 durable handshake

`s17` 的重点是：

- 团队成员可以在协议约束下长期自治运行

所以两章的关系更像：

- `s16` 教的是 structured coordination
- `s17` 教的是 autonomous execution under coordination

具体来说，`s17` 在 `s16` 基础上新增了三件事：

1. teammate lifecycle 从单次 loop 扩成长期 loop
2. task 变成 role-aware claimable work item
3. idle polling + auto-claim 让 teammate 可以脱离 lead 持续推进工作

---

## 当前实现最值得注意的设计点

### 1. 内部控制流和外部持久状态已经重新对齐

这一版最重要的收敛之一，是：

- teammate 的 lifecycle 只围绕 `Working / Idle / Shutdown`
- 状态只在真正变化时写一次

这比“每轮 loop 都写一次状态”要合理得多，也避免了无意义的锁和落盘。

### 2. shutdown 的主导权在 teammate，不在 manager

这一点非常关键。

当前语义下：

- lead 只能请求停
- manager 只能设 stop signal
- teammate 自己决定退出

这样 shutdown 和 idle-timeout 的语义才统一。

### 3. runtime cleanup 被放回了 task 退出后

cleanup 现在不再是“manager 觉得该收了就收”，而是：

- task 结束
- spawn 包装层看到它结束
- 再统一清理 runtime 注册状态

这让 stop / exit / cleanup 三件事的边界清楚很多。

### 4. `LoopState` 的内聚性明显提高了

当前和 autonomous worker 本身强相关的东西都已经在 `LoopState`：

- identity
- inbox 注入
- idle 恢复
- auto-claim 恢复
- work / idle / shutdown 状态流转

这比把这些逻辑散在 manager 里更像“从一开始就这么设计”的 Rust 风格。

## 当前实现的边界

虽然 `s17` 已经能跑出一个很清楚的 autonomous teammate 模型，但它仍然有明确边界。

### 1. idle polling 仍然是简单定时轮询

现在还是：

- 固定 `POLL_INTERVAL_SECS`
- 固定 `IDLE_TIMEOUT_SECS`

它还不是：

- event-driven mailbox wakeup
- 真正的 queue consumer

### 2. auto-claim 仍然是最简单的 first-ready policy

当前只是：

- 扫描可认领任务
- 取第一条

它还没有：

- 优先级
- fairness
- 租约 / lease
- 重试和回收

### 3. plan approval 仍然只是 protocol，不是 scheduler constraint engine

现在的语义是：

- 大工作前可以走审批

但它还不是：

- 强制的 task execution policy engine
- 可由系统自动阻断的一整套审批栅栏

### 4. cleanup 现在语义正确，但还是最小实现

目前 cleanup 已经回到正确层级：

- task 自己结束
- manager 统一做注册表回收

但它还没有扩展成：

- 显式 wait/join API
- 批量 shutdown / wait
- 更完整的 runtime supervision

---

## 这章真正新增的能力

如果只保留高层结论，`s17` 真正新增的是这几件事：

- autonomous teammate lifecycle
- `Working / Idle / Shutdown` 三态 worker 模型
- idle polling
- role-aware auto-claim
- task 从被分配对象变成可被拉取的工作对象
- protocol 进入长期自治 worker 语境

其中最重要的一句可以总结成：

**从这一章开始，teammate 已经不是“被 lead 驱动的一次性执行器”，而是“会自己等活、找活、接活、停活”的长期 worker。**

---

## 小结

`s17` 的核心，不是“再加一个 idle 工具”，也不是“再加一个 task claim API”。

它真正完成的是一次系统层次上的推进：

- `s16` 里，team protocol 已经 durable
- `s17` 里，team worker 本身也开始 autonomous

因此这一章最值得记住的，不是某个具体函数，而是下面这组边界：

- `LoopState` 负责 autonomous worker lifecycle
- `SharedTeammateManager` 负责 team runtime infrastructure
- `SharedTaskManager` 负责 durable work items
- protocol 负责约束协作，而不是替代自治

如果继续往后推进，这一章就是后续更复杂 multi-agent runtime 的真正起点。
