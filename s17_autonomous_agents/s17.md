# s17: Autonomous Agents

s16 让团队协作具备了结构化协议，但 teammate 仍然主要由 lead 显式驱动。本章把 teammate 改造成长期运行的 worker：它可以工作、进入 idle、轮询消息、自动认领任务，并在长时间无事可做时退出。

这一步的重点不是多加几个工具，而是让 agent harness 开始拥有 autonomous worker lifecycle。

## 运行方式

```bash
cargo run -p s17_autonomous_agents
```

运行后会使用：

```text
.team/
.team/inbox/
.team/requests/
.tasks/
```

团队状态、协议请求和任务记录都会落盘。

## 本章新增能力

- teammate 可以长期存活，而不是只执行一次 prompt。
- 新增 `Working / Idle / Shutdown` 生命周期状态。
- 新增 `idle` 工具，让 teammate 主动结束当前工作回合。
- idle phase 会定时轮询 inbox。
- teammate 可以按角色自动认领 ready task。
- task 记录增加 owner、claim 信息和 required role。
- shutdown signal 和 runtime cleanup 分离。

## 为什么需要 Autonomous Worker

一个团队 agent 如果完全依赖 lead 驱动，会出现两个问题：

- lead 必须记住什么时候继续唤醒 teammate。
- teammate 没有自己的“等活”和“接活”能力。

本章引入的 autonomous teammate 运行方式是：

```text
work phase
  -> 没有更多工具调用或主动 idle
  -> idle polling phase
  -> 收到 inbox 消息或 claim 到任务
  -> 回到 work phase
  -> 长时间空闲或收到 stop signal
  -> shutdown
```

这让 teammate 从一次性执行器变成一个最小 worker runtime。

## LoopState

本章的 `LoopState` 不只负责“请求模型并执行工具”，还负责 autonomous teammate 的生命周期。

主要分成几层：

- `agent_loop()`：执行一个工作回合。
- `run_autonomous_teammate_loop()`：长期 teammate loop。
- `run_work_phase()`：把一次工作回合映射成 Idle 或 Shutdown。
- `run_idle_polling_phase()`：空闲轮询 inbox 和可认领任务。

这样可以把两个问题分开：

- agent 如何思考和调用工具。
- teammate 什么时候工作、等待或退出。

## TeammateStatus

teammate 的持久状态收敛为三种：

```rust
Working
Idle
Shutdown
```

含义分别是：

- `Working`：正在主动运行模型和工具。
- `Idle`：worker 仍然存活，但暂时没有工作。
- `Shutdown`：worker loop 已结束，runtime 资源已回收。

这里最重要的是区分 `Idle` 和 `Shutdown`。空闲不是死亡，它仍然会轮询 inbox 和任务列表。

## Work Phase

一个 work phase 仍然是熟悉的 agent loop：

```text
构造请求
  -> 调用模型
  -> 执行 tool_use
  -> 回填 tool_result
  -> 继续或结束
```

但它的退出结果不再只有“结束”。

本章把退出原因映射成：

- `AgentLoopExit::Idle`
- `AgentLoopExit::Shutdown`

模型停止调用工具、调用 `idle`、或达到单次回合上限，都会进入 `Idle`。只有收到 stop signal 或明确 shutdown，才进入 `Shutdown`。

## Idle Polling

idle phase 是一个轻量轮询循环：

```text
sleep
  -> 检查 stop signal
  -> 读取 inbox
  -> 如果有消息，恢复 Working
  -> 如果没消息，尝试 auto-claim task
  -> 如果 claim 成功，恢复 Working
  -> 超时后 Shutdown
```

这不是完整事件驱动队列，但已经表达了长期 worker 的基本行为：空闲时不占用模型调用，有新工作时再恢复。

## Identity Context

长期 teammate 从 idle 恢复时，必须记得自己是谁。

本章在 `LoopState` 中保存：

- `name`
- `role`
- `team_name`

并在恢复工作时重新注入 identity context。这样 teammate 不会因为上下文变化而变成匿名 agent，它始终以自己的角色继续执行。

## Task Claim

s17 的 task 不只是任务记录，而是可以被 worker 认领的工作项。

`TaskRecord` 增加了：

- `owner`
- `claimed_at`
- `claim_source`
- `claim_role`
- `required_role`

一个 task 可以被认领，需要满足：

- 状态是 `pending`。
- 没有 owner。
- 没有未完成的 blocking task。
- required role 和 teammate role 匹配。

这让 `.tasks/` 目录开始具备最小 work queue 的语义。

## Auto Claim

`SharedTeammateManager::auto_claim_task(...)` 会为 idle teammate 查找可认领任务：

```text
扫描 pending tasks
  -> 找到 role 匹配且未被阻塞的任务
  -> 写入 owner / claim 信息
  -> 把 task 注入 teammate 上下文
  -> 恢复 work phase
```

这一步让任务分配不再完全依赖 lead。lead 可以创建 backlog，teammate 自己从 backlog 拉取合适任务。

## Shutdown 和 Cleanup

本章把三件事拆开：

- shutdown request：lead 请求某个 teammate 停止。
- stop signal：manager 设置协作式停止标记。
- cleanup：teammate task 真的结束后，回收 runtime 状态。

这比在收到 shutdown response 时立即“删除 teammate”更清楚。真正退出发生在 teammate 自己的 loop 中，manager 只负责发信号和善后。

## 工具集合

lead 侧工具包括：

- `spawn_teammate`
- `shutdown_request`
- `plan_approval`
- `task_create`
- `task_list`
- `task_update`
- `task_claim`
- message / inbox / broadcast 工具

teammate 侧工具包括：

- `idle`
- `task_claim`
- `shutdown_response`
- `plan_approval`
- `send_message`
- `read_inbox`
- 文件和 shell 工具

工具集合的差异说明：lead 和 teammate 不只是 prompt 不同，它们的动作边界也不同。

## Rust 实现要点

- 用 enum 表达 worker 生命周期。
- 用 `tokio::spawn` 启动长期 teammate task。
- 用共享 manager 保存 runtime handle 和 stop signal。
- 用 durable task record 表达 claim 状态。
- 用 idle polling 避免空闲时持续消耗模型调用。
- 用 role-aware claim 连接 teammate 身份和任务调度。

## 本章的局限

- idle polling 是固定间隔轮询，不是事件驱动唤醒。
- auto-claim 使用 first-ready 策略，没有优先级和公平性。
- claim 没有 lease、超时回收和重试机制。
- plan approval 仍然是协议，不是强制 scheduler policy。
- runtime supervision 只做了最小 cleanup。
