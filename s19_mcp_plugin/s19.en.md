# s19: MCP & Plugin System

Earlier chapters use native tools implemented in Rust. This chapter extends the tool system so external MCP servers can provide tools to the same agent loop.

The goal is not to build a full MCP platform. The goal is smaller and more important:

```text
External MCP tools enter the same tool pool, pass through the same permission gate, and return through the same tool_result loop.
```

## Run

```bash
cargo run -p s19_mcp_plugin
```

The program scans:

```text
.claude-plugin/plugin.json
```

for MCP server definitions.

## New Capabilities

- Load plugin manifests.
- Start MCP servers as stdio child processes.
- Initialize MCP clients with `rmcp`.
- Call `tools/list` and convert MCP tools into model-visible tool specs.
- Route `mcp__...` tool names back to the correct server.
- Run native and MCP tools through the same permission model.
- Return MCP results as ordinary tool results.

## Native Tools and MCP Tools

Native tools are implemented in this Rust crate:

- `bash`
- `read_file`
- `write_file`
- `edit_file`

MCP tools are provided by external server processes.

The agent exposes both to the model in one tool list:

```text
native tool specs + MCP tool specs
```

When the model calls a tool:

```text
model tool_use
  -> permission check
  -> native ToolRouter or MCPToolRouter
  -> normalized tool_result
  -> model context
```

## Plugin Manifest

Example:

```json
{
  "name": "local-test-mcp",
  "version": "1.0.0",
  "mcpServers": {
    "everything": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-everything@2025.8.18"],
      "env": {}
    }
  }
}
```

`PluginLoader` only discovers and parses manifests. MCP connection logic lives elsewhere.

## Naming

Each server gets an internal name:

```text
{plugin_name}__{server_name}
```

Model-visible MCP tool names use:

```text
mcp__{plugin_name}__{server_name}__{tool_name}
```

Example:

```text
mcp__local-test-mcp__everything__echo
```

This prevents name collisions and makes tool origin explicit.

## McpClient

`McpClient::try_new`:

```text
spawn server process
  -> initialize rmcp client service
  -> list tools
  -> cache tools
  -> return initialized client
```

If connection or tool discovery fails, construction returns an error. The router never holds a half-initialized client.

## MCPToolRouter

The router:

- Parses the full MCP tool name.
- Extracts server and real tool name.
- Finds the matching client.
- Calls the MCP server.

`McpToolName` owns parsing rules so the router stays focused on dispatch.

## Tool Specs

MCP `tools/list` results are converted into Anthropic tool specs and added to the same list as native tools.

The model does not need to know whether a tool is native or remote. It only sees a schema.

## Permission

MCP tools must not bypass permissions.

Before dispatch, the harness normalizes a tool call into a capability intent:

- source: native or MCP
- server name
- real tool name
- risk: read, write, or high

Then `PermissionManager` decides allow, deny, or ask.

## Runtime Commands

Inside the CLI:

```text
/mcp
/tools
/mode auto
```

These let you inspect connected MCP servers, available tools, and permission mode.

## Limits

- Only stdio transport.
- No HTTP/SSE/Streamable HTTP transport.
- No OAuth or token refresh.
- No MCP resources or prompts.
- No plugin install, update, or signature verification.
- No automatic reconnect.
- No concurrent request multiplexing.
