# s19: MCP & Plugin System

## 目标

`s19` 的核心目标，是在前面已经稳定的 agent loop / tool / permission 基础上，把工具来源从“只来自本地 Rust 代码”扩展成“也可以来自外部 MCP server”。

这一章不追求完整 MCP 平台实现，而是实现最小可教学路径：

1. 启动一个外部 MCP server 进程
2. 通过 `rmcp` 的 client transport 建立连接
3. 调用 MCP SDK 提供的接口获取外部工具
4. 把外部工具标准化成 agent tool pool 里的普通工具
5. 模型调用 `mcp__{server}__{tool}` 时路由回对应 server
6. MCP 工具和 native 工具一样经过统一权限闸门
7. 工具结果统一包装成主 loop 能消费的 `tool_result`

一句话说，`s19` 要证明的是：

**MCP / plugin 不是另一套 agent 系统，而是外部能力进入同一控制面和执行面的入口。**

---

## 和前面章节的关系

`s19` 建议直接基于 `s07_permission_system` 继续推进。

原因是这一章的主要关注点是：

- 外部工具如何进入 tool pool
- MCP 工具如何继续经过 permission gate
- native / external capability 如何回到同一个 `tool_result` 回路

如果基于 `s18`，task / worktree / sub-agent 会把读者注意力带到任务隔离和多工作区协作上，反而稀释 MCP 这一章的主线。

本章最重要的复用对象是：

- `Tool` trait：所有能力都要变成统一工具
- agent loop：模型产出 `tool_use`，程序执行工具，再把 `tool_result` 放回上下文
- permission：执行前先判断 allow / ask / deny
- native tools：`bash`、`read_file`、`write_file`、`edit_file`

本章新增的是：

- `rmcp` client integration
- `PluginLoader`
- `MCPToolRouter`
- MCP tool wrapper
- capability intent normalization

---

## 非目标

为了让教学主线不失焦，`s19` 不实现以下内容：

- HTTP / SSE / streamable HTTP 等其他 MCP transport
- OAuth、token refresh、浏览器授权流
- resources / prompts / elicitation 等非 tool capability
- marketplace 安装、升级、签名校验
- server 自动重连、健康检查、后台生命周期管理
- 并发请求 multiplexing
- 长输出流式回传
- 手写完整 MCP JSON-RPC 协议栈

这些都是真实系统需要考虑的内容，但不属于本章最小实现。

## 核心数据结构

### 1. Capability Intent

所有工具调用先被归一化成 capability intent。

```rust
pub struct CapabilityIntent {
    pub source: CapabilitySource,
    pub server: Option<String>,
    pub tool: String,
    pub risk: CapabilityRisk,
}

pub enum CapabilitySource {
    Native,
    Mcp,
}

pub enum CapabilityRisk {
    Read,
    Write,
    High,
}
```

归一化规则：

- `mcp__{server}__{tool}` 表示 MCP 来源
- 其他工具表示 native 来源
- `read_file` 或 `read/list/get/show/search/query/inspect` 开头的工具视为 read
- `bash` 默认 write，但包含 `rm -rf`、`sudo`、`shutdown`、`reboot` 时视为 high
- `delete/remove/drop/shutdown` 开头的工具视为 high
- 其他工具默认 write

### 2. MCP server config

plugin manifest 里的 MCP server 配置解析成：

```rust
pub struct McpServerConfig {
    pub command: String,
    pub args: Vec<String>,
    pub env: HashMap<String, String>,
}
```

### 3. MCP tool spec

MCP server 返回的 tool 转成 agent tool spec：

```rust
ToolSpec {
    name: format!("mcp__{server_name}__{tool_name}"),
    description: tool.description,
    input_schema: tool.input_schema,
}
```

## MCP Client 规格

`s19` 不建议手写 MCP JSON-RPC client。

建议直接使用 `rmcp`，原因是：

- 它是 MCP 官方 Rust SDK
- 它已经覆盖 client / server 基础协议对象
- 它提供 stdio child process transport，正好匹配本章需求
- 教学上可以顺便展示 Rust 生态里“协议层交给 SDK，agent 层保留自己的控制面”
- 代码会更聚焦在本章真正要讲的内容：发现、命名、路由、权限、结果归一化

本章里的 `McpClient` 应该只是一个薄封装，而不是协议实现。

它负责：

- 根据 plugin config 创建 `rmcp` client transport
- 建立 MCP client service
- 保存已发现的 tool list
- 调用外部工具
- 关闭 client / child process

建议接口：

```rust
pub struct McpClient {
    pub server_name: String,
    service: McpServiceHandle,
    tools: Vec<McpTool>,
}

impl McpClient {
    pub async fn connect(&mut self) -> anyhow::Result<()>;
    pub async fn list_tools(&mut self) -> anyhow::Result<Vec<McpTool>>;
    pub async fn call_tool(&mut self, tool_name: &str, arguments: Value) -> anyhow::Result<String>;
    pub fn agent_tools(&self) -> Vec<ToolSpec>;
    pub async fn disconnect(&mut self);
}
```

这里的 `McpServiceHandle` 不是要求自定义一个真实类型，而是 spec 里的占位名称。实现时应替换成 `rmcp` 当前版本暴露的 client service / peer 类型。

最小连接流程：

```text
connect()
  -> 根据 command + args 创建 rmcp child process transport
  -> 启动 rmcp client service

list_tools()
  -> 调用 rmcp client list_tools

call_tool()
  -> 调用 rmcp client call_tool { name, arguments }
  -> join text content
```

教学版可以假设：

- 只接 stdio child process transport
- 每个 server 一个 client service
- content 里优先读取 text 字段
- `rmcp` API 版本可能变化，代码以实际锁定版本为准

## Plugin Loader 规格

`PluginLoader` 只做发现，不做连接。

扫描规则：

```text
{search_dir}/.claude-plugin/plugin.json
```

manifest 示例：

```json
{
  "name": "my-db-tools",
  "version": "1.0.0",
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {}
    }
  }
}
```

命名规则：

```text
{plugin_name}__{server_name}
```

这样可以避免不同 plugin 里的 server 重名。

接口建议：

```rust
pub struct PluginLoader {
    search_dirs: Vec<PathBuf>,
    plugins: HashMap<String, PluginManifest>,
}

impl PluginLoader {
    pub fn scan(&mut self) -> anyhow::Result<Vec<String>>;
    pub fn mcp_servers(&self) -> HashMap<String, McpServerConfig>;
}
```

## MCP Tool Router 规格

`MCPToolRouter` 管理多个 MCP client，并把 prefixed tool name 路由回对应 server。

接口建议：

```rust
pub struct MCPToolRouter {
    clients: HashMap<String, McpClient>,
}

impl MCPToolRouter {
    pub fn register_client(&mut self, client: McpClient);
    pub fn is_mcp_tool(tool_name: &str) -> bool;
    pub async fn call(&mut self, tool_name: &str, arguments: Value) -> anyhow::Result<String>;
    pub fn all_tools(&self) -> Vec<ToolSpec>;
    pub async fn disconnect_all(&mut self);
}
```

路由规则：

```text
mcp__{server}__{tool}
```

如果格式不合法，返回错误。

如果 server 不存在，返回错误。

## Tool Pool 合并规则

启动时构建完整工具池：

```text
native tools
  +
mcp_router.all_tools()
  =
agent visible tools
```

冲突策略：

- native tool 优先
- MCP tool 本身带 `mcp__` 前缀，正常不会和 native 重名
- 如果重复，保留先注册的 native tool

## Permission 规格

权限闸门必须在路由前执行。

执行顺序：

```text
tool_use
  -> permission_gate.normalize(name, input)
  -> permission_gate.check(intent)
  -> allow / ask / deny
  -> native handler 或 MCP router
```

模式：

```rust
pub enum PermissionMode {
    Default,
    Auto,
}
```

行为：

- read：直接 allow
- auto + 非 high：allow
- high：ask
- default + write：ask

这里和 `s07` 的完整权限系统不完全相同，原因是本章重点是“native / MCP 共享同一闸门”，不是完整规则引擎。

## Tool Result 规格

不论 native 还是 MCP，工具结果都包装成统一 JSON 字符串返回给模型。

```json
{
  "source": "mcp",
  "server": "my-db-tools__postgres",
  "tool": "query",
  "risk": "read",
  "status": "ok",
  "preview": "..."
}
```

字段说明：

| 字段 | 含义 |
|---|---|
| `source` | `native` 或 `mcp` |
| `server` | MCP server 名；native 为 null |
| `tool` | 真实工具名，不含 `mcp__` 前缀 |
| `risk` | `read` / `write` / `high` |
| `status` | `ok` / `error` |
| `preview` | 工具输出前 500 字符 |

教学版只回传 preview 是可以接受的，因为重点是统一总线。后续可以扩展成完整 payload + preview。

## CLI 行为

启动流程：

```text
1. dotenv
2. 获取 cwd
3. 初始化 native tools
4. scan plugin manifest
5. 对每个 MCP server:
   - connect
   - list_tools
   - register_client
6. 打印工具池数量
7. 进入交互 loop
```

交互命令：

| 命令 | 行为 |
|---|---|
| `/tools` | 列出 native + MCP tools |
| `/mcp` | 列出已连接 MCP server 和 tool 数量 |
| `q` / `exit` / 空输入 | 退出 |

退出时：

- 调用 `disconnect_all()`
- 尽量发送 `shutdown`
- terminate child process

## 验收标准

实现完成后，至少应该满足：

1. 无 plugin 时仍能启动，工具池只有 native tools
2. `.claude-plugin/plugin.json` 存在时能扫描到 plugin
3. MCP server command 不存在时打印错误，但不影响 native agent 启动
4. MCP server 连接成功时能调用 `tools/list`
5. `/tools` 能看到 `mcp__{plugin}__{server}__{tool}` 风格工具
6. 模型调用 MCP tool 时会先走 permission gate
7. permission 拒绝时不会调用 MCP server
8. MCP tool 调用结果会被包装成统一 JSON payload
9. native tool 调用结果也使用同一包装格式
10. 退出时清理所有 MCP server 子进程

## 建议测试

单元测试优先覆盖纯逻辑：

- `CapabilityPermissionGate::normalize`
- MCP tool name 解析
- plugin manifest 解析
- native + MCP tool pool 合并
- tool result normalization

集成测试可以用一个 tiny fake MCP server：

- 用 `rmcp` server API 暴露一个 `echo` tool
- 测试端通过 `rmcp` child process transport 连接它
- `tools/list` 能发现 `echo`
- `tools/call` 能拿到 text content

这样不依赖真实外部 MCP 包，也不需要网络。

## 实现顺序

建议按下面顺序开工：

1. 从 `s07_permission_system` 复制出 `s19_mcp_plugin` 的起点
2. 加 `permission.rs` 和统一 result payload
3. 加 `plugin.rs`，只做 manifest 解析
4. 加 `rmcp` 依赖和 `mcp_client.rs` 薄封装，跑通 fake MCP server
5. 加 `router.rs`，把 MCP tools 包装进 tool pool
6. 加 `/tools` 和 `/mcp`
7. 补单元测试和 fake server 集成测试

这样每一步都有可验证产物，不需要一次性把所有复杂度堆上去。
