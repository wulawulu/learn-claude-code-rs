# s19: MCP & Plugin System

## 目标

`s19` 的核心目标，是把前面章节里已经稳定下来的本地 tool system，扩展成一个可以接入外部 MCP server 的 agent。

这一章要讲清楚的不是“怎么做一个完整 MCP 平台”，而是更小的一件事：

**外部工具如何进入同一个 agent loop，并继续共享同一套 tool pool、permission gate 和 tool_result 回路。**

在前面的章节里，模型能调用的工具都来自 Rust 代码本身，例如：

- `bash`
- `read_file`
- `write_file`
- `edit_file`

到了 `s19`，工具来源变成了两类：

- native tool：仍然由当前 Rust crate 实现
- MCP tool：由外部 MCP server 进程提供

但这两类工具进入 agent 后，并不会分裂成两套执行系统。

它们共享同一条主路径：

```text
model tool_use
  -> permission check
  -> native tool 或 MCP router
  -> normalized tool_result
  -> 回到模型上下文
```

所以这一章的重点是：

- MCP server 如何被发现和连接
- MCP tool 如何被转成模型可见的 tool spec
- MCP tool name 如何路由回正确的 server
- native / MCP tool 如何共用权限判断
- native / MCP tool result 如何回到同一个结果格式

---

## 当前 Rust 版做了什么

当前 `s19_mcp_plugin` 可以概括成一句话：

**在 `s07_permission_system` 的 agent loop 和 permission gate 之上，加了一层最小 MCP plugin 接入。**

启动时，程序会从当前工作目录扫描：

```text
.claude-plugin/plugin.json
```

这个文件描述外部 MCP server 的启动方式。

比如当前项目根目录里的测试配置会通过 `npx` 启动两个 MCP server：

- `@modelcontextprotocol/server-everything`
- `@modelcontextprotocol/server-sequential-thinking`

程序启动后会做这几件事：

1. 初始化 native tools
2. 扫描 plugin manifest
3. 为每个 MCP server 启动一个 stdio child process
4. 用 `rmcp` 完成 MCP client 初始化
5. 调用 `tools/list` 获取 server 暴露的工具
6. 把这些工具包装成模型可见的 tool spec
7. 把 native tools 和 MCP tools 合并成同一个 tool pool
8. 进入原来的 agent loop

这里的关键点是第 7 步。

MCP tool 不是在模型外面单独处理的，也不是通过另一个 agent 调用的。它只是被转换成普通 tool spec 后，进入模型已经熟悉的工具列表。

---

## 为什么是 plugin manifest

如果只为了测试 MCP，代码里硬编码一个 server command 也能跑。

但那样会把两个概念混在一起：

- agent loop 如何使用工具
- 外部工具从哪里来

`s19` 把后者单独抽成 `.claude-plugin/plugin.json`。

manifest 的结构很小：

```json
{
  "name": "local-test-mcp",
  "version": "1.0.0",
  "mcpServers": {
    "everything": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-everything@2025.8.18"],
      "env": {}
    }
  }
}
```

`PluginLoader` 只负责两件事：

- 找到 manifest
- 把 manifest 解析成 `McpServerConfig`

它不负责连接 MCP server。

这个边界很重要：

- plugin loader 是发现层
- MCP client 是连接层
- router 是调用层
- loop 是执行控制层

这样每一层的职责都比较窄。

---

## MCP server 的命名

manifest 里每个 server 都会被转换成内部 server name：

```text
{plugin_name}__{server_name}
```

例如：

```text
local-test-mcp__everything
```

模型最终看到的 MCP tool name 是：

```text
mcp__{plugin_name}__{server_name}__{tool_name}
```

例如：

```text
mcp__local-test-mcp__everything__echo
```

这个命名方案解决了两个问题。

第一，它让模型和程序都能从名字上看出工具来源：

- 没有 `mcp__` 前缀的是 native tool
- 有 `mcp__` 前缀的是 MCP tool

第二，它避免了不同 plugin 或 server 暴露同名 tool 时互相覆盖。

一个 `echo` tool 单独看没有唯一性。

但下面这个名字是唯一的：

```text
mcp__local-test-mcp__everything__echo
```

## McpClient：连接后就是可用对象

`mcp_client.rs` 是 `rmcp` 的薄封装。

这一章没有手写 MCP JSON-RPC 协议。协议细节交给 `rmcp`，当前 crate 只关心 agent 层的几件事：

- 如何启动 server 进程
- 如何获取 tools
- 如何调用 tool
- 如何关闭连接

当前 `McpClient` 的状态是：

```rust
pub struct McpClient {
    pub server_name: String,
    service: RunningService<RoleClient, ()>,
    tools: Vec<McpTool>,
}
```

这里没有 `Option<service>`。

这是一个刻意的设计。

`McpClient::try_new` 完成完整初始化：

```text
try_new(server_name, config)
  -> spawn child process
  -> rmcp serve 初始化 client service
  -> list_all_tools
  -> 缓存 tools
  -> 返回 McpClient
```

所以只要 `McpClient` 存在，就表示：

- MCP server 已经连接成功
- tools 已经发现并缓存
- 这个 client 可以被 router 注册

如果连接失败，或者 `tools/list` 失败，`try_new` 会返回错误。

这样 router 不会持有半初始化的 client，后续 `call_tool` 也不需要反复处理“未连接”的分支。

---

## MCPToolRouter：把名字路由回 server

`MCPToolRouter` 管理多个 `McpClient`。

模型调用 MCP tool 时，传入的是完整 tool name：

```text
mcp__local-test-mcp__everything__echo
```

router 需要把它拆成：

```text
server = local-test-mcp__everything
tool   = echo
```

这个解析规则被放在 `McpToolName` 上：

```rust
impl TryFrom<&str> for McpToolName
```

这样 `McpToolName` 自己负责“什么是合法 MCP tool name”。

router 的职责就只剩下：

1. 把字符串转成 `McpToolName`
2. 根据 `server` 找到对应 client
3. 用真实 `tool` 名调用 MCP server

这种拆法比在 router 里放一个 `parse_tool_name` 方法更清楚，因为解析规则属于类型本身。

---

## Tool pool：MCP tool 变成普通 tool spec

模型并不知道某个工具背后是 Rust 函数还是 MCP server。

它只看到 tool spec。

native tool 的 spec 来自本地 `Tool` trait：

```rust
tool.tool_spec()
```

MCP tool 的 spec 来自 server 的 `tools/list` 返回值。

`McpClient::agent_tools()` 会把 MCP tool 转成 Anthropic tool spec，并加上统一前缀：

```text
mcp__{server_name}__{tool_name}
```

最终 `LoopState::all_tool_specs()` 只是把两边接起来：

```rust
self.tools
    .values()
    .map(|tool| tool.tool_spec())
    .chain(self.mcp_router.all_tools())
    .collect()
```

这里没有额外去重。

原因是 MCP tool 自带 `mcp__` 前缀，和 native tool 不在同一个命名空间里。

## Permission：按 capability 判断，而不是按实现来源判断

如果 MCP tool 绕过权限系统，那它就不是 agent 的一等工具，而是一个外部逃逸通道。

所以 `s19` 的权限闸门放在路由之前：

```text
tool_use
  -> normalize_capability(name, input)
  -> permission_manager.check(...)
  -> allow / ask / deny
  -> native execute 或 MCP router
```

这里最重要的是 `CapabilityIntent`：

```rust
pub struct CapabilityIntent {
    pub source: CapabilitySource,
    pub server: Option<String>,
    pub tool: String,
    pub risk: CapabilityRisk,
}
```

它把一次 tool call 归一化成四个问题：

- 来源是什么：`native` 还是 `mcp`
- 如果是 MCP，server 是谁
- 真实 tool 名是什么
- 风险等级是什么

风险等级目前只有三类：

- `read`
- `write`
- `high`

分类规则保持简单：

- `read_file` 或 `read/list/get/show/search/query/inspect` 开头的工具是 `read`
- `bash` 默认是 `write`
- `bash` 中包含 `rm -rf`、`sudo`、`shutdown`、`reboot` 是 `high`
- `delete/remove/drop/shutdown` 开头的工具是 `high`
- 其他工具默认是 `write`

当前 permission mode 有三种：

```rust
pub enum PermissionMode {
    Default,
    Plan,
    Auto,
}
```

行为是：

- `read`：直接 allow
- `plan + 非 read`：deny
- `high`：ask
- `auto + 非 high`：allow
- `default + write`：ask

用户在 ask 时可以选择：

- `allow once`
- `deny`
- `always allow this tool`

`always allow this tool` 只是把完整 tool name 加入 allowlist。

高风险工具仍然优先 ask，不会因为 allowlist 自动放行。

这让权限模型保持了当前章节需要的重点：

- native / MCP 共享同一判断
- read/write/high 是控制面的核心语义
- 复杂 rule engine 不是这一章的主题

## Tool result：统一回到一个 JSON payload

工具执行之后，不管来源是 native 还是 MCP，都会包装成同一个结构：

```rust
pub struct ToolResultPayload {
    pub source: String,
    pub server: Option<String>,
    pub tool: String,
    pub risk: String,
    pub status: ToolResultStatus,
    pub preview: String,
}
```

`status` 是 enum：

```rust
pub enum ToolResultStatus {
    Ok,
    Error,
}
```

一个 MCP echo 的结果大概长这样：

```json
{
  "source": "mcp",
  "server": "local-test-mcp__everything",
  "tool": "echo",
  "risk": "write",
  "status": "ok",
  "preview": "Echo: hello from s19"
}
```

这里没有把完整输出无限制塞回上下文，而是只放 `preview`，并截断到 500 字符。

这个取舍是为了让本章聚焦在统一结果总线上。

后续如果要做真实产品，可以把完整 payload、附件、结构化输出和 preview 分开管理。

## CLI 行为

启动后，程序先连接 MCP server，再进入交互 loop。

启动日志里会看到：

```text
[Plugins: local-test-mcp]
[MCP connected: local-test-mcp__everything (... tools)]
[MCP connected: local-test-mcp__sequential-thinking (... tools)]
```

然后选择 permission mode：

```text
default - ask for writes
plan - read only
auto - allow non-high operations
```

交互 loop 中有几个控制命令：

| 命令 | 作用 |
|---|---|
| `/tools` | 列出 native + MCP tools |
| `/mcp` | 列出已连接 MCP server 和工具数量 |
| `/rules` | 列出 always-allowed tool allowlist |
| `/mode` | 交互式切换 permission mode |
| `/mode default` | 切到 default |
| `/mode plan` | 切到 plan |
| `/mode auto` | 切到 auto |
| `q` / `exit` / `exit()` / 空输入 | 退出 |

退出时会调用：

```text
mcp_router.disconnect_all()
```

router 会 drain 所有 clients，并逐个 shutdown。

## 运行时可以观察什么

可以用下面命令启动：

```bash
cargo run -p s19_mcp_plugin
```

进入 loop 后，先输入：

```text
/mcp
```

这能看到当前连接了哪些 MCP server。

再输入：

```text
/tools
```

这能看到 native tools 和 MCP tools 已经在同一个 tool pool 里。

然后让模型调用 MCP echo：

```text
Call mcp__local-test-mcp__everything__echo with message "hello from s19".
```

这条调用会经过完整链路：

```text
model tool_use
  -> permission check
  -> MCPToolRouter
  -> McpClient::call_tool
  -> ToolResultPayload
  -> model context
```

如果切到 auto 模式：

```text
/mode auto
```

非 high 的 MCP tool 会自动通过 permission。

这可以很直观地看到：MCP tool 虽然来自外部 server，但进入 agent 后已经和 native tool 一样受同一个权限模型控制。

---

## 这一章没有做什么

当前实现只覆盖教学主线需要的最小闭环。

它没有实现：

- HTTP / SSE / Streamable HTTP transport
- OAuth 和 token refresh
- MCP resources 暴露给模型
- MCP prompts 暴露给模型
- plugin 安装、升级、签名校验
- MCP server 自动重连
- 长输出流式回传
- 并发 MCP request multiplexing

这些都是真实 MCP 平台会遇到的问题。

但在 `s19` 里，先把最小控制面跑通更重要：

**外部 MCP tool 进入 tool pool，经过 permission gate，被 router 执行，再以统一 tool_result 回到 agent loop。**
